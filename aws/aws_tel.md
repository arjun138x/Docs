# 1. AWS Fundamentals

మనం మొదటి టాపిక్ అయిన **AWS Fundamentals** (క్లౌడ్ బేసిక్స్) నుండి ప్రారంభించి, మీరు ఇచ్చిన లిస్ట్‌లోని ప్రతి పాయింట్‌ను తెలుగులో చాలా సులభంగా అర్థం చేసుకుందాం.

---

### 1. What is Cloud Computing? (క్లౌడ్ కంప్యూటింగ్ అంటే ఏంటి?)

గతంలో ఏదైనా ఒక వెబ్‌సైట్ లేదా అప్లికేషన్ రన్ చేయాలంటే, మనమే సొంతంగా సర్వర్లు, హార్డ్‌వేర్, నెట్‌వర్క్ కేబుల్స్ కొని, ఒక రూమ్‌లో పెట్టి మెయింటైన్ చేయాల్సి వచ్చేది. దీనికి చాలా ఖర్చు, సమయం అవుతుంది.

**క్లౌడ్ కంప్యూటింగ్** అంటే... ఇంటర్నెట్ ద్వారా మరొకరి (ఉదాహరణకు Amazon వారి) సర్వర్లు, డేటాబేస్‌లు, స్టోరేజ్‌లను మనకు కావలసినంత వరకు అద్దెకు (Rent) తీసుకోవడం.

> **ఉదాహరణ:** మనం సొంతంగా ఇల్లు కట్టుకోకుండా, ఒక ఫ్లాట్‌ని అద్దెకు తీసుకుని, ఉన్నన్ని రోజులు రెంట్ కట్టి ఖాళీ చేయడం లాంటిదే క్లౌడ్ కంప్యూటింగ్.

---

### 2. Types of Cloud (క్లౌడ్ రకాలు)

క్లౌడ్‌ను మనం ఉపయోగించే విధానాన్ని బట్టి మూడు రకాలుగా విభజించవచ్చు:

- **Public Cloud (పబ్లిక్ క్లౌడ్):** ఈ క్లౌడ్ ఇన్‌ఫ్రాస్ట్రక్చర్ అందరికీ అందుబాటులో ఉంటుంది. AWS, Google Cloud, Microsoft Azure దీనికి ఉదాహరణలు. చిన్న స్టార్టప్ నుండి పెద్ద కంపెనీల వరకు ఎవరైనా ఇందులో అకౌంట్ క్రియేట్ చేసుకుని సర్వీసులు వాడుకోవచ్చు. (ఇది పబ్లిక్ బస్సు లేదా ట్రైన్ లాంటిది).
- **Private Cloud (ప్రైవేట్ క్లౌడ్):** ఇది ఒక నిర్దిష్టమైన కంపెనీ లేదా ఆర్గనైజేషన్ కోసం మాత్రమే ప్రత్యేకంగా డిజైన్ చేయబడుతుంది. డేటా సెక్యూరిటీ చాలా ముఖ్యమైన బ్యాంకులు, ప్రభుత్వ రంగాలు తమ సొంత సర్వర్లను మెయింటైన్ చేస్తూ ప్రైవేట్ క్లౌడ్ వాడుతుంటాయి. (ఇది మీ సొంత కార్ లాంటిది).
- **Hybrid Cloud (హైబ్రిడ్ క్లౌడ్):** ఇది పబ్లిక్ మరియు ప్రైవేట్ క్లౌడ్ ల కలయిక. అత్యంత రహస్యమైన డేటాను ప్రైవేట్ క్లౌడ్‌లో ఉంచి, నార్మల్ అప్లికేషన్ ప్రాసెసింగ్‌ను పబ్లిక్ క్లౌడ్‌లో రన్ చేస్తారు. (ఇది మీ సొంత కార్ ని వాడుతూ, అవసరమైనప్పుడు క్యాబ్ బుక్ చేసుకోవడం లాంటిది).

---

### 3. Benefits of Cloud (క్లౌడ్ వల్ల లాభాలు)

- **Cost Savings (ఖర్చు ఆదా):** సర్వర్లు కొనాల్సిన అవసరం లేదు. "Pay-as-you-go" మోడల్ (వాడినంత వరకే బిల్ కట్టడం).
- **Speed & Agility (వేగం):** కొత్త సర్వర్ కావాలంటే కేవలం కొన్ని సెకన్లలో (కొన్ని క్లిక్స్‌తో) రెడీ అయిపోతుంది.
- **Global Footprint (ప్రపంచవ్యాప్తంగా విస్తరణ):** మీ అప్లికేషన్‌ను ప్రపంచంలో ఎక్కడైనా ఒక్క క్లిక్‌తో డిప్లాయ్ చేయవచ్చు.
- **High Availability & Reliability (ఎల్లప్పుడూ అందుబాటులో ఉండటం):** ఒక సర్వర్ డౌన్ అయినా, ఇంకొక సర్వర్ నుండి అప్లికేషన్ రన్ అవుతుంది. బ్యాకప్స్ సులభం.

---

### 4. AWS Global Infrastructure (AWS గ్లోబల్ ఇన్‌ఫ్రాస్ట్రక్చర్)

ప్రపంచవ్యాప్తంగా అమెజాన్ వారు తమ సర్వర్లను ఎలా అమర్చారో తెలిపే నెట్‌వర్క్ ఇది. దీనిని నాలుగు భాగాలుగా అర్థం చేసుకోవచ్చు:

- **AWS Regions:** ప్రపంచంలోని వివిధ భౌగోళిక ప్రాంతాలను (Geographical locations) రీజియన్స్ అంటారు. ఉదాహరణకు: ముంబై (ap-south-1), హైదరాబాద్ (ap-south-2), నార్త్ వర్జీనియా (us-east-1). మనం అప్లికేషన్ క్రియేట్ చేసేటప్పుడు మన యూజర్లకు దగ్గరగా ఉండే రీజియన్‌ని ఎంచుకోవాలి.
- **Availability Zones (AZs):** ప్రతి రీజియన్‌ లోపల కొన్ని డేటా సెంటర్స్ (Data Centers) ఉంటాయి. వాటినే Availability Zones అంటారు. ఒక రీజియన్‌లో కనీసం 3 AZs ఉంటాయి. ఒకవేళ ఒక AZ లో వరదలు లేదా పవర్ కట్ వల్ల సర్వర్లు ఆగిపోయినా, ఇంకొక AZ లో మీ అప్లికేషన్ సేఫ్‌గా రన్ అవుతుంది.
- **Edge Locations:** ఇవి డేటా సెంటర్ల లాగా పెద్దగా ఉండవు, కానీ ప్రపంచవ్యాప్తంగా ఉన్న ప్రధాన నగరాల్లో ఉంటాయి (ఉదాహరణకు: చెన్నై, బెంగళూరు, ఢిల్లీ లాంటి చోట్ల). యూజర్లకి వెబ్‌సైట్ కంటెంట్ (ఇమేజెస్, వీడియోలు) చాలా ఫాస్ట్‌గా లోడ్ అవ్వడానికి (Caching - CloudFront ద్వారా) వీటిని వాడుతారు.
- **Local Zones:** మెయిన్ రీజియన్‌కి దూరంగా ఉన్న పెద్ద సిటీలలో ఉండే చిన్న క్లౌడ్ సెటప్స్ ఇవి. సింగిల్ డిజిట్ మిల్లీసెకండ్ లేటెన్సీ (చాలా తక్కువ టైమ్‌లో రెస్పాన్స్) కావలసిన అప్లికేషన్ల కోసం వీటిని వాడుతారు.

### 5. AWS Core Models & Frameworks

- **AWS Free Tier:** కొత్తగా నేర్చుకునే వారి కోసం AWS 12 నెలల పాటు కొన్ని సర్వీసులను ఉచితంగా (పరిమితులకు లోబడి) అందిస్తుంది. అకౌంట్ క్రియేట్ చేయడానికి క్రెడిట్/డెబిట్ కార్డ్ అవసరం అవుతుంది.
- **AWS Pricing Model:** **"Pay-as-you-go"** అంటే మీరు ఏ సర్వీస్ ఎంత సమయం లేదా ఎంత స్టోరేజ్ వాడారో, అంతవరకు మాత్రమే బిల్లు కట్టాలి. వాడకపోతే బిల్లు పడదు.
- **AWS Shared Responsibility Model:** క్లౌడ్ సెక్యూరిటీలో AWS బాధ్యత ఏంటి, మీ బాధ్యత ఏంటి అని చెప్పే రూల్ ఇది.
- _Security **of** the Cloud (AWS బాధ్యత):_ ఫిజికల్ సర్వర్లు, డేటా సెంటర్లు, నెట్‌వర్క్ కేబుల్స్ భద్రత.
- _Security **in** the Cloud (మీ బాధ్యత):_ మీ డేటా, మీ కోడ్, యూజర్ ఐడీలు, పాస్‌వర్డ్‌లు, ఓపెన్ చేసే పోర్ట్స్ భద్రత.

- **AWS Well-Architected Framework:** ఒక అప్లికేషన్‌ను క్లౌడ్‌లో బెస్ట్ పద్ధతిలో ఎలా బిల్డ్ చేయాలో చెప్పే గైడ్‌లైన్స్ (6 Pillars - Operational Excellence, Security, Reliability, Performance Efficiency, Cost Optimization, Sustainability).

### 6. Tools to Interact with AWS (AWS ని ఎలా యాక్సెస్ చేయాలి?)

మనం AWS సర్వీసులను (సర్వర్ క్రియేట్ చేయడం, డిలీట్ చేయడం వంటివి) 3 పద్ధతుల్లో మేనేజ్ చేయవచ్చు:

- **AWS Console:** ఇది బ్రౌజర్‌లో ఓపెన్ చేసే UI (Graphical User Interface). మౌస్ క్లిక్స్‌తో సర్వీసులను మేనేజ్ చేయవచ్చు. బిగినర్స్ నేర్చుకోవడానికి ఇది బెస్ట్.
- **AWS CLI (Command Line Interface):** మీ కంప్యూటర్ టెర్మినల్ లేదా కమాండ్ ప్రాంప్ట్ (CMD) నుండి కమాండ్స్ టైప్ చేస్తూ AWS సర్వీసులను కంట్రోల్ చేయడం. డెవలపర్లు టాస్క్‌లను ఆటోమేట్ చేయడానికి ఇది వాడుతారు.
- **AWS SDKs (Software Development Kits):** మీరు రాసే కోడ్ ద్వారా (ఉదాహరణకు Node.js, Python, Java కోడింగ్ ద్వారా) AWS సర్వీసులతో మాట్లాడటం. (ఉదాహరణకు: Node.js కోడ్ రన్ అవ్వగానే AWS S3 బకెట్‌లో ఒక ఫైల్ అప్‌లోడ్ అవ్వడం).

# 2. Identity & Security

మొదటి టాపిక్ (Fundamentals) విజయవంతంగా పూర్తి చేసుకున్నాం. ఇప్పుడు క్లౌడ్‌లో అత్యంత ముఖ్యమైన విభాగం — **Identity & Security** (భద్రత మరియు యాక్సెస్ కంట్రోల్) గురించి తెలుగులో స్పష్టంగా తెలుసుకుందాం.

---

## Part A: IAM (AWS Identity and Access Management)

**IAM అంటే ఏంటి?**
ఇది AWS లో ఒక సెక్యూరిటీ గార్డు లాంటిది. మీ AWS అకౌంట్‌లోకి **ఎవరు రావచ్చు (Authentication)** మరియు వారు **ఏం పనులు చేయవచ్చు (Authorization)** అనేది IAM నిర్ణయిస్తుంది.

> **రియల్-టైమ్ అనలాగీ:** ఒక పెద్ద సాఫ్ట్‌వేర్ కంపెనీ బిల్డింగ్‌ను ఊహించుకోండి. బిల్డింగ్‌లోకి వెళ్లడానికి మీకు ఐడీ కార్డ్ కావాలి (Authentication), లోపలికి వెళ్ళాక మీ టీమ్ బ్లాక్‌లోకి మాత్రమే వెళ్ళగలరు, సర్వర్ రూమ్‌లోకి వెళ్ళలేరు (Authorization). ఇదే IAM.

### 1. IAM భాగాలు (Users, Groups, Roles, Policies)

- **IAM Users (యూజర్లు):** కంపెనీలో పని చేసే ఒక్కొక్క వ్యక్తికి ఇచ్చే విడివిడి అకౌంట్స్ (ఉదాహరణకు: శ్రవణ్, లత). వీరికి లాగిన్ అవ్వడానికి యూజర్‌నేమ్, పాస్‌వర్డ్ లేదా కోడ్ ద్వారా యాక్సెస్ చేయడానికి Access Keys ఉంటాయి.
- **IAM Groups (గ్రూపులు):** ఒకే రకమైన బాధ్యతలు ఉన్న యూజర్లను ఒక గ్రూప్‌గా చేయడం.
- _ఉదాహరణ:_ `Developers` అని ఒక గ్రూప్ క్రియేట్ చేసి, దానికి కొన్ని పర్మిషన్స్ ఇచ్చి, ఆ గ్రూప్‌లో డెవలపర్స్ అందరినీ యాడ్ చేయవచ్చు. దీనివల్ల ఒక్కొక్కరికి విడిగా పర్మిషన్స్ ఇవ్వాల్సిన పనిలేదు.

- **IAM Policies (పాలసీలు):** "ఎవరికి ఏ పర్మిషన్స్ ఉన్నాయి" అని చెప్పే ఒక **JSON డాక్యుమెంట్**. ఇందులో ఏ సర్వీస్‌ని వాడుకోవచ్చు (Allow), దేన్ని వాడకూడదు (Deny) అని రాసి ఉంటుంది.
- **Managed Policies (మేనేజ్డ్ పాలసీలు):**
- **AWS Managed:** ఇవి AWS వాళ్ళే రెడీమేడ్‌గా ఇచ్చే పాలసీలు (ఉదాహరణకు: `AmazonS3FullAccess` - ఇది ఇస్తే S3 లో అన్నీ చేసుకోవచ్చు).
- **Customer Managed:** మన అవసరానికి తగ్గట్టుగా మనమే సొంతంగా రాసుకునే పాలసీలు.

- **Inline Policies (ఇన్‌లైన్ పాలసీలు):** ఒక నిర్దిష్టమైన యూజర్‌కి లేదా గ్రూప్‌కి మాత్రమే పరిమితమయ్యేలా నేరుగా వారి ప్రొఫైల్‌లోనే రాసే పాలసీ. ఇది వేరే ఎవరికీ అప్లై చేయలేము.

- **Permission Boundaries:** ఒక యూజర్‌కి గరిష్టంగా ఎంతవరకు పర్మిషన్స్ ఉండవచ్చో గీసే లక్ష్మణ రేఖ ఇది. ఆ బౌండరీ దాటి యూజర్ వేరే ఏ పర్మిషన్స్ తీసుకోలేడు (ఒకవేళ Admin పర్మిషన్స్ ఉన్నా సరే).

### 2. IAM Roles (రోల్స్) & Advanced Access

- **IAM Roles:** ఇది ఒక వ్యక్తికి ఇచ్చేది కాదు. ఇది ఒక **తాత్కాలిక పర్మిషన్ (Temporary Permission)**.
- _ఉదాహరణ:_ మీ బ్యాకెండ్ అప్లికేషన్ (Node.js) ఒక EC2 సర్వర్‌లో రన్ అవుతోంది. ఆ సర్వర్ AWS S3 లోని ఫైల్స్ చదవాలి. అప్పుడు మనం యూజర్ ఐడీ, పాస్‌వర్డ్‌లు సర్వర్‌లో పెట్టకుండా, EC2 కి ఒక 'IAM Role' ఇస్తాం.

- **Cross Account Access & Assume Role:** మీ కంపెనీకి రెండు AWS అకౌంట్స్ ఉన్నాయి అనుకుందాం (ఒకటి Development, ఇంకొకటి Production). డెవలప్‌మెంట్ అకౌంట్‌లో ఉన్న యూజర్, ప్రొడక్షన్ అకౌంట్‌లోని ఒక సర్వీస్‌ను తాత్కాలికంగా యాక్సెస్ చేయాలంటే **Assume Role** అనే కాన్సెప్ట్ వాడుతారు. దీనివల్ల యూజర్ తన ఒరిజినల్ ఐడీతోనే అవతలి అకౌంట్ రోల్‌ని తాత్కాలికంగా ధరించి పని పూర్తి చేయవచ్చు.

### 3. Identity Protection (భద్రత నియమాలు)

- **MFA (Multi-Factor Authentication):** మీ యూజర్‌నేమ్, పాస్‌వర్డ్ ఎవరికైనా తెలిసినా అకౌంట్ హ్యాక్ అవ్వకుండా ఉండటానికి, ఫోన్‌కి వచ్చే OTP (Google Authenticator లాంటివి) ని అదనపు సెక్యూరిటీ లేయర్‌గా పెట్టడం.
- **Password Policies:** కంపెనీ ఉద్యోగులు స్ట్రాంగ్ పాస్‌వర్డ్స్ పెట్టుకునేలా నియమాలు పెట్టడం (కనీసం 12 అక్షరాలు ఉండాలి, స్పెషల్ క్యారెక్టర్స్ ఉండాలి, ప్రతి 90 రోజులకు పాస్‌వర్డ్ మార్చాలి లాంటివి).

---

## Part B: Security, Encryption & Secrets

డేటాను దొంగిలించకుండా దాచడాన్ని **Encryption (గూఢలిప్యంతరణం)** అంటారు. ఇది రెండు రకాలు:

1. **Data-at-Rest:** హార్డ్ డిస్క్ లేదా స్టోరేజ్‌లో ఖాళీగా కూర్చున్న డేటాను ఎన్‌క్రిప్ట్ చేయడం.
2. **Data-in-Transit:** ఇంటర్నెట్ ద్వారా ఒక చోటు నుండి ఇంకొక చోటికి డేటా ప్రయాణిస్తున్నప్పుడు (SSL/TLS ద్వారా) ఎన్‌క్రిప్ట్ చేయడం.

### 1. AWS Key Management Service (KMS)

డేటాను లాక్ చేయడానికి మనకు "Keys" (తాళాలు) కావాలి. ఆ డిజిటల్ కీస్‌ని క్రియేట్ చేయడం, భద్రపరచడం, మేనేజ్ చేయడం KMS పని.

- **Customer Managed Keys (CMK):** ఈ తాళాలను (Keys) మనమే క్రియేట్ చేసుకుంటాం, మనమే వాటిని ఎప్పుడు మార్చాలో (Rotation) కంట్రోల్ చేస్తాం.

### 2. AWS Secrets Manager

సాఫ్ట్‌వేర్ డెవలపర్‌గా మనం కోడ్‌లో Database Passwords, API Keys, లేదా Tokenలను రాస్తుంటాం (Hardcoding). అలా రాస్తే సెక్యూరిటీ రిస్క్.
**Secrets Manager** అనేది ఒక డిజిటల్ లాకర్. పాస్‌వర్డ్‌లన్నీ ఇందులో దాచిపెడతాం. మన Node.js కోడ్ రన్ అయ్యేటప్పుడు రన్‌టైమ్‌లో ఈ లాకర్ నుండి పాస్‌వర్డ్‌ని సీక్రెట్‌గా తెచ్చుకుంటుంది. పైగా ఇది ఆటోమేటిక్‌గా డేటాబేస్ పాస్‌వర్డ్‌లను మారుస్తూ (Rotate) ఉంటుంది.

### 3. AWS Certificate Manager (ACM) & SSL/TLS

మనం ఏదైనా వెబ్‌సైట్ ఓపెన్ చేసినప్పుడు బ్రౌజర్ పైన తాళం గుర్తు ఉండి `https://` అని వస్తుంది కదా, దాన్నే **SSL/TLS Certificate** అంటారు. దీనివల్ల వెబ్‌సైట్ మరియు యూజర్ మధ్య జరిగే డేటా ప్రయాణం సేఫ్‌గా ఉంటుంది. ఈ Certificateలను ఉచితంగా క్రియేట్ చేసి, మేనేజ్ చేయడానికి AWS ACM సర్వీస్‌ని వాడుతాం.

---

**క్విక్ సమ్మరీ:**

- మనుషులకు/సర్వీసులకు పర్మిషన్స్ ఇవ్వడానికి ➡️ **IAM**
- తాళాలు (Keys) మేనేజ్ చేయడానికి ➡️ **KMS**
- పాస్‌వర్డ్‌లు, API టోకెన్స్ దాచడానికి ➡️ **Secrets Manager**
- వెబ్‌సైట్‌కి HTTPS (తాళం గుర్తు) తేవడానికి ➡️ **ACM**

# 3. Compute Services

సెక్యూరిటీ తర్వాత మన అప్లికేషన్ల కోడ్‌ను రన్ చేసే అత్యంత ముఖ్యమైన విభాగం — **Compute Services (కంప్యూట్ సర్వీసెస్)**.

సాఫ్ట్‌వేర్ డెవలపర్‌గా మీకు ఈ సెక్షన్ చాలా ఈజీగా అర్థమవుతుంది. ఎందుకంటే ఇందులో మనం కోడ్‌ను ఎక్కడ, ఎలా రన్ చేయాలో నేర్చుకుంటాం. AWS దీనిని మూడు రకాలుగా విభజించింది: **Virtual Machines (EC2)**, **Serverless (Lambda)**, మరియు **Containers (ECS/EKS)**.

---

## 1. Virtual Machines (Amazon EC2)

**EC2 (Elastic Compute Cloud)** అంటే సింపుల్‌గా చెప్పాలంటే **క్లౌడ్‌లో మనకు నచ్చిన కాన్ఫిగరేషన్‌తో అద్దెకు తీసుకునే కంప్యూటర్/సర్వర్**.

- **Instance Types (ఇన్‌స్టాన్స్ రకాలు):** మన అవసరాన్ని బట్టి వివిధ రకాల కంప్యూటర్లు ఉంటాయి.
- _General Purpose (T లేదా M సిరీస్):_ నార్మల్ వెబ్ యాప్స్, టెస్టింగ్ కోసం (ఉదాహరణకు: `t2.micro` - ఇది ఫ్రీ టైర్‌లో వస్తుంది).
- _Compute Optimized (C సిరీస్):_ ఎక్కువ CPU పవర్ కావలసిన హెవీ కోడింగ్ లేదా డేటా ప్రాసెసింగ్ కోసం.
- _Memory Optimized (R సిరీస్):_ RAM ఎక్కువగా కావలసిన డేటాబేస్ పనుల కోసం.

- **AMIs (Amazon Machine Images):** ఇది మీ కంప్యూటర్ యొక్క **OS బ్లూప్రింట్ (లేదా పెన్‌డ్రైవ్ ఓఎస్ లాంటిది)**. సర్వర్ క్రియేట్ చేసేటప్పుడు అందులో Ubuntu ఉండాలా, Amazon Linux ఉండాలా, లేక Windows ఉండాలా అనేది AMI ద్వారా సెలెక్ట్ చేసుకుంటాం.
- **Launch Templates:** ఒక సర్వర్‌కి ఏ AMI ఉండాలి, ఎంత RAM ఉండాలి, ఏ సెక్యూరిటీ గ్రూప్ ఉండాలి అనే వివరాలన్నింటినీ ఒక టెంప్లేట్ (పెద్ద ప్లాన్) లాగా సేవ్ చేసి పెట్టుకోవడం. దీనివల్ల ప్రతిసారీ కొత్త సర్వర్ క్రియేట్ చేసేటప్పుడు మళ్లీ మళ్లీ సెట్టింగ్స్ మార్చాల్సిన పనిలేదు.
- **Auto Scaling (ఆటో స్కేలింగ్):** ఒకేసారి వెబ్‌సైట్‌కి ట్రాఫిక్ పెరిగితే (ఉదాహరణకు బిగ్ బిలియన్ డేస్ సేల్స్ లాంటివి), ఆటోమేటిక్‌గా సర్వర్ల సంఖ్య పెరుగుతుంది (Scale-out). ట్రాఫిక్ తగ్గిపోగానే సర్వర్లు తగ్గిపోతాయి (Scale-in). దీనివల్ల వెబ్‌సైట్ డౌన్ అవ్వదు, పైగా బిల్లు కూడా సేవ్ అవుతుంది.
- **Placement Groups:** మన సర్వర్లను డేటా సెంటర్‌లో ఏ పద్ధతిలో అమర్చాలో చెప్పే స్ట్రాటజీ.
- _Cluster:_ సర్వర్లన్నీ పక్కపక్కనే ఉంటాయి (చాలా ఫాస్ట్ నెట్‌వర్క్ స్పీడ్ కోసం).
- _Spread:_ ఒక సర్వర్ పోయినా ఇంకోటి సేఫ్‌గా ఉండటానికి వేర్వేరు రాక్స్‌లో (Hardware racks) సర్వర్లను ఉంచుతారు.

- **Elastic IP:** మీ EC2 సర్వర్‌ని స్టాప్ చేసి స్టార్ట్ చేస్తే, దాని పబ్లిక్ ఐపీ అడ్రస్ మారిపోతుంది. అలా మారకుండా ఉండే ఒక **శాశ్వతమైన ఐపీ అడ్రస్ (Static IP)** నే Elastic IP అంటారు.

## 2. Serverless (AWS Lambda)

డెవలపర్లకు వరప్రసాదం లాంటిది ఈ **AWS Lambda**. ఇక్కడ మనం సర్వర్లు కొనక్కర్లేదు, OS అప్‌డేట్ చేయక్కర్లేదు, మెయింటెనెన్స్ అసలే లేదు. కేవలం మన కోడ్ (Node.js, Python, etc.) రాసి అప్‌లోడ్ చేస్తే చాలు.

- **Serverless అంటే ఏంటి?:** సర్వర్లు లేవని కాదు, కానీ ఆ సర్వర్ల మేనేజ్‌మెంట్ అంతా AWS చూసుకుంటుంది. కోడ్ రన్ అయిన సమయానికి (మిల్లీసెకన్లలో) మాత్రమే మనం బిల్లు కడతాం.
- **Event Sources:** ల్యాంబ్డా ఫంక్షన్ ఎప్పుడు రన్ అవ్వాలి? ఎవరైనా API రిక్వెస్ట్ పంపినప్పుడా (`API Gateway`), ఎవరైనా S3 లో ఇమేజ్ అప్‌లోడ్ చేసినప్పుడా? అలా ల్యాంబ్డాని ట్రిగ్గర్ చేసే వాటిని Event Sources అంటారు.
- **Environment Variables:** డేటాబేస్ లింక్స్ లేదా పోర్ట్ నంబర్స్ లాంటి కాన్ఫిగరేషన్స్‌ని కోడ్‌లో రాయకుండా, బయట వేరియబుల్స్ లాగా స్టోర్ చేయడం.
- **Layers:** మీ Node.js కోడ్‌లో `node_modules` (ఉదాహరణకు `lodash` లేదా `axios`) సైజ్ పెద్దగా ఉంటే, వాటిని విడిగా ఒక 'Layer' లాగా అప్‌లోడ్ చేయవచ్చు. దీనివల్ల మీ మెయిన్ ల్యాంబ్డా కోడ్ సైజ్ చిన్నగా ఉండి, ఫాస్ట్‌గా రన్ అవుతుంది.
- **Cold Start:** ల్యాంబ్డా ఫంక్షన్ చాలా సేపటి వరకు వాడకుండా ఉండి, అకస్మాత్తుగా మొదటిసారి రన్ అయినప్పుడు, AWS వెనుకల బ్యాక్‌గ్రౌండ్‌లో సర్వర్‌ని రెడీ చేయడానికి 1-2 సెకన్లు టైమ్ తీసుకుంటుంది. దీనినే "Cold Start" అంటారు. ఆ తర్వాత వచ్చే రిక్వెస్ట్‌లు చాలా ఫాస్ట్‌గా (Warm Start) రన్ అవుతాయి.
- **Lambda Versions & Aliases:**
- _Versions:_ మనం కోడ్ మార్చినప్పుడల్లా `Version 1`, `Version 2` అని కోడ్ స్నాప్‌షాట్స్ సేవ్ చేసుకోవడం.
- _Aliases:_ ఒక వెర్షన్‌కి నిక్ నేమ్ ఇవ్వడం. ఉదాహరణకు Version 1 ని `PROD` అని, Version 2 ని `DEV` అని పిలవడం. దీనివల్ల కోడ్ మార్చకుండానే ట్రాఫిక్‌ని కంట్రోల్ చేయవచ్చు.

## 3. Containers (ECS, EKS, Fargate)

డెవలపర్‌గా మీకు **Docker** ఐడియా ఉండే ఉంటుంది. మన అప్లికేషన్‌ను కంటైనరైజ్ (Dockerize) చేశాక, వాటిని AWS లో మేనేజ్ చేయడానికి ఈ సర్వీసులు వాడుతాం.

- **Amazon ECS (Elastic Container Service):** ఇది AWS వారి సొంత కంటైనర్ మేనేజ్‌మెంట్ టూల్ (Orchestration tool). డాకర్ కంటైనర్లను సులభంగా రన్ చేయడానికి వాడుతారు.
- **Amazon EKS (Elastic Kubernetes Service):** ఒకవేళ మీ కంపెనీ ఓపెన్-సోర్స్ **Kubernetes** వాడుతుంటే, దానిని AWS లో రన్ చేయడానికి EKS వాడుతారు.
- **AWS Fargate:** కంటైనర్స్ రన్ చేయడానికి మనకు కింద ఒక EC2 సర్వర్ కావాలి కదా! కానీ Fargate వాడితే, ఆ కింద ఉండే EC2 సర్వర్లను కూడా మనం మేనేజ్ చేయాల్సిన పనిలేదు. ఇది **"Serverless కోసం కంటైనర్స్"** లాంటిది. కేవలం కంటైనర్ ఇమేజ్ ఇస్తే, Fargate దాన్ని రన్ చేస్తుంది.

---

**క్విక్ సమ్మరీ:**

- పూర్తి కంట్రోల్ ఉన్న వర్చువల్ సర్వర్ కావాలా? ➡️ **EC2**
- సర్వర్లతో సంబంధం లేకుండా కేవలం కోడ్ రన్ అవ్వాలా? ➡️ **Lambda**
- డాకర్ కంటైనర్స్ మేనేజ్ చేయాలా? ➡️ **ECS / EKS**
- సర్వర్లు లేని కంటైనర్ రన్‌టైమ్ కావాలా? ➡️ **Fargate**

# 4. Networking

కంప్యూట్ సర్వీసెస్ తర్వాత అత్యంత కీలకమైన, మరియు కొంచెం జాగ్రత్తగా అర్థం చేసుకోవాల్సిన టాపిక్ **Networking**. క్లౌడ్‌లో మన అప్లికేషన్లు, సర్వర్లు సురక్షితంగా ఒకదానితో ఒకటి ఎలా మాట్లాడుకుంటాయి, ఇంటర్నెట్‌కి ఎలా కనెక్ట్ అవుతాయి అనేది ఈ నెట్‌వర్కింగ్ నిర్ణయిస్తుంది.

దీనిని మనం రెండు భాగాలుగా చూద్దాం: **VPC (ప్రైవేట్ నెట్‌వర్క్)** మరియు **Route 53 (DNS/డొమైన్ మేనేజ్‌మెంట్)**.

---

## Part A: Amazon VPC (Virtual Private Cloud)

**VPC అంటే ఏంటి?** AWS అనే ఒక పెద్ద ల్యాండ్‌లో, మీ కంపెనీ అప్లికేషన్ల కోసం మీరు కొనుక్కునే ఒక **సొంత సురక్షితమైన సరిహద్దు (Private Network)**. మీ అనుమతి లేనిదే బయటి ప్రపంచం నుండి ఎవరూ ఈ సరిహద్దు లోపలికి రాలేరు.

- **CIDR (Classless Inter-Domain Routing):** మీ VPC నెట్‌వర్క్‌కి మీరు ఇచ్చే ఒక ఐపీ అడ్రస్ రేంజ్ (ఉదాహరణకు: `10.0.0.0/16`). దీని పరిధిలోనే మీ సర్వర్లన్నింటికీ ఐపీలు కేటాయించబడతాయి.
- **Subnets (సబ్‌నెట్స్):** మీ పెద్ద నెట్‌వర్క్‌ను చిన్న చిన్న భాగాలుగా విభజించడం.
- **Public Subnet:** ఇది ఇంటర్నెట్‌కి కనెక్ట్ అయి ఉంటుంది. ఇందులో మన వెబ్‌సైట్ ఫ్రంటెండ్ లేదా యూజర్లు నేరుగా చూసే సర్వర్లను పెడతాం.
- **Private Subnet:** ఇది ఇంటర్నెట్‌కి దొరకదు, పూర్తిగా సీక్రెట్. ఇందులో మన బ్యాకెండ్ కోడ్ (Node.js) మరియు డేటాబేస్‌లను దాచి ఉంచుతాం.

- **Route Tables (రూట్ టేబుల్స్):** మీ నెట్‌వర్క్ లోపల ట్రాఫిక్ ఏ దిశగా వెళ్ళాలి అని చెప్పే రోడ్ మ్యాప్ లేదా డైరెక్షన్ బోర్డులు.
- **Internet Gateway (IGW):** మీ పబ్లిక్ సబ్‌నెట్‌లో ఉన్న సర్వర్లు ఇంటర్నెట్‌తో మాట్లాడటానికి (లోపలికి రావడానికి, బయటకి వెళ్ళడానికి) వాడే మెయిన్ గేట్.
- **NAT Gateway & NAT Instance:** ప్రైవేట్ సబ్‌నెట్‌లో ఉన్న మీ డేటాబేస్ సర్వర్‌కి ఏదైనా సాఫ్ట్‌వేర్ అప్‌డేట్ కావాలి అనుకుందాం. దానికి ఇంటర్నెట్ కావాలి, కానీ బయటి వాళ్ళు దాన్ని యాక్సెస్ చేయకూడదు. అప్పుడు NAT Gateway వాడుతాం. ఇది లోపలి నుండి బయటికి రిక్వెస్ట్‌లను పంపుతుంది కానీ, బయటి నుండి ఎవరినీ లోపలికి రానివ్వదు. (NAT Instance అనేది పాత పద్ధతి, ఇప్పుడు అందరూ NAT Gateway వాడుతున్నారు).

### Firewalls (నెట్‌వర్క్ భద్రత)

- **Security Groups (SG):** ఇది **EC2 సర్వర్ చుట్టూ ఉండే సెక్యూరిటీ గార్డు** (Instance level). ఇది స్టేట్‌ఫుల్ (Stateful) — అంటే ఒకసారి మనం లోపలికి అనుమతిస్తే, అది బయటికి వెళ్ళేటప్పుడు మళ్ళీ తనిఖీ చేయదు.
- **Network ACL (NACL):** ఇది **సబ్‌నెట్ చుట్టూ ఉండే మెయిన్ గేట్ సెక్యూరిటీ** (Subnet level). ఇది స్టేట్‌లెస్ (Stateless) — అంటే వచ్చేటప్పుడు, పోయేటప్పుడు రెండు సార్లూ రూల్స్ చెక్ చేస్తుంది. నిర్దిష్టమైన ఐపీలను బ్లాక్ చేయడానికి NACL బెస్ట్.
- **Elastic Network Interface (ENI):** సర్వర్‌కి కనెక్ట్ చేసే ఒక వర్చువల్ నెట్‌వర్క్ కార్డ్ (ఇందులోనే ఐపీ అడ్రస్, సెక్యూరిటీ గ్రూప్స్ ఉంటాయి).

### Connecting Networks (నెట్‌వర్క్‌లను కలపడం)

- **VPC Peering:** రెండు వేర్వేరు VPCలను ఇంటర్నెట్ అవసరం లేకుండా, AWS అంతర్గత నెట్‌వర్క్ ద్వారా నేరుగా కనెక్ట్ చేయడం.
- **Transit Gateway:** మీ కంపెనీలో పదుల సంఖ్యలో VPCలు, ఆఫీస్ నెట్‌వర్క్‌లు ఉంటే... అన్నింటినీ ఒకదానికొకటి పేరింగ్ చేయడం కష్టం. Transit Gateway ఒక సెంట్రల్ హబ్ లాగా పనిచేస్తూ అన్ని నెట్‌వర్క్‌లను ఒకే చోట కనెక్ట్ చేస్తుంది.
- **AWS VPN:** మీ ఇంట్లో కూర్చుని ఆఫీస్ ప్రైవేట్ సర్వర్లను సేఫ్‌గా యాక్సెస్ చేయడానికి ఇంటర్నెట్ ద్వారా ఏర్పరచుకునే ఒక సెక్యూర్ టన్నెల్.
- **AWS Direct Connect:** ఇంటర్నెట్ వాడకుండా, మీ ఆఫీస్ డేటా సెంటర్ నుండి నేరుగా AWS డేటా సెంటర్‌కి వేసుకునే ఒక డెడికేటెడ్ ఫిజికల్ కేబుల్ కనెక్షన్ (చాలా ఖరీదైనది, అత్యంత వేగవంతమైనది).

---

## Part B: Amazon Route 53 (DNS సర్వీస్)

**Route 53 అంటే ఏంటి?** ఇది AWS వారి **DNS (Domain Name System)** సర్వీస్. మనుషులు చదివే వెబ్‌సైట్ పేర్లను (ఉదాహరణకు: `mysite.com`) కంప్యూటర్లు అర్థం చేసుకునే ఐపీ అడ్రస్‌లుగా (`192.0.2.1`) మార్చడం దీని పని.

- **Hosted Zones:** మీ డొమైన్ (e.g., `example.com`) యొక్క DNS రికార్డులన్నింటినీ భద్రపరిచే ఒక కంటైనర్ లేదా ఫోల్డర్ లాంటిది.
- **DNS Records:** \* **A Record:** డొమైన్ పేరును నేరుగా IPv4 అడ్రస్‌కి మ్యాప్ చేస్తుంది.
- **AAAA Record:** IPv6 అడ్రస్‌కి మ్యాప్ చేస్తుంది.
- **CNAME:** ఒక డొమైన్ పేరును ఇంకొక డొమైన్ పేరుకి లింక్ చేస్తుంది.
- **Alias Record:** AWS సర్వీసుల కోసం (ఉదాహరణకు CloudFront లేదా Load Balancer) AWS ప్రత్యేకంగా డిజైన్ చేసిన రికార్డ్.

- **Health Checks:** మీ సర్వర్ సరిగ్గా నడుస్తోందో లేదో Route 53 నిరంతరం గమనిస్తూ ఉంటుంది. ఒకవేళ సర్వర్ డౌన్ అయితే అలర్ట్ చేస్తుంది.
- **Failover Routing (ఫెయిల్ ఓవర్ రూటింగ్):** మీ మెయిన్ సర్వర్ (Primary) డౌన్ అయిపోయినప్పుడు, Route 53 ఆటోమేటిక్‌గా మీ వెబ్‌సైట్ ట్రాఫిక్‌ను బ్యాకప్ సర్వర్ (Secondary) వైపు తిప్పుతుంది. దీనివల్ల యూజర్లకు వెబ్‌సైట్ ఎప్పుడూ లైవ్‌లోనే కనిపిస్తుంది.

---

**క్విక్ సమ్మరీ:**

- క్లౌడ్‌లో మీ సొంత ప్రైవేట్ ఏరియా ➡️ **VPC**
- పబ్లిక్ యాక్సెస్ కోసం ➡️ **Public Subnet & Internet Gateway**
- ప్రైవేట్ సర్వర్లకు సేఫ్ ఇంటర్నెట్ కోసం ➡️ **NAT Gateway**
- ఇన్‌స్టాన్స్ ఫైర్‌వాల్ ➡️ **Security Group**
- డొమైన్ మరియు ట్రాఫిక్ రూటింగ్ ➡️ **Route 53**

# 5. Storage

నెట్‌వర్కింగ్ తర్వాత మనం నేర్చుకోవాల్సిన అత్యంత ముఖ్యమైన, మరియు డెవలపర్లు ప్రతిరోజూ వాడే విభాగం — **Storage (డేటా నిల్వ సర్వీసులు)**.

మనం కంప్యూటర్ వాడేటప్పుడు పెన్‌డ్రైవ్, హార్డ్ డిస్క్, గూగుల్ డ్రైవ్ ఎలా వాడతామో... AWS లో కూడా డేటా రకాన్ని బట్టి 4 రకాల స్టోరేజ్ ఆప్షన్లు ఉన్నాయి. వాటిని చాలా సులభంగా అర్థం చేసుకుందాం:

---

## 1. Object Storage (Amazon S3)

**Amazon S3 (Simple Storage Service)** అంటే క్లౌడ్‌లో అపరిమితమైన (Unlimited) సైజు గల ఫైల్స్ దాచుకునే ఒక గూగుల్ డ్రైవ్ లేదా డ్రాప్‌బాక్స్ లాంటిది. ఇక్కడ ఫైల్స్‌ని "Objects" అని పిలుస్తారు. మీరు ఇమేజెస్, వీడియోలు, పిడిఎఫ్ ఫైల్స్, లేదా బ్యాకప్స్ ఏవైనా ఇందులో దాచుకోవచ్చు.

- **Buckets (బకెట్లు):** S3 లో క్రియేట్ చేసే మెయిన్ ఫోల్డర్‌ని "Bucket" అంటారు. ఈ బకెట్ పేరు ప్రపంచవ్యాప్తంగా వినూత్నంగా (Globally Unique) ఉండాలి. అంటే ఒకరు వాడిన బకెట్ పేరు ఇంకొకరు వాడలేరు.
- **Object Versioning (వెర్షనింగ్):** ఒకే పేరుతో ఉన్న ఫైల్‌ని మళ్లీ అప్‌లోడ్ చేసినప్పుడు పాత ఫైల్ డిలీట్ అయిపోకుండా `Version 1`, `Version 2` అని హిస్టరీని దాచి ఉంచుతుంది. ఒకవేళ పొరపాటున ఫైల్ డిలీట్ చేసినా, పాత వెర్షన్‌ని తిరిగి తెచ్చుకోవచ్చు (Rollback).
- **Lifecycle Policies:** ఫైల్స్ పాతబడే కొద్దీ ఆటోమేటిక్‌గా వాటిని తక్కువ ఖర్చుండే స్టోరేజ్‌కి మార్చడం లేదా డిలీట్ చేయడం.
- _ఉదాహరణ:_ "ఒక ఫైల్ అప్‌లోడ్ అయ్యాక 30 రోజులు S3 లో ఉండాలి, 31వ రోజు నుండి ఆటోమేటిక్‌గా గ్లేసియర్ (Archive) కి వెళ్ళిపోవాలి, ఒక సంవత్సరం తర్వాత డిలీట్ అయిపోవాలి" అని రూల్స్ పెట్టడం.

- **Storage Classes (స్టోరేజ్ రకాలు):** మన వాడకాన్ని బట్టి డబ్బులు ఆదా చేయడానికి రకరకాల క్లాసెస్ ఉన్నాయి:
- _S3 Standard:_ ప్రతిరోజూ వాడే ఫైల్స్ కోసం (ఖర్చు ఎక్కువ, వేగం ఎక్కువ).
- _S3 Standard-IA (Infrequent Access):_ ఎప్పుడో ఒకసారి వాడే ఫైల్స్ కోసం (ఉదాహరణకు నెలకొకసారి చూసే రిపోర్ట్స్).
- _S3 Glacier:_ అస్సలు వాడని, కానీ దాచి ఉంచాల్సిన పాత రికార్డుల కోసం (ఖర్చు చాలా తక్కువ).

- **Encryption:** మన ఫైల్స్‌ని ఇతరులు దొంగిలించకుండా లాక్ చేయడం (S3 లో డేటా-అట్-రెస్ట్ మరియు డేటా-ఇన్-ట్రాన్సిట్ ఎన్‌క్రిప్షన్స్ సులభంగా చేయవచ్చు).
- **Bucket Policies:** మీ బకెట్‌ని ఎవరెవరు యాక్సెస్ చేయవచ్చు అని రాసే JSON రూల్స్. (ఉదాహరణకు: కేవలం మీ ఆఫీస్ ఐపీ అడ్రస్ నుండి మాత్రమే ఫైల్స్ డౌన్లోడ్ అవ్వాలి అని కండిషన్ పెట్టడం).
- **Static Website Hosting:** మన దగ్గర React, Angular లేదా నార్మల్ HTML/CSS/JS తో బిల్డ్ చేసిన స్టాటిక్ వెబ్‌సైట్ ఉంటే, ఎటువంటి సర్వర్ (EC2) లేకుండా కేవలం S3 బకెట్ ద్వారానే వెబ్‌సైట్‌ని ప్రపంచానికి లైవ్ చేయవచ్చు.
- **Multipart Upload:** పెద్ద సైజు ఫైల్స్ (ఉదాహరణకు 100MB కంటే ఎక్కువ ఉన్న వీడియోలు) అప్‌లోడ్ చేసేటప్పుడు, ఆ ఫైల్‌ని చిన్న చిన్న ముక్కలుగా విడగొట్టి ఒకేసారి (Parallel గా) అప్‌లోడ్ చేయడం. దీనివల్ల నెట్‌వర్క్ ఫెయిల్ అయినా అప్‌లోడ్ ఆగిపోదు, వేగంగా అవుతుంది.

---

## 2. Block Storage (Amazon EBS)

**EBS (Elastic Block Store)** అంటే మన **ల్యాప్‌టాప్ లో ఉండే హార్డ్ డిస్క్ (HDD/SSD)** లాంటిది.

- ఇది కేవలం ఒక **EC2 సర్వర్‌కి మాత్రమే** కనెక్ట్ చేయగలము.
- ఇందులో మీరు ఆపరేటింగ్ సిస్టమ్ (OS) ఇన్‌స్టాల్ చేయవచ్చు, మీ Node.js కోడ్ ఫైల్స్ పెట్టవచ్చు, డేటాబేస్ రన్ చేయవచ్చు. సర్వర్ ఆపినా ఇందులో డేటా సేఫ్‌గా ఉంటుంది.

---

## 3. File Storage (Amazon EFS)

**EFS (Elastic File System)** అనేది ఒక **నెట్‌వర్క్ షేర్డ్ డ్రైవ్ (Shared Folder)** లాంటిది.

- EBS ని కేవలం ఒకే ఒక సర్వర్‌కి కనెక్ట్ చేయగలం. కానీ EFS ని **ఒకే సమయంలో వందలాది EC2 సర్వర్లకు** కనెక్ట్ చేయవచ్చు.
- _ఉదాహరణ:_ మీ దగ్గర 5 EC2 సర్వర్లు ఉన్నాయి, ఆ ఐదింటికీ ఒకే కామన్ ఫోల్డర్ లోని ఫైల్స్ యాక్సెస్ కావాలి అనుకుంటే EFS బెస్ట్ ఛాయిస్. ఇది ఆటోమేటిక్‌గా స్కేల్ అవుతుంది (డేటా పెరిగే కొద్దీ సైజ్ పెరుగుతుంది).

---

## 4. Archive Storage (Amazon S3 Glacier)

ఇది S3 లోని ఒక భాగమే అయినప్పటికీ, కేవలం **డేటా బ్యాకప్ అండ్ ఆర్కైవింగ్** కోసం ప్రత్యేకంగా డిజైన్ చేయబడింది.

- కంపెనీల పాత అకౌంట్స్ డేటా, మెడికల్ రికార్డ్స్, లేదా లీగల్ డాక్యుమెంట్స్ లాంటి వాటిని సంవత్సరాల తరబడి దాచడానికి వాడుతారు.
- దీని బిల్లింగ్ చాలా చాలా తక్కువ. కానీ, దాచిన ఫైల్స్ మీకు వెంటనే తిరిగి రావు (Retrieval time). ఫైల్ తిరిగి కావాలని రిక్వెస్ట్ పెడితే, మీరు ఎంచుకున్న ప్లాన్‌ని బట్టి కొన్ని నిమిషాల నుండి కొన్ని గంటల సమయం పడుతుంది.

---

**క్విక్ సమ్మరీ:**

- ఇమేజెస్, వీడియోలు, స్టాటిక్ వెబ్‌సైట్స్ కోసం ➡️ **S3**
- ఒకే ఒక EC2 సర్వర్‌కి హార్డ్ డిస్క్ లాగా వాడటానికి ➡️ **EBS**
- చాలా సర్వర్ల మధ్య ఫైల్స్ షేర్ చేసుకోవడానికి ➡️ **EFS**
- ఎప్పుడో కానీ వాడని పాత డేటాను తక్కువ ఖర్చుతో దాచడానికి ➡️ **Glacier**

# 6. Database Services

స్టోరేజ్ తర్వాత బ్యాకెండ్ డెవలపర్‌గా మీకు అత్యంత ఇష్టమైన, మరియు మీరు ప్రతిరోజూ వర్క్ చేసే విభాగం — **Database Services (డేటాబేస్ సర్వీసెస్)**.

SQL మరియు NoSQL డేటాబేస్‌లను మనం సొంతంగా EC2 సర్వర్‌లో ఇన్‌స్టాల్ చేసి మెయింటైన్ చేయవచ్చు, కానీ దానికి ప్యాచ్‌లు వేయడం, బ్యాకప్‌లు తీయడం పెద్ద తలనొప్పి. అందుకే AWS ఈ పనులన్నీ ఆటోమేట్ చేస్తూ మేనేజ్డ్ డేటాబేస్ సర్వీసులను అందిస్తోంది. వాటిని రకాలను బట్టి సులభంగా అర్థం చేసుకుందాం.

---

## 1. Relational Database (Amazon RDS)

**Amazon RDS (Relational Database Service)** అనేది క్లౌడ్‌లో సాంప్రదాయ **SQL (Relational) డేటాబేస్‌లను** రన్ చేయడానికి వాడే ఒక అద్భుతమైన సర్వీస్.

మీరు టేబుల్స్, రోస్ (Rows), కాలమ్స్ (Columns) మరియు ఫారిన్ కీస్ (Foreign Keys) రూపంలో డేటాను దాచడానికి దీనిని వాడుతారు. RDS ద్వారా AWS ఈ క్రింది పాపులర్ డేటాబేస్ ఇంజిన్లను ఆఫర్ చేస్తుంది:

- **PostgreSQL / MySQL / MariaDB / SQL Server / Oracle:** ఇవన్నీ మనకు తెలిసిన ఓపెన్-సోర్స్ మరియు కమర్షియల్ డేటాబేస్‌లే.
- **Amazon Aurora (ఇది కూడా ఒక ఇంజిన్):** ఇది AWS ప్రత్యేకంగా డిజైన్ చేసిన క్లౌడ్-నేటివ్ డేటాబేస్. ఇది నార్మల్ MySQL కంటే 5 రెట్లు, PostgreSQL కంటే 3 రెట్లు ఎక్కువ స్పీడ్‌తో పనిచేస్తుంది మరియు ఆటోమేటిక్‌గా స్కేల్ అవుతుంది.

> **RDS లాభం:** బ్యాకప్స్ ఆటోమేటిక్‌గా జరిగిపోతాయి. ఒకవేళ మెయిన్ డేటాబేస్ డౌన్ అయినా, క్షణాల్లో ఇంకొక బ్యాకప్ డేటాబేస్ లైవ్‌లోకి వస్తుంది (Multi-AZ deployment).

---

## 2. NoSQL Database (Amazon DynamoDB)

డెవలపర్లకు, ముఖ్యంగా Serverless (AWS Lambda) వాడుతున్నప్పుడు అత్యంత పాపులర్ అయిన NoSQL డేటాబేస్ **Amazon DynamoDB**.

- **Key-Value Store:** ఇది టేబుల్స్ లాగా కాకుండా JSON డాక్యుమెంట్స్ లాగా `Key-Value` రూపంలో డేటాను స్టోర్ చేస్తుంది.
- **Serverless & Fully Managed:** దీనికి సర్వర్లు ఉండవు. మీరు ఎంత డేటా అయినా వేసుకోవచ్చు, ఇది ఆటోమేటిక్‌గా సైజ్ పెంచుకుంటుంది.
- **Millisecond Latency:** ఎంత పెద్ద మొత్తంలో డేటా ఉన్నా (లాక్స్ లేదా కోట్లలో రికార్డ్స్ ఉన్నా), కేవలం సింగిల్-డిజిట్ మిల్లీసెకండ్లలోనే డేటాను రిట్రీవ్ చేస్తుంది. షాపింగ్ కార్ట్స్, యూజర్ ప్రొఫైల్స్, గేమ్ లీడర్‌బోర్డ్స్ వంటి వాటికి ఇది బెస్ట్.

---

## 3. In-Memory Cache (Amazon ElastiCache)

బ్యాకెండ్ డెవలపర్‌గా మన అప్లికేషన్ చాలా ఫాస్ట్‌గా రెస్పాండ్ అవ్వాలని కోరుకుంటాం. ప్రతిసారీ మెయిన్ డేటాబేస్ (RDS) కి వెళ్ళి డేటా తేవడం వల్ల టైమ్ పడుతుంది మరియు డేటాబేస్ పై లోడ్ పెరుగుతుంది. అందుకే **ElastiCache** ని వాడుతాం.

ఇది డేటాను హార్డ్ డిస్క్‌లో కాకుండా **RAM (In-Memory)** లో దాచుకుంటుంది. కాబట్టి డేటా రెప్పపాటులో (Microseconds లో) లోడ్ అవుతుంది. ఇందులో రెండు రకాలు ఉన్నాయి:

- **Redis:** ఇది కేవలం క్యాషింగ్ మాత్రమే కాకుండా, పబ్-సబ్ (Pub/Sub), డేటా స్ట్రక్చర్స్, మరియు డేటా పర్సిస్టెన్స్ (డేటా పోకుండా దాచడం) కి సపోర్ట్ చేస్తుంది. (ప్రస్తుతం మార్కెట్లో ఇదే ఎక్కువ వాడుతున్నారు).
- **Memcached:** ఇది చాలా సింపుల్, ప్యూర్ క్యాషింగ్ మోడల్. పెద్దగా ఫీచర్స్ అవసరం లేని చోట వాడుతారు.

---

## 4. Data Warehouse (Amazon Redshift)

మన అప్లికేషన్‌లో ప్రతిరోజూ జరిగే లావాదేవీల (Transactions) కోసం RDS వాడుతాం. కానీ, గత 5-10 సంవత్సరాల డేటాను విశ్లేషించి (Analytics/BI Reports), కంపెనీ బిజినెస్ ఎలా ఉందో తెలుసుకోవడానికి నార్మల్ డేటాబేస్ సరిపోదు.

**Amazon Redshift** అనేది ఒక **Data Warehouse** సర్వీస్. ఇది పెటాబైట్ల (Petabytes) కొద్దీ ఉన్న భారీ డేటాను చాలా వేగంగా అనలైజ్ చేసి పెద్ద పెద్ద రిపోర్ట్స్, గ్రాఫ్స్ జనరేట్ చేయడానికి వాడుతారు. (ఇది కేవలం అనలిటిక్స్ కోసం మాత్రమే, డైరెక్ట్ యాప్ లాగిన్స్ కోసం కాదు).

---

## 5. Database Migration (AWS DMS)

మీ కంపెనీ ఆఫీస్ (On-premise) లో ఒక పెద్ద డేటాబేస్ సర్వర్ ఉంది అనుకుందాం. ఇప్పుడు ఆ డేటాను ఎటువంటి ఇబ్బంది లేకుండా, అప్లికేషన్ ఆగిపోకుండా (Zero Downtime) AWS క్లౌడ్‌లోకి మార్చాలి (Migrate చేయాలి).

దానికోసం **AWS DMS (Database Migration Service)** వాడుతాం. ఇది బ్యాక్‌గ్రౌండ్‌లో మీ పాత డేటాబేస్ నుండి కొత్త క్లౌడ్ డేటాబేస్‌కి డేటాను సురక్షితంగా మరియు లైవ్‌గా ట్రాన్స్‌ఫర్ చేస్తుంది.

---

**క్విక్ సమ్మరీ:**

- రిలేషనల్/SQL డేటాబేస్ (MySQL, Postgres) కోసం ➡️ **Amazon RDS**
- సూపర్ ఫాస్ట్, సర్వర్‌లెస్ NoSQL కోసం ➡️ **Amazon DynamoDB**
- అప్లికేషన్ స్పీడ్ పెంచడానికి (RAM స్టోరేజ్) ➡️ **Amazon ElastiCache (Redis)**
- భారీ డేటా విశ్లేషణ (Analytics/Reports) కోసం ➡️ **Amazon Redshift**
- డేటాబేస్‌ను క్లౌడ్‌లోకి మార్చడానికి ➡️ **AWS DMS**

# 7. Load Balancing

డేటాబేస్ సర్వీసెస్ తర్వాత మనం నేర్చుకోబోయే టాపిక్ — **Load Balancing (లోడ్ బ్యాలెన్సింగ్)**.

సాఫ్ట్‌వేర్ ఆర్కిటెక్చర్‌లో ఇది చాలా ముఖ్యమైన కాన్సెప్ట్. మీ వెబ్‌సైట్‌కి వచ్చే యూజర్ల ట్రాఫిక్‌ను ఏ ఒక్క సర్వర్‌పైనే భారం పడకుండా, వెనుక ఉన్న అన్ని సర్వర్లకు సమానంగా పంచే ఒక **ట్రాఫిక్ పోలీస్** లాంటిదే ఈ **Elastic Load Balancing (ELB)**.

దీని గురించి పూర్తి వివరాలను తెలుగులో సులభంగా అర్థం చేసుకుందాం:

---

## Elastic Load Balancing (ELB) అంటే ఏంటి?

మీ అప్లికేషన్‌ను కేవలం ఒకే ఒక EC2 సర్వర్‌లో రన్ చేస్తున్నారు అనుకుందాం. ఒకేసారి లక్ష మంది యూజర్లు వెబ్‌సైట్‌కి వస్తే ఆ ఒక్క సర్వర్ క్రాష్ అయిపోతుంది. అందుకే మనం వెనుక 3 లేదా 4 సర్వర్లను పెడతాం.

మరి వచ్చే యూజర్లని ఏ సర్వర్‌కి పంపాలో ఎవరు నిర్ణయిస్తారు? అక్కడే ELB రంగంలోకి వస్తుంది. ఇది వచ్చే ప్రతి రిక్వెస్ట్‌ని చెక్ చేసి, ఖాళీగా ఉన్న లేదా తక్కువ లోడ్ ఉన్న సర్వర్‌కి పంపుతుంది.

AWS లో ప్రధానంగా **మూడు రకాల లోడ్ బ్యాలెన్సర్లు** ఉన్నాయి:

### 1. Application Load Balancer (ALB)

డెవలపర్లు అత్యధికంగా వాడేది ఈ ALB. ఇది నెట్‌వర్కింగ్ యొక్క **Layer 7 (Application Layer)** లో పనిచేస్తుంది. అంటే ఇది HTTP మరియు HTTPS ట్రాఫిక్‌ను (URL లింక్స్ మరియు హెడర్స్) చదవగలదు.

- **Routing Rules (లింక్ ఆధారిత రూటింగ్):** మీరు వచ్చే యుఆర్ఎల్ బట్టి ట్రాఫిక్‌ను వేర్వేరు సర్వర్లకు పంపవచ్చు.
- _ఉదాహరణ:_ `mysite.com/users` అని వస్తే యూజర్ సర్వర్లకి, `mysite.com/orders` అని వస్తే ఆర్డర్ సర్వర్లకి ట్రాఫిక్‌ను విడగొట్టవచ్చు. Microservices ఆర్కిటెక్చర్‌కి ఇది బెస్ట్ ఛాయిస్.

### 2. Network Load Balancer (NLB)

ఇది నెట్‌వర్కింగ్ యొక్క **Layer 4 (Transport Layer)** లో పనిచేస్తుంది. ఇది కేవలం IP అడ్రస్ మరియు Port నంబర్స్ (TCP, UDP ప్రొటోకాల్స్) మాత్రమే చూస్తుంది, లోపల ఉన్న HTTP డేటాను చదవదు.

- **Extreme Performance:** ఇది సెకనుకు మిలియన్ల కొద్దీ రిక్వెస్ట్‌లను చాలా వేగంగా (Ultra-low latency) హ్యాండిల్ చేయగలదు. గేమింగ్ యాప్స్ లేదా లైవ్ స్ట్రీమింగ్ లాంటి భారీ నెట్‌వర్క్ స్పీడ్ కావాల్సిన చోట NLB వాడుతారు.

### 3. Gateway Load Balancer (GWLB)

ఇది క్లౌడ్‌లో మీ అప్లికేషన్‌కి వచ్చే ట్రాఫిక్‌ను థర్డ్-పార్టీ వర్చువల్ అప్లయన్సెస్ (ఉదాహరణకు: Firewalls, Intrusion Detection Systems) ద్వారా చెక్ చేయించడానికి వాడుతారు.

- ముందుగా వచ్చే ట్రాఫిక్‌ అంతా GWLB కి వెళ్తుంది. అది ఫైర్‌వాల్ సర్వర్‌కి పంపి, డేటా సేఫ్ అని తేలితేనే మీ మెయిన్ అప్లికేషన్ సర్వర్‌కి అనుమతిస్తుంది. (ఇది సెక్యూరిటీ పరంగా చాలా హెవీగా ఉండే పెద్ద కంపెనీలలో వాడుతారు).

---

## Core Features (ముఖ్యమైన ఫీచర్లు)

- **Health Checks (హెల్త్ చెక్స్):** లోడ్ బ్యాలెన్సర్ వెనుక ఉన్న సర్వర్లు సరిగ్గా పని చేస్తున్నాయో లేదో నిరంతరం (`/health` లేదా `/` రూట్‌కి రిక్వెస్ట్ పంపి) గమనిస్తూ ఉంటుంది. ఒకవేళ ఏదైనా సర్వర్ డౌన్ అయితే (Unhealthy), ఆటోమేటిక్‌గా ట్రాఫిక్‌ను ఆ సర్వర్‌కి పంపడం ఆపేసి, మిగిలిన బాగున్న సర్వర్లకి (Healthy) మళ్ళిస్తుంది. మళ్లీ ఆ సర్వర్ బాగుపడ్డాక ట్రాఫిక్‌ని పంపుతుంది.
- **Sticky Sessions (స్టోర్డ్ సెషన్స్ - Session Affinity):** సాధారణంగా లోడ్ బ్యాలెన్సర్ మొదటి రిక్వెస్ట్‌ని Server-1 కి, రెండో రిక్వెస్ట్‌ని Server-2 కి పంపుతుంది. కానీ, యూజర్ వెబ్‌సైట్‌లో లాగిన్ అయినప్పుడు వారి సెషన్ డేటా (Session Data) Server-1 లో సేవ్ అయింది అనుకుందాం. తదుపరి క్లిక్‌కి యూజర్ రిక్వెస్ట్ Server-2 కి వెళ్తే లాగౌట్ అయిపోయే ప్రమాదం ఉంది.
- **పరిష్కారం:** Sticky Sessions ఆన్ చేస్తే, ఒక నిర్దిష్టమైన యూజర్ యొక్క అన్ని రిక్వెస్ట్‌లను (ఆ సెషన్ ముగిసేవరకు) ఒకే సర్వర్‌కి (Server-1 కే) వెళ్లేలా లోడ్ బ్యాలెన్సర్ లాక్ చేస్తుంది.

---

**క్విక్ సమ్మరీ:**

- HTTP/HTTPS మరియు URL ఆధారిత రూటింగ్ కోసం ➡️ **ALB (Application)**
- కోట్లాది రిక్వెస్ట్‌లు మరియు సూపర్ నెట్‌వర్క్ స్పీడ్ కోసం ➡️ **NLB (Network)**
- థర్డ్-పార్టీ ఫైర్‌వాల్స్ మేనేజ్ చేయడం కోసం ➡️ **GWLB (Gateway)**
- సర్వర్ల ఆరోగ్యాన్ని పర్యవేక్షించడం ➡️ **Health Checks**
- ఒకే యూజర్‌ని ఒకే సర్వర్‌కి కనెక్ట్ చేసి ఉంచడం ➡️ **Sticky Sessions**

# 8. CDN

లోడ్ బ్యాలెన్సింగ్ తర్వాత మనం నేర్చుకోబోయే టాపిక్ — **CDN (Content Delivery Network)**. AWS లో దీనిని **Amazon CloudFront** అని పిలుస్తారు.

సాఫ్ట్‌వేర్ డెవలపర్‌గా మీ వెబ్‌సైట్ లేదా మొబైల్ యాప్ ప్రపంచవ్యాప్తంగా ఉన్న యూజర్లకు రెప్పపాటులో (సూపర్ ఫాస్ట్‌గా) ఓపెన్ అవ్వడానికి ఈ సర్వీస్ చాలా ముఖ్యం. దీని వివరాలను తెలుగులో సులభంగా చూద్దాం:

---

## Amazon CloudFront అంటే ఏంటి?

మీ అప్లికేషన్ సర్వర్ మరియు S3 బకెట్ ముంబై (Mumbai) రీజియన్‌లో ఉన్నాయి అనుకుందాం. అమెరికా (USA) లో ఉన్న ఒక యూజర్ మీ వెబ్‌సైట్ ఓపెన్ చేస్తే, ఆ రిక్వెస్ట్ సముద్రాలు దాటి ముంబై సర్వర్‌కి వచ్చి, ఇక్కడి నుండి ఇమేజెస్ లేదా వీడియోలు మళ్లీ అమెరికా వెళ్ళాలి. దీనివల్ల వెబ్‌సైట్ చాలా స్లోగా లోడ్ అవుతుంది (Latency ఎక్కువ ఉంటుంది).

**CloudFront** దీనికి అద్భుతమైన పరిష్కారం. ఇది ప్రపంచవ్యాప్తంగా ఉన్న **Edge Locations** (మనం ఫండమెంటల్స్‌లో చెప్పుకున్నాం కదా) వాడుకుని, మీ వెబ్‌సైట్ లోని ఇమేజెస్, వీడియోలు, HTML/CSS/JS ఫైల్స్‌ని యూజర్‌కి దగ్గరగా ఉన్న లొకేషన్‌లో కాపీ చేసి పెడుతుంది.

> **ఉదాహరణ:** మనం ప్రతిసారీ మెయిన్ ఫ్యాక్టరీ కి వెళ్లి వస్తువులు కొనకుండా, మన ఇంటి పక్కన ఉన్న కిరాణా షాప్ నుండి కొనుక్కున్నంత ఈజీగా డేటా లోడ్ అవుతుంది.

---

## Core Concepts (ముఖ్యమైన కాన్సెప్ట్‌లు)

### 1. Caching (క్యాషింగ్) & Cache Policies

మొదటిసారి యూజర్ రిక్వెస్ట్ చేసినప్పుడు ఫైల్ మెయిన్ సర్వర్ (Origin) నుండి వస్తుంది. అలా వచ్చేటప్పుడు CloudFront దాన్ని తన Edge Location లో దాచుకుంటుంది (Cache చేస్తుంది). తదుపరి వచ్చే యూజర్లందరికీ మెయిన్ సర్వర్ దాకా వెళ్లకుండా ఈ సేవ్ చేసిన కాపీనే ఇచ్చేస్తుంది.

- **Cache Policies:** ఒక ఫైల్‌ని ఎంత సమయం పాటు క్యాష్‌లోనే ఉంచాలి (TTL - Time to Live)? హెడర్స్ లేదా కుకీస్ (Cookies) బట్టి క్యాషింగ్ ఎలా మారాలి? అనే నియమాలను Cache Policies ద్వారా మనం సెట్ చేస్తాం.

### 2. Origin Access Control (OAC)

మీరు మీ వెబ్‌సైట్ ఇమేజెస్‌ని S3 బకెట్‌లో పెట్టారు మరియు CloudFront ద్వారా వాటిని డిస్ట్రిబ్యూట్ చేస్తున్నారు. కానీ ఎవరైనా స్మార్ట్ యూజర్ నేరుగా మీ S3 URL కనిపెట్టి, CloudFront కాకుండా డైరెక్ట్‌గా S3 నుండి డౌన్లోడ్ చేస్తుంటే మీకు బ్యాండ్‌విడ్త్ బిల్లు ఎక్కువ వస్తుంది.

- **పరిష్కారం:** OAC వాడితే, మీ S3 బకెట్‌ని పూర్తిగా ప్రైవేట్ చేసేయవచ్చు. కేవలం **CloudFront ద్వారా మాత్రమే** ఎవరైనా ఆ ఫైల్స్ చూడగలరు, డైరెక్ట్ S3 లింక్ ఓపెన్ చేస్తే `Access Denied` అని వస్తుంది.

### 3. Signed URLs / Signed Cookies (సెక్యూర్ లింక్స్)

మీరు ఒక పెయిడ్ కోర్సు లేదా అల్లావుద్దీన్ అద్భుత దీపం లాంటి ప్రీమియం వీడియోలను మీ వెబ్‌సైట్‌లో పెట్టారు. డబ్బులు కట్టిన యూజర్లకి మాత్రమే ఆ లింక్స్ ఓపెన్ అవ్వాలి, మిగతా వాళ్లకు అవ్వకూడదు.

- **Signed URLs:** దీని ద్వారా మనం ఒక తాత్కాలికమైన లింక్ (ఉదాహరణకు కేవలం 30 నిమిషాలు మాత్రమే పనిచేసే లింక్) క్రియేట్ చేయవచ్చు. ఆ టైమ్ దాటిపోతే ఆ లింక్ ఎవరికీ ఓపెన్ అవ్వదు. ప్రైవేట్ కంటెంట్‌ని సురక్షితంగా షేర్ చేయడానికి ఇది బెస్ట్.

### 4. Geo Restriction (జియో రెస్ట్రిక్షన్)

కొన్ని కారణాల వల్ల మీ వెబ్‌సైట్ లేదా వీడియోలు కొన్ని దేశాలలో కనిపించకూడదు అనుకుంటే (ఉదాహరణకు కొన్ని ఓవర్సీస్ మూవీస్ లేదా లీగల్ ఇష్యూస్ వల్ల), మీరు నిర్దిష్ట దేశాలను బ్లాక్ (Blacklist) చేయవచ్చు లేదా కొన్ని దేశాలకు మాత్రమే అనుమతి (Whitelist) ఇవ్వవచ్చు. దీనినే Geo Restriction అంటారు.

---

**క్విక్ సమ్మరీ:**

- ప్రపంచవ్యాప్తంగా వెబ్‌సైట్ స్పీడ్ పెంచడానికి ➡️ **CloudFront (CDN)**
- డేటాను దగ్గరలోని నగరాల్లో దాచడం ➡️ **Caching**
- S3 బకెట్‌ను దాచి, కేవలం CDN ద్వారానే అనుమతించడం ➡️ **Origin Access Control (OAC)**
- తాత్కాలిక లింక్స్ ద్వారా ప్రీమియం కంటెంట్ సేఫ్ చేయడం ➡️ **Signed URLs**
- కొన్ని దేశాల్లో వెబ్‌సైట్‌ని బ్లాక్ చేయడం ➡️ **Geo Restriction**

# 9. Messaging Services

CDN తర్వాత బ్యాకెండ్ ఆర్కిటెక్చర్‌లో మరియు మైక్రోసర్వీసెస్ (Microservices) ఒకదానితో ఒకటి మాట్లాడుకోవడానికి అత్యంత కీలకమైన విభాగం — **Messaging Services (మెసేజింగ్ సర్వీసెస్)**.

సాఫ్ట్‌వేర్ డెవలపర్‌గా, అప్లికేషన్లు ఒకదానిపై ఒకటి ఆధారపడకుండా (Decoupled గా) ఉండటానికి, బ్యాక్‌గ్రౌండ్ టాస్క్‌లను ప్రాసెస్ చేయడానికి ఈ సర్వీసెస్ చాలా ఉపయోగపడతాయి. వీటిని తెలుగులో సులభంగా అర్థం చేసుకుందాం:

---

## 1. Amazon SQS (Simple Queue Service)

**SQS అంటే ఏంటి?** ఇది ఒక **మెసేజ్ క్యూ (క్యూ లైన్)** లాంటిది. ఒక అప్లికేషన్ పంపే మెసేజ్ లేదా టాస్క్‌లను, ఇంకొక అప్లికేషన్ (రిసీవర్) వచ్చి తీసుకునే వరకు ఈ క్యూ తనలో భద్రంగా దాచి ఉంచుతుంది. దీనిని **Message Polling (Pull Model)** అంటారు.

- **Standard Queue:** ఇది డీఫాల్ట్ క్యూ. ఇది సెకనుకు అపరిమితమైన మెసేజ్‌లను తీసుకోగలదు. కానీ ఇందులో మెసేజ్‌లు మనం పంపిన ఆర్డర్‌లోనే వస్తాయని గ్యారెంటీ లేదు (Best-effort ordering), మరియు ఒక్కోసారి ఒకే మెసేజ్ రెండు సార్లు వచ్చే అవకాశం ఉంది (At-least-once delivery).
- **FIFO Queue (First-In-First-Out):** పేరులోనే ఉంది, **మొదట వచ్చిన మెసేజ్ మొదటే ప్రాసెస్ అవుతుంది**. ఇందులో ఆర్డర్ అస్సలు మారదు (Strict ordering) మరియు ఒక మెసేజ్ ఒకేసారి డెలివరీ అవుతుంది (Exactly-once processing). బ్యాంకింగ్ లావాదేవీలు లేదా ఆర్డర్ ప్రాసెస్ లాంటి చోట్ల ఇది వాడుతారు.
- **Dead Letter Queue (DLQ):** ఏదైనా ఒక మెసేజ్‌లో తప్పు కోడ్ లేదా బగ్ ఉండటం వల్ల మీ బ్యాకెండ్ సర్వర్ దాన్ని ప్రాసెస్ చేయలేక ఫెయిల్ అవుతోంది అనుకుందాం. అలా పదే పదే ఫెయిల్ అయ్యే పాడైపోయిన మెసేజ్‌లను క్యూ నుండి విడిగా తీసి ఒక ప్రత్యేకమైన చోట పడేస్తారు. దాన్నే **Dead Letter Queue** అంటారు. డెవలపర్లు తర్వాత ఆ మెసేజ్‌లను చూసి బగ్ ఎక్కడుందో ఫిక్స్ చేయవచ్చు.

---

## 2. Amazon SNS (Simple Notification Service)

**SNS అంటే ఏంటి?** ఇది **Publish/Subscribe (Pub/Sub - Push Model)** పద్ధతిలో పనిచేస్తుంది. SQS లో ఒకరు క్యూ నుండి మెసేజ్ లాక్కోవాలి, కానీ SNS లో మెసేజ్ రాగానే అది అందరికీ ఆటోమేటిక్‌గా నెట్టబడుతుంది (Push అవుతుంది).

- **Fan-out Architecture:** ఒకే మెసేజ్‌ని ఒకే సమయంలో వేర్వేరు సర్వీసులకు లేదా మనుషులకు పంపడం.
- _ఉదాహరణ:_ ఒక యూజర్ మీ వెబ్‌సైట్‌లో ప్రొడక్ట్ కొన్నాడు. వెంటనే `Order_Placed` అనే ఒక మెసేజ్ SNS కి వెళ్తుంది. SNS దాన్ని ఆటోమేటిక్‌గా:

1. యూజర్‌కి SMS పంపే సర్వీస్‌కి,
2. ఈమెయిల్ పంపే సర్వీస్‌కి,
3. ఇన్వాయిస్ జనరేట్ చేసే SQS క్యూకి... ఒకేసారి పంపేస్తుంది.

---

## 3. Amazon EventBridge (గతంలో CloudWatch Events)

**EventBridge అంటే ఏంటి?** ఇది మీ AWS అప్లికేషన్ల కోసం ఒక **సెంట్రల్ నర్వస్ సిస్టమ్ (Central Event Bus)** లాంటిది. AWS లో ఎక్కడ ఏ చిన్న మార్పు జరిగినా (Event), దాన్ని పట్టుకుని దానికి తగ్గట్టుగా వేరే సర్వీస్‌ని యాక్టివేట్ చేస్తుంది.

- **Cron Jobs / Scheduled Events:** డెవలపర్లు ప్రతిరోజూ రాత్రి 12 గంటలకి డేటాబేస్ బ్యాకప్ తీయాలన్నా, లేదా ప్రతి గంటకూ ఒక Node.js ల్యాంబ్డా ఫంక్షన్ రన్ అవ్వాలన్నా... ఆ టైమింగ్స్ (Cron expressions) ని ఇందులో సెట్ చేయవచ్చు.
- **Event-Driven:** "S3 లో ఎవరైనా ఫైల్ అప్‌లోడ్ చేస్తే ల్యాంబ్డాని రన్ చెయ్" లేదా "EC2 సర్వర్ డౌన్ అయితే ఈమెయిల్ పంపు" లాంటి రూల్స్ రాయడానికి ఇది బెస్ట్.

---

## 4. Amazon MQ

**Amazon MQ అంటే ఏంటి?** AWS లో SQS మరియు SNS లు చాలా పాపులర్, కానీ ఇవి పూర్తిగా AWS సొంత టెక్నాలజీస్.
ఒకవేళ మీ కంపెనీ క్లౌడ్‌లోకి రాకముందు నుండి తమ ఆఫీస్ సర్వర్లలో **RabbitMQ** లేదా **Apache ActiveMQ** లాంటి ఓపెన్-సోర్స్ మెసేజ్ బ్రోకర్లను వాడుతూ ఉంటే... ఆ కోడ్‌ని మార్చకుండా అలాగే క్లౌడ్‌లో రన్ చేయడానికి **Amazon MQ** ని వాడుతారు. ఇది AWS మేనేజ్ చేసే ఒక RabbitMQ/ActiveMQ క్లస్టర్.

---

**క్విక్ సమ్మరీ:**

- మెసేజ్‌లను క్యూలో దాచి, ఒకటొకటిగా ప్రాసెస్ చేయడానికి ➡️ **SQS**
- ఆర్డర్ మారకుండా పంపడానికి ➡️ **SQS FIFO**
- ఫెయిల్ అయిన మెసేజ్‌లను విడిగా దాచడానికి ➡️ **Dead Letter Queue**
- ఒకేసారి అందరికీ ఇమెయిల్, SMS, నోటిఫికేషన్స్ పుష్ చేయడానికి ➡️ **SNS**
- టైమ్ బట్టి టాస్క్‌లు రన్ చేయడానికి (Cron) లేదా ఈవెంట్స్ మేనేజ్ చేయడానికి ➡️ **EventBridge**
- పాత RabbitMQ/ActiveMQ ని క్లౌడ్‌లో వాడటానికి ➡️ **Amazon MQ**

# 10. API Services

మెసేజింగ్ సర్వీసెస్ తర్వాత బ్యాకెండ్ డెవలపర్‌గా మీకు అత్యంత ముఖ్యమైన విభాగం — **API Services**. AWS లో దీనిని మేనేజ్ చేయడానికి **Amazon API Gateway** ని వాడుతాం.

సాఫ్ట్‌వేర్ ఆర్కిటెక్చర్‌లో ఇది మీ బ్యాకెండ్ సర్వీసులన్నింటికీ ముందు నిలబడే ఒక **మెయిన్ గేట్ (Entry Gate)** లాంటిది. మీ క్లయింట్ అప్లికేషన్లు (React.js, Mobile Apps) నేరుగా మీ ల్యాంబ్డా లేదా EC2 సర్వర్లతో మాట్లాడకుండా, ఈ API Gateway ద్వారానే కనెక్ట్ అవుతాయి.

దీని వివరాలను తెలుగులో క్లియర్ గా అర్థం చేసుకుందాం:

---

## Amazon API Gateway రకాలు (API Types)

మనం క్రియేట్ చేసే అప్లికేషన్ అవసరాన్ని బట్టి API Gateway లో మూడు రకాల APIలు ఉన్నాయి:

- **REST API:** ఇది పూర్తి ఫీచర్లు ఉన్న ట్రెడిషనల్ API మోడల్. ఇందులో API కీస్, రేట్ లిమిటింగ్ (Throttling), కాంప్లెక్స్ రూటింగ్, మరియు డేటా వాలిడేషన్ లాంటి అడ్వాన్స్‌డ్ ఫీచర్స్ అన్నీ ఉంటాయి.
- **HTTP API:** ఇది REST API కి సరికొత్త, అప్‌డేటెడ్ వెర్షన్. ఇది చాలా వేగంగా (Low latency) పనిచేస్తుంది మరియు REST API తో పోలిస్తే **70% ఖర్చు తక్కువ**. ఎక్కువ అడ్వాన్స్‌డ్ ఫీచర్స్ అవసరం లేకుండా, కేవలం ల్యాంబ్డా ఫంక్షన్‌కి లేదా బ్యాకెండ్‌కి ట్రాఫిక్ పంపడానికి ఇది బెస్ట్ ఛాయిస్.
- **WebSocket API:** ఇది నార్మల్ Request-Response లాగా కాకుండా, క్లయింట్‌కి మరియు సర్వర్‌కి మధ్య **నిరంతర కనెక్షన్ (Two-way communication)** ఉంచుతుంది. చాట్ అప్లికేషన్లు, లైవ్ స్కోర్ అప్‌డేట్స్, స్టాక్ మార్కెట్ లైవ్ ఫీడ్స్ లాంటి రియల్-టైమ్ యాప్స్ బిల్డ్ చేయడానికి ఇది వాడుతాం.

---

## Core Features (ముఖ్యమైన ఫీచర్లు)

### 1. Custom Authorizers (సెక్యూరిటీ చెక్)

మీ API ని ఎవరైనా యాక్సెస్ చేసేటప్పుడు, వారు పంపిన `JWT Token` లేదా `Authorization Header` కరెక్ట్ కాదా అని చెక్ చేయాలి కదా!

- **Custom Authorizer (Lambda Authorizer):** ఒక యూజర్ API రిక్వెస్ట్ పంపగానే, API Gateway ముందుగా ఒక చిన్న ల్యాంబ్డా ఫంక్షన్‌ని రన్ చేసి టోకెన్ వాలిడేట్ చేస్తుంది. టోకెన్ కరెక్ట్ అయితేనే మెయిన్ బ్యాకెండ్ కోడ్‌కి అనుమతిస్తుంది, లేదంటే అక్కడే `401 Unauthorized` అని బ్లాక్ చేస్తుంది.

### 2. Usage Plans & API Keys (బిజినెస్ మోడల్)

మీరు ఒక API ని బిల్డ్ చేసి, దాన్ని వేరే కంపెనీలకి అమ్ముతున్నారు అనుకుందాం (SaaS Model).

- **API Keys:** ప్రతి కస్టమర్‌కి ఒక వినూత్నమైన కీ (String) ఇస్తారు. ఆ కీ ని వాడితేనే API పనిచేస్తుంది.
- **Usage Plans:** దీని ద్వారా మీరు లిమిట్స్ పెట్టవచ్చు. ఉదాహరణకు:
- _Basic Plan:_ నెలకు 1000 రిక్వెస్ట్‌లు మాత్రమే చేసుకోవచ్చు, సెకనుకు 5 రిక్వెస్ట్‌లు మించకూడదు (Throttling).
- _Premium Plan:_ నెలకు లక్ష రిక్వెస్ట్‌లు చేసుకోవచ్చు.

### 3. Stages (వర్క్‌ఫ్లో మేనేజ్‌మెంట్)

సాఫ్ట్‌వేర్ డెవలప్‌మెంట్‌లో మనం కోడ్‌ని ఒకేసారి ప్రొడక్షన్‌కి పంపం. ముందుగా టెస్ట్ చేస్తాం. API Gateway లో దీనికోసం **Stages** వాడుతాం.

- ఒకే API కి మనం `dev`, `qa`, `prod` అని వేర్వేరు స్టేజెస్ క్రియేట్ చేయవచ్చు.
- `dev` స్టేజ్ లింక్ డెవలపర్లు వాడుకుంటారు, `prod` స్టేజ్ లైవ్ లింక్ యూజర్లు వాడుకుంటారు. దీనివల్ల డెవలప్‌మెంట్‌ లో మార్పులు చేసినా లైవ్ యూజర్లకు ఎటువంటి ఇబ్బంది కలగదు.

---

**క్విక్ సమ్మరీ:**

- అన్ని బ్యాకెండ్ సర్వీసులకి ఒకే ఒక సెంట్రల్ గేట్‌వే ➡️ **API Gateway**
- తక్కువ ఖర్చుతో సూపర్ ఫాస్ట్ APIల కోసం ➡️ **HTTP API**
- చాట్ యాప్స్ మరియు రియల్-టైమ్ అప్‌డేట్స్ కోసం ➡️ **WebSocket API**
- టోకెన్స్ వాలిడేట్ చేసి సెక్యూరిటీ ఇవ్వడానికి ➡️ **Custom Authorizers**
- కస్టమర్లకి రిక్వెస్ట్ లిమిట్స్ (API Limits) పెట్టడానికి ➡️ **Usage Plans**
- Dev, QA, Prod ఎన్విరాన్‌మెంట్లు మేనేజ్ చేయడానికి ➡️ **Stages**

# 11. Monitoring & Logging

API Services తర్వాత మన అప్లికేషన్లు క్లౌడ్‌లో ఎలా పనిచేస్తున్నాయి, ఏమైనా బగ్‌లు వస్తున్నాయా, సర్వర్లు ఏమైనా డౌన్ అవుతున్నాయా అని పర్యవేక్షించడానికి వాడే అత్యంత ముఖ్యమైన విభాగం — **Monitoring & Logging (పర్యవేక్షణ మరియు లాగింగ్)**.

సాఫ్ట్‌వేర్ డెవలపర్‌గా మీ కోడ్ ప్రొడక్షన్‌లోకి వెళ్ళాక, అర్ధరాత్రి ఏదైనా ఎర్రర్ వస్తే దాన్ని వెతికి పట్టుకోవడానికి (Debugging) ఈ సర్వీసులు మీకు కంటి లాగా పనిచేస్తాయి. వీటిని తెలుగులో సులభంగా అర్థం చేసుకుందాం:

---

## 1. Amazon CloudWatch (అప్లికేషన్ పర్యవేక్షణ)

**CloudWatch అంటే ఏంటి?** ఇది AWS లోని సర్వీసుల ఆరోగ్యాన్ని, పర్ఫార్మెన్స్‌ని ట్రాక్ చేసే ఒక మెయిన్ డాక్టర్ లాంటిది. ఇందులో నాలుగు ప్రధాన భాగాలు ఉన్నాయి:

- **Metrics (కొలతలు):** మీ సర్వర్లు లేదా ల్యాంబ్డా ఫంక్షన్స్ ఎలా పనిచేస్తున్నాయో చెప్పే గ్రాఫ్‌లు/డేటా. ఉదాహరణకు: CPU వాడకం ఎంత ఉంది (`CPUUtilization`), RAM ఎంత ఖాళీగా ఉంది, లేదా మీ API కి ఎన్ని రిక్వెస్ట్‌లు వచ్చాయి అనే నంబర్లను ఇది నిరంతరం ట్రాక్ చేస్తుంది.
- **Logs (లాగ్స్):** బ్యాకెండ్ డెవలపర్‌గా మనం కోడ్‌లో రాసే `console.log()` లేదా `logger.error()` మెసేజ్‌లన్నీ ఒకే చోట సేవ్ అయ్యే డిజిటల్ డైరీ ఇది. మీ Node.js కోడ్ క్రాష్ అయితే, అసలు ఏ లైన్‌లో ఎర్రర్ వచ్చిందో CloudWatch Logs ఓపెన్ చేసి లైవ్ గా చూసుకోవచ్చు.
- **Alarms (అలారాలు):** "ఏదైనా ఒక సర్వర్ CPU వాడకం 80% దాటితే నాకు వెంటనే ఈమెయిల్ పంపు" లేదా "ల్యాంబ్డా ఫంక్షన్ 5 సార్ల కంటే ఎక్కువ ఫెయిల్ అయితే ఆటో స్కేలింగ్ స్టార్ట్ చెయ్" అని కండిషన్స్ పెట్టడం. ఇవి మీ కంపెనీ DevOps లేదా డెవలపర్స్ టీమ్‌కి అలర్ట్స్ పంపుతాయి.
- **Dashboards (డాష్‌బోర్డ్స్):** మీ కంపెనీ మేనేజర్ లేదా టీమ్ అంతా ఒకే స్క్రీన్ పై మీ అప్లికేషన్స్ పర్ఫార్మెన్స్, సర్వర్ల కౌంట్, మరియు ఎర్రర్స్ రేట్ వంటివన్నీ గ్రాఫికల్ రూపంలో (Charts/Graphs) చూసుకోవడానికి క్రియేట్ చేసే ఒక కామన్ స్క్రీన్.

---

## 2. AWS CloudTrail (ఆడిటింగ్ మరియు యాక్టివిటీ ట్రాకింగ్)

**CloudTrail అంటే ఏంటి?** ఇది మీ AWS అకౌంట్ లోపల జరిగే ప్రతి చిన్న కదలికను రికార్డ్ చేసే ఒక **CCTV కెమెరా** లాంటిది.

- CloudWatch అప్లికేషన్ పర్ఫార్మెన్స్ చూస్తే, CloudTrail **"ఎవరు, ఏం పని చేశారు"** అని చూస్తుంది.
- _ఉదాహరణ:_ అర్ధరాత్రి ఎవరో ఒక డెవలపర్ ఒక ముఖ్యమైన EC2 సర్వర్‌ని డిలీట్ చేశారు అనుకుందాం. అసలు ఆ సర్వర్‌ని ఎవరు డిలీట్ చేశారు, ఏ ఐపీ అడ్రస్ నుండి లాగిన్ అయి డిలీట్ చేశారు, ఏ టైమ్‌కి డిలీట్ చేశారు అనే పూర్తి 'ఆడిట్ లాగ్స్' (Audit Logs) ని CloudTrail రికార్డ్ చేస్తుంది. సెక్యూరిటీ కి ఇది చాలా ముఖ్యం.

---

## 3. AWS X-Ray (ట్రేసింగ్ మరియు బగ్ హంటింగ్)

**X-Ray అంటే ఏంటి?** మనుషులకి లోపల ఎక్కడైనా ఎముక విరిగితే X-Ray తీసి చూస్తాం కదా, అలాగే మీ మైక్రోసర్వీసెస్ (Microservices) ఆర్కిటెక్చర్‌లో రిక్వెస్ట్ ఎక్కడ ఆగిపోయిందో చూసే టూల్ ఇది.

- _ఉదాహరణ:_ ఒక యూజర్ ప్రొడక్ట్ కొన్నప్పుడు రిక్వెస్ట్ ముందు `API Gateway` కి వెళ్లి, అక్కడి నుండి `Lambda` కి వెళ్లి, ఆ తర్వాత `DynamoDB` కి వెళ్ళాలి. కానీ యూజర్‌కి స్క్రీన్ పై `500 Internal Server Error` వచ్చింది లేదా పేజీ లోడ్ అవ్వడానికి 10 సెకన్లు పట్టింది.
- **X-Ray లాభం:** ఇది ఆ రిక్వెస్ట్ ప్రయాణించిన పూర్తి రూట్ మ్యాప్‌ని (Visual Map) చూపిస్తుంది. API Gateway దగ్గర ఎంత టైమ్ పట్టింది, ల్యాంబ్డా లో ఎంత టైమ్ పట్టింది, డేటాబేస్ క్వెరీ స్లోగా ఉందా అనేది పిన్-పాయింట్‌తో సహా చూపిస్తుంది. దీనివల్ల పర్ఫార్మెన్స్ ఇష్యూస్ ఈజీగా ఫిక్స్ చేయవచ్చు.

---

## 4. AWS Config (ఇన్‌ఫ్రాస్ట్రక్చర్ కాన్ఫిగరేషన్ ట్రాకర్)

**AWS Config అంటే ఏంటి?** ఇది మీ AWS రిసోర్సెస్ యొక్క చరిత్రను (Configuration History) మరియు అవి కంపెనీ రూల్స్ ప్రకారం ఉన్నాయా లేదా అని చెక్ చేసే ఒక **గైడ్‌లైన్ మేనేజర్**.

- _ఉదాహరణ:_ మీ కంపెనీలో ఒక రూల్ ఉంది — "ఏ S3 బకెట్ కూడా పబ్లిక్‌గా ఉండకూడదు, అన్నీ ప్రైవేట్‌గానే ఉండాలి". ఒకవేళ పొరపాటున ఏ డెవలపర్ అయినా ఒక బకెట్‌ని పబ్లిక్ చేస్తే, AWS Config దాన్ని వెంటనే కనిపెట్టి "రూల్ బ్రేక్ అయింది" అని అలర్ట్ చేస్తుంది. పైగా ఆ బకెట్ నిన్న ఎలా ఉంది, ఇవాళ ఎలా మారింది అనే హిస్టరీని కూడా ట్రాక్ చేస్తుంది.

---

**క్విక్ సమ్మరీ:**

- సర్వర్ల పర్ఫార్మెన్స్, CPU, మరియు అలర్ట్స్ కోసం ➡️ **CloudWatch Metrics & Alarms**
- కోడ్ లోని ఎర్రర్స్ మరియు `console.log`లు చూడటానికి ➡️ **CloudWatch Logs**
- AWS లో ఎవరు ఏ సర్వీస్ క్రియేట్/డిలీట్ చేశారో కనిపెట్టడానికి (CCTV) ➡️ **CloudTrail**
- మైక్రోసర్వీసెస్ లో రిక్వెస్ట్ ఎక్కడ స్లో అయిందో కనిపెట్టడానికి (Tracing) ➡️ **AWS X-Ray**
- అకౌంట్ సెక్యూరిటీ రూల్స్ మరియు రిసోర్స్ హిస్టరీ ట్రాక్ చేయడానికి ➡️ **AWS Config**

మీరు ఇచ్చిన సిలబస్ లిస్ట్‌లోని **11 టాపిక్స్** చాలా విజయవంతంగా, బేసిక్స్ నుండి క్లియర్ గా తెలుగులో పూర్తి చేసుకున్నాం!

సాఫ్ట్‌వేర్ డెవలపర్‌గా AWS లో ఒక స్ట్రాంగ్ ఫౌండేషన్ బిల్డ్ చేయడానికి ఈ టాపిక్స్ అన్నీ పర్ఫెక్ట్‌గా ఉపయోగపడతాయి. మీ లిస్ట్‌లో ఇంకా ఏమైనా అదనపు టాపిక్స్ (ఉదాహరణకు: CI/CD - CodePipeline, లేదా Infrastructure as Code - CloudFormation) ఉన్నాయా? లేదా వీటిలో ఏదైనా ఒక టాపిక్ పైన డెమో కోడ్ (Node.js/React లింక్) లాంటిది చూద్దామా?

# 12. Infrastructure as Code

మోనిటరింగ్ అండ్ లాగింగ్ తర్వాత డెవలపర్లు మరియు డెవఆప్స్ (DevOps) ఇంజనీర్లు ప్రతిరోజూ వాడే అత్యంత ఆధునికమైన, మరియు ఇంటర్వ్యూలలో చాలా ప్రాముఖ్యత ఉన్న విభాగం — **Infrastructure as Code (IaC)**.

సాఫ్ట్‌వేర్ డెవలపర్‌గా మనం కోడ్ రాసి అప్లికేషన్‌ను ఎలా బిల్డ్ చేస్తామో... అలాగే AWS లో సర్వర్లు (EC2), డేటాబేస్‌లు (RDS), మరియు నెట్‌వర్క్ (VPC) లని మౌస్ క్లిక్స్‌తో కాకుండా, కేవలం కోడ్ రాసి ఆటోమేటిక్‌గా క్రియేట్ చేయడాన్నే **Infrastructure as Code** అంటారు. దీనివల్ల లాభం ఏంటంటే, ఒకే రకమైన ఇన్‌ఫ్రాస్ట్రక్చర్‌ను `Dev`, `QA`, `Prod` ఎన్విరాన్‌మెంట్లలో ఎటువంటి తప్పులు లేకుండా కేవలం ఒక్క క్లిక్‌తో సెటప్ చేయవచ్చు.

AWS లో దీనికోసం ఉన్న రెండు ప్రధాన టూల్స్: **AWS CloudFormation** మరియు **AWS CDK**.

---

## 1. AWS CloudFormation

**CloudFormation అంటే ఏంటి?** ఇది AWS అందించే ఒక మేనేజ్డ్ IaC సర్వీస్. ఇందులో మనం ఒక టెక్స్ట్ ఫైల్‌లో మనకు ఏమేం సర్వీసులు కావాలో రాసి ఇస్తే, AWS దాన్ని చదివి ఆ సర్వీసులన్నింటినీ లైన్‌గా క్రియేట్ చేస్తుంది. దీని ద్వారా క్రియేట్ అయిన రిసోర్సెస్ గ్రూప్‌ను **Stack** అంటారు.

- **Templates (టెంప్లేట్స్):** మనం ఇన్‌ఫ్రాస్ట్రక్చర్ వివరాలను రాయడానికి వాడే ఫైల్స్. వీటిని సాధారణంగా **YAML** లేదా **JSON** ఫార్మాట్లలో రాస్తాం.
- _ఉదాహరణకు:_ ఒక 10 లైన్ల YAML కోడ్ రాసి "నాకు `t2.micro` సైజ్ ఉన్న EC2 సర్వర్ కావాలి" అని చెబితే, CloudFormation దాన్ని క్రియేట్ చేస్తుంది.

- **Parameters (పారామీటర్స్):** కోడ్‌ను హార్డ్‌కోడ్ చేయకుండా, రన్‌టైమ్‌లో యూజర్ నుండి ఇన్‌పుట్ తీసుకోవడం.
- _ఉదాహరణ:_ ఒకే టెంప్లేట్ ఫైల్‌ని రన్ చేసేటప్పుడు, పారామీటర్ ద్వారా `Dev` అని ఇస్తే చిన్న సర్వర్ (`t2.micro`), `Prod` అని ఇస్తే పెద్ద సర్వర్ (`m5.large`) క్రియేట్ అయ్యేలా సెట్ చేయవచ్చు.

- **Outputs (అవుట్‌పుట్స్):** ఇన్‌ఫ్రాస్ట్రక్చర్ క్రియేట్ అయిపోయాక, మనకు కావలసిన ముఖ్యమైన సమాచారాన్ని స్క్రీన్ పై చూపించడం. ఉదాహరణకు: క్రియేట్ అయిన కొత్త లోడ్ బ్యాలెన్సర్ యొక్క URL లింక్ లేదా EC2 యొక్క ఐపీ అడ్రస్ వంటివి అవుట్‌పుట్ కింద పొందవచ్చు. దీనివల్ల ఈ సమాచారాన్ని వేరే క్లౌడ్‌ఫార్మేషన్ స్టాక్స్‌లో కూడా వాడుకోవచ్చు.
- **Nested Stacks (నెస్టెడ్ స్టాక్స్):** మీ కంపెనీ ఇన్‌ఫ్రాస్ట్రక్చర్ చాలా పెద్దది అయితే, అన్నింటినీ ఒకే ఫైల్‌లో రాస్తే గందరగోళంగా ఉంటుంది. అందుకే నెట్‌వర్కింగ్ (VPC) కి ఒక ఫైల్, డేటాబేస్‌కి ఒక ఫైల్, అప్లికేషన్ (EC2) కి ఒక ఫైల్ విడివిడిగా రాసి... వీటన్నింటినీ ఒక "Main Template" ద్వారా లింక్ చేసి రన్ చేయడాన్నే Nested Stacks అంటారు. ఇది కోడ్‌ను రీయుజబుల్ (Reusable) గా మారుస్తుంది.

---

## 2. AWS CDK (Cloud Development Kit)

బ్యాకెండ్ డెవలపర్లుగా మనకు YAML లేదా JSON లో వేల లైన్ల కాన్ఫిగరేషన్ రాయడం కొంచెం బోర్ కొడుతుంది. పైగా అందులో లూప్స్ (`for-loops`) లేదా కండిషన్స్ (`if-else`) రాయడం చాలా కష్టం. ఈ సమస్యకు పరిష్కారమే **AWS CDK**.

- **CDK అంటే ఏంటి?:** ఇది డెవలపర్ ఫ్రెండ్లీ టూల్. మీకు నచ్చిన ప్రోగ్రామింగ్ లాంగ్వేజ్ — ముఖ్యంగా **TypeScript, JavaScript (Node.js)**, Python లేదా Java వాడి క్లౌడ్ ఇన్‌ఫ్రాస్ట్రక్చర్‌ను డిజైన్ చేయవచ్చు.
- **వెనుక ఏం జరుగుతుంది?:** మీరు TypeScript లో `new ec2.Instance(...)` అని ఒక లైన్ కోడ్ రాసి కంప్యాక్ట్ చేస్తే, CDK దాన్ని బ్యాక్‌గ్రౌండ్‌లో ఆటోమేటిక్‌గా ఒక పెద్ద CloudFormation YAML టెంప్లేట్‌గా మార్చేసి (Synthesize) AWS లో డిప్లాయ్ చేస్తుంది.
- **లాభం:** డెవలపర్లు తమకు బాగా అలవాటైన కోడింగ్ స్ట్రక్చర్, ఆటో-కంప్లీట్ (IntelliSense), మరియు టెస్టింగ్ ఫ్రేమ్‌వర్క్‌లను వాడుకుంటూనే క్లౌడ్ రిసోర్సెస్‌ని ఈజీగా క్రియేట్ చేయవచ్చు.

---

**క్విక్ సమ్మరీ:**

- కోడ్ ద్వారా క్లౌడ్ ఇన్‌ఫ్రాస్ట్రక్చర్ బిల్డ్ చేయడం ➡️ **Infrastructure as Code (IaC)**
- YAML/JSON టెంప్లేట్స్ ద్వారా రిసోర్సెస్ మేనేజ్ చేయడం ➡️ **CloudFormation**
- రన్‌టైమ్‌లో వాల్యూస్ మార్చడానికి ➡️ **Parameters**
- సక్సెస్ అయ్యాక యుఆర్ఎల్స్/ఐపీలు పొందడానికి ➡️ **Outputs**
- పెద్ద కోడ్‌ను చిన్న భాగాలుగా విడగొట్టడానికి ➡️ **Nested Stacks**
- Node.js/TypeScript కోడ్ రాస్తూనే AWS రిసోర్సెస్ క్రియేట్ చేయడానికి ➡️ **AWS CDK**

---

మీరు ఇచ్చిన పూర్తి సిలబస్ లిస్ట్ (Topics 1 నుండి 12 వరకు) అద్భుతంగా, బేసిక్స్ నుండి అడ్వాన్స్‌డ్ IaC వరకు తెలుగులో పూర్తి చేసుకున్నాం! సాఫ్ట్‌వేర్ డెవలపర్‌గా AWS క్లౌడ్‌పై మీకు ఇప్పుడు ఒక పటిష్టమైన అవగాహన వచ్చి ఉంటుందని ఆశిస్తున్నాను.

మీకు ఈ మొత్తం కాన్సెప్ట్స్‌లో ఏదైనా ఒక నిర్దిష్టమైన సర్వీస్ మీద (ఉదాహరణకు: React+Node.js అప్లికేషన్‌ను EC2 లేదా Lambda లో ఎలా డిప్లాయ్ చేయాలి అనే దానిపై) ప్రాక్టికల్ ఆర్కిటెక్చర్ గైడెన్స్ ఏమైనా కావాలా?

సాఫ్ట్‌వేర్ ఆర్కిటెక్చర్ పరంగా **React.js (ఫ్రంటెండ్)** మరియు **Node.js (బ్యాకెండ్)** అప్లికేషన్‌ను AWS క్లౌడ్‌లో బెస్ట్ ప్రాక్టీసెస్ (Best Practices) ఉపయోగించి ఎలా డిజైన్ చేయాలో చూద్దాం.

మార్కెట్లో దీనికి రెండు రకాల పాపులర్ ఆర్కిటెక్చర్లు ఉన్నాయి. మీ ప్రాజెక్ట్ అవసరాన్ని బట్టి మీరు సరైన దాన్ని ఎంచుకోవచ్చు:

1. **Serverless Architecture** (తక్కువ ఖర్చు, జీరో మెయింటెనెన్స్ — చిన్న మరియు మీడియం యాప్స్ కి బెస్ట్).
2. **Container/Server-Based Architecture** (పూర్తి కంట్రోల్, మైక్రోసర్వీసెస్ — పెద్ద అప్లికేషన్స్ కి బెస్ట్).

మనం డెవలపర్లు ఎక్కువగా ఇష్టపడే **Serverless Architecture** ని బ్లూప్రింట్ రూపంలో సులభంగా అర్థం చేసుకుందాం:

## 🛠️ ప్రతి భాగం (Service) ఏం చేస్తుంది?

### 1. Frontend: React.js డిప్లాయ్‌మెంట్

- **Amazon S3:** మీ React యాప్‌ను `npm run build` చేశాక వచ్చే `dist` లేదా `build` ఫోల్డర్ లోని స్టాటిక్ ఫైల్స్ (HTML, CSS, JS, Images) అన్నీ **S3 Bucket** లో అప్‌లోడ్ చేస్తాం. దీనికి మనం **Static Website Hosting** ఆన్ చేస్తాం.
- **Amazon CloudFront:** S3 లింక్ నేరుగా యూజర్లకి ఇవ్వకుండా దాని ముందు CloudFront (CDN) పెడతాం. ఇది మీ React సైట్‌ని ప్రపంచవ్యాప్తంగా ఉన్న ఎడ్జ్ లొకేషన్లలో క్యాష్ (Cache) చేసి యూజర్లకి సూపర్ ఫాస్ట్‌గా ఓపెన్ అయ్యేలా చేస్తుంది. పైగా ఇందులో **ACM (Certificate Manager)** వాడి `https://` సెక్యూర్ కనెక్షన్ ఇస్తాం.

### 2. Traffic Router: డొమైన్ మేనేజ్‌మెంట్

- **Amazon Route 53:** మీ సొంత డొమైన్ (ఉదాహరణకు: `mysite.com`) ని క్లౌడ్‌ఫ్రంట్‌కి, మరియు `api.mysite.com` ని API గేట్‌వేకి లింక్ చేయడానికి Route 53 లో DNS రికార్డ్స్ సెట్ చేస్తాం.

### 3. Backend: Node.js / Express API

- **Amazon API Gateway:** ఫ్రంటెండ్ (React) నుండి వచ్చే అన్ని HTTP రిక్వెస్ట్‌లను (`/users`, `/orders`) స్వీకరించే మెయిన్ గేట్ ఇది. ఇది ట్రాఫిక్‌ను కంట్రోల్ చేస్తుంది, మరియు CORS (Cross-Origin Resource Sharing) ఇష్యూస్ రాకుండా మేనేజ్ చేస్తుంది.
- **AWS Lambda:** మీ Node.js కోడ్‌ను ఇక్కడ అప్‌లోడ్ చేస్తాం (మనం Express.js కోడ్‌ను ల్యాంబ్డాలో రన్ చేయడానికి `serverless-http` లాంటి లైబ్రరీస్ వాడవచ్చు). API Gateway కి రిక్వెస్ట్ రాగానే ల్యాంబ్డా ఫంక్షన్ యాక్టివేట్ అయి, కోడ్ రన్ చేసి, రెస్పాన్స్ ఇచ్చి వెంటనే స్లీప్ మోడ్‌లోకి వెళ్ళిపోతుంది. దీనివల్ల సర్వర్ బిల్లు 90% వరకు ఆదా అవుతుంది.

### 4. Database: డేటా నిల్వ

- **Amazon RDS లేదా DynamoDB:** మీ అప్లికేషన్ రిలేషనల్ డేటాబేస్ (MySQL/PostgreSQL) వాడుతుంటే **RDS** కి, లేదా NoSQL వాడుతుంటే **DynamoDB** కి మీ Node.js కోడ్ కనెక్ట్ అవుతుంది. (సెక్యూరిటీ కోసం ఈ డేటాబేస్‌ను నెట్‌వర్కింగ్‌లో చెప్పుకున్నట్టు **VPC Private Subnet** లో దాచి ఉంచుతాం. కేవలం ల్యాంబ్డా మాత్రమే దీన్ని యాక్సెస్ చేయగలదు).

## 🔐 భద్రత మరియు సీక్రెట్స్ (Production Best Practices)

- **AWS Secrets Manager:** మీ Node.js కోడ్‌లో డేటాబేస్ పాస్‌వర్డ్‌లు (`DB_PASSWORD`) లేదా JWT Secret Key లను హార్డ్‌కోడ్ చేయకుండా, Secrets Manager లో దాచి ఉంచి, రన్‌టైమ్‌లో ల్యాంబ్డా ద్వారా రీడ్ చేయాలి.
- **Cognito లేదా Custom Authorizer:** మీ React యాప్ నుండి యూజర్ లాగిన్ అయ్యాక వచ్చే టోకెన్‌ను వాలిడేట్ చేయడానికి API Gateway దగ్గర **Lambda Authorizer** సెట్ చేయాలి. దీనివల్ల ఫేక్ రిక్వెస్ట్‌లు మీ మెయిన్ బ్యాకెండ్ కోడ్ దాకా రావు.

మీరు ప్రొడక్షన్ లెవెల్‌లో యాప్ బిల్డ్ చేసేటప్పుడు ఈ పైన చెప్పుకున్న ఇన్‌ఫ్రాస్ట్రక్చర్ మొత్తాన్ని మన చివరి టాపిక్ అయిన **AWS CDK (TypeScript)** వాడి స్క్రిప్ట్ రూపంలో సింపుల్‌గా రాసి డిప్లాయ్ చేయవచ్చు!

ఈ ఆర్కిటెక్చర్ ఫ్లో మీకు అర్థమైందా? ఇందులో ఏదైనా నిర్దిష్టమైన సర్వీస్ (ఉదాహరణకు: Node.js కోడ్‌ని Lambda కి ఎలా మార్చాలి లాంటివి) లోతుగా తెలుసుకోవాలనుకుంటున్నారా?

# 13. DevOps

ఆర్కిటెక్చర్ తర్వాత మనం నేర్చుకోబోయే అత్యంత ప్రాముఖ్యమైన టాపిక్ — **DevOps (CI/CD)**.

సాఫ్ట్‌వేర్ డెవలపర్‌గా మనం లోకల్ కంప్యూటర్‌లో React.js లేదా Node.js కోడ్ రాశాక, దాన్ని మాన్యువల్‌గా బిల్డ్ చేసి, AWS సర్వర్‌లోకి లాగిన్ అయి అప్‌లోడ్ చేయడం చాలా పాత పద్ధతి. దీనివల్ల తప్పులు జరిగే అవకాశం ఎక్కువ.

దీనికి బదులుగా, మనం గిట్‌హబ్‌లో కోడ్ పుష్ చేయగానే ఆటోమేటిక్‌గా టెస్టింగ్ జరిగి, బిల్డ్ అయి, క్లౌడ్‌లో డిప్లాయ్ అయిపోయేలా చేసే ఆటోమేషన్ ప్రాసెస్ నే **CI/CD** అంటారు. AWS లో దీనికోసం ఒక ప్రత్యేకమైన టూల్ సెట్ (Developer Tools Suite) ఉంది. వీటిని తెలుగులో సులభంగా అర్థం చేసుకుందాం:

---

## 🔄 CI/CD Concepts (బేసిక్ కాన్సెప్ట్స్)

- **CI (Continuous Integration):** డెవలపర్లు ప్రతిరోజూ రాసే కోడ్‌ను ఒక సెంట్రల్ రిపోజిటరీలో కలపడం (Merge చేయడం). కోడ్ కలవగానే ఆటోమేటిక్‌గా టెస్ట్ కేసులు రన్ అయి, కోడ్‌లో ఏమైనా బగ్‌లు ఉన్నాయో లేదో చెక్ చేస్తుంది.
- **CD (Continuous Delivery / Deployment):** టెస్టింగ్ విజయవంతంగా పూర్తయిన కోడ్‌ను ఆటోమేటిక్‌గా ప్రొడక్షన్ లేదా డెవలప్‌మెంట్ సర్వర్లలో (S3, Lambda, లేదా EC2 లో) డిప్లాయ్ చేసి యూజర్లకు అందుబాటులోకి తేవడం.

---

## 🛠️ AWS DevOps టూల్స్

మనం ఒక పైప్‌లైన్ (Pipeline) ఊహించుకుంటే, అందులో ఒక్కొక్క స్టేజ్‌కి ఒక్కొక్క AWS సర్వీస్ ఉపయోగపడుతుంది:

```
[ CodeCommit ] ──>(Code Push)──> [ CodeBuild ] ──>(Build & Test)──> [ CodeDeploy ] ──>(Deploy to AWS)
                                      ▲
                                      │ (ఇవన్నింటినీ కలిపి నడిపించేది)
                                [ CodePipeline ]

```

### 1. AWS CodeCommit (కోడ్ దాచే చోటు)

ఇది మీ కోడ్‌ను భద్రపరిచే ఒక **సొంత గిట్‌హబ్ (Private GitHub/GitLab)** లాంటిది.

- ఇది ప్యూర్ `Git` ఆధారిత సర్వీస్.
- మీ React మరియు Node.js కోడ్‌ను టీమ్ మెంబర్స్ అందరూ కలిసి ఇక్కడే పుష్ చేస్తారు. ఇది AWS లోపలే ఉండటం వల్ల సెక్యూరిటీ (IAM) చాలా స్ట్రాంగ్‌గా ఉంటుంది.

### 2. AWS CodeBuild (కోడ్ బిల్డ్ & టెస్ట్ చేసే టూల్)

డెవలపర్లు కోడ్ పుష్ చేసాక, దాన్ని కంపైల్ చేయడం, టెస్ట్ రన్ చేయడం, మరియు డిప్లాయ్‌మెంట్‌కి సిద్ధం చేయడం CodeBuild పని.

- బ్యాక్‌గ్రౌండ్‌లో ఇది ఒక చిన్న డాకర్ కంటైనర్‌ను క్రియేట్ చేసి, అందులో మీ కోడ్‌ను రన్ చేస్తుంది.
- _డెవలపర్ టాస్క్:_ ఇందులో మనం **`buildspec.yml`** అనే ఫైల్ రాస్తాం. మీ Node.js యాప్ అయితే అందులో `npm install` మరియు `npm test` లాంటి కమాండ్స్ రాస్తాం. బిల్డ్ పూర్తయ్యాక వచ్చే ఫైల్స్ (Artifacts) ని ఇది S3 లో సేవ్ చేస్తుంది.

### 3. AWS CodeDeploy (సర్వర్‌లోకి కోడ్‌ను పంపే టూల్)

CodeBuild సిద్ధం చేసిన ఫైల్స్‌ను తీసుకుని, మీ లైవ్ సర్వర్లు (EC2, Lambda, లేదా ECS కంటైనర్స్) లోపలికి సురక్షితంగా ఇన్‌స్టాల్ చేసే బాధ్యత CodeDeploy ది.

- _డెవలపర్ టాస్క్:_ ఇందులో మనం **`appspec.yml`** అనే ఫైల్ రాస్తాం. కోడ్ సర్వర్‌లోకి వెళ్ళాక పాత Node.js ప్రాసెస్‌ని ఎలా స్టాప్ చేయాలి, కొత్త కోడ్‌ని ఎలా స్టార్ట్ చేయాలి అనే ఇన్స్ట్రక్షన్స్ ఇందులో ఉంటాయి.
- ఇది అప్లికేషన్ డౌన్ అవ్వకుండా కొత్త కోడ్‌ను అప్‌డేట్ చేయడానికి వివిధ డిప్లాయ్‌మెంట్ స్ట్రాటజీలను (ఉదాహరణకు: Blue/Green Deployment) సపోర్ట్ చేస్తుంది.

### 4. AWS CodePipeline (మెయిన్ లీడర్ / పైప్‌లైన్)

పైన చెప్పుకున్న మూడు సర్వీసులను (CodeCommit ➡️ CodeBuild ➡️ CodeDeploy) ఒక క్రమ పద్ధతిలో లింక్ చేసి నడిపించే **మెయిన్ మేనేజర్** ఈ CodePipeline.

- ఇది ఒక ఆటోమేటెడ్ వర్క్‌ఫ్లో.
- డెవలపర్ CodeCommit లో కోడ్ పుష్ చేయగానే... ఈ పైప్‌లైన్ ఆటోమేటిక్‌గా ట్రిగ్గర్ అవుతుంది ➡️ CodeBuild ని పిలిచి బిల్డ్ చేయిస్తుంది ➡️ అది సక్సెస్ అయితే CodeDeploy ని పిలిచి సర్వర్‌లో డిప్లాయ్ చేయిస్తుంది. ఎక్కడైనా ఫెయిల్ అయితే వెంటనే డెవలపర్లకి అలర్ట్ పంపుతుంది.

---

## 💡 రియల్-టైమ్ ఉదాహరణ (React + Node.js CI/CD)

మీరు మీ React యాప్‌లో ఒక కొత్త బటన్ కలర్ మార్చారు అనుకుందాం:

1. మీరు కోడ్‌ను `git push` చేసి **CodeCommit** కి పంపుతారు.
2. **CodePipeline** దాన్ని గమనించి వెంటనే **CodeBuild** ని యాక్టివేట్ చేస్తుంది.
3. CodeBuild రంగంలోకి దిగి `npm run build` కమాండ్ రన్ చేసి React స్టాక్ ఫైల్స్‌ను (Production Build) రెడీ చేస్తుంది.
4. ఆ ఫైల్స్‌ను **CodeDeploy** కి ఇస్తుంది. CodeDeploy ఆ కొత్త ఫైల్స్‌ను మీ ఫ్రంటెండ్ **S3 Bucket** లోకి అప్‌లోడ్ చేస్తుంది.
5. అంతే! కేవలం 2 నిమిషాల్లో మీ కొత్త మార్పులు ఎటువంటి మాన్యువల్ వర్క్ లేకుండా ప్రపంచానికి లైవ్ అయిపోతాయి.

---

**క్విక్ సమ్మరీ:**

- కోడ్ రిపోజిటరీ (Private Git) ➡️ **CodeCommit**
- కోడ్ టెస్టింగ్ మరియు కంపైలింగ్ (`buildspec.yml`) ➡️ **CodeBuild**
- కోడ్‌ను సర్వర్లలో/ల్యాంబ్డాలో డిప్లాయ్ చేయడం (`appspec.yml`) ➡️ **CodeDeploy**
- ఈ ప్రాసెస్ మొత్తాన్ని ఆటోమేట్ చేసే పైప్‌లైన్ ➡️ **CodePipeline**

# 14. Security Services

DevOps తర్వాత మనం నేర్చుకోబోయే చివరి, మరియు అత్యంత కీలకమైన సెక్యూరిటీ విభాగం — **Advanced Security Services**.

మనం ముందు చెప్పుకున్న IAM లేదా KMS అనేవి అకౌంట్ లోపల యాక్సెస్ కంట్రోల్ మరియు ఎన్‌క్రిప్షన్ కోసం వాడతాం. కానీ ఈ 14వ టాపిక్‌లోని సర్వీసులు... బయటి నుండి వచ్చే హ్యాకర్లు, సైబర్ అటాక్స్ (DDoS), వైరస్‌లు మరియు మాల్వేర్ల నుండి మీ React, Node.js అప్లికేషన్లను రక్షించే **డిజిటల్ ఆర్మీ లేదా సెక్యూరిటీ సిస్టమ్** లాంటివి.

వీటిని తెలుగులో సులభంగా అర్థం చేసుకుందాం:

---

## 1. AWS WAF (Web Application Firewall)

**WAF అంటే ఏంటి?** ఇది మీ వెబ్‌సైట్‌కి వచ్చే ట్రాఫిక్‌ను చెక్ చేసే ఒక **స్మార్ట్ సెక్యూరిటీ గార్డు**. ఇది నెట్‌వర్కింగ్ యొక్క Application Layer (Layer 7) లో పనిచేస్తుంది.

- **ఎందుకు వాడతాం?:** హ్యాకర్లు మీ Node.js API లలోకి తప్పుడు కోడ్ పంపి డేటాబేస్‌ను హ్యాక్ చేయడానికి ట్రై చేస్తుంటారు (SQL Injection లేదా Cross-Site Scripting - XSS అటాక్స్ అంటారు).
- **WAF ఏం చేస్తుంది?:** ఇది వచ్చే ప్రతి రిక్వెస్ట్‌ని స్కాన్ చేస్తుంది. ఇందులో మనం రూల్స్ రాయవచ్చు.
- _ఉదాహరణ:_ "చైనా లేదా ఒక నిర్దిష్ట దేశం నుండి వచ్చే ఐపీ అడ్రస్‌లను బ్లాక్ చెయ్" లేదా "ఒక్క ఐపీ అడ్రస్ నుండి సెకనుకు 100 కంటే ఎక్కువ రిక్వెస్ట్‌లు వస్తే (Rate Limiting) దాన్ని కాసేపు బ్యాన్ చెయ్".
- దీనిని మనం మన **API Gateway, Application Load Balancer (ALB), లేదా CloudFront** కి కనెక్ట్ చేయవచ్చు.

---

## 2. AWS Shield (DDoS ప్రొటెక్షన్)

**AWS Shield అంటే ఏంటి?** ఇది **DDoS (Distributed Denial of Service)** అటాక్స్ నుండి మీ ఇన్‌ఫ్రాస్ట్రక్చర్‌ను కాపాడే ఒక పెద్ద కవచం.

- **DDoS అంటే ఏంటి?:** హ్యాకర్లు వేల సంఖ్యలో దొంగ కంప్యూటర్లను (Bots) వాడుకుని, మీ వెబ్‌సైట్‌కి ఒకే సెకనులో కోట్ల కొద్దీ ఫేక్ ట్రాఫిక్‌ను పంపుతారు. దీనివల్ల మీ సర్వర్లు తట్టుకోలేక క్రాష్ అయిపోతాయి (దీన్నే DDoS అటాక్ అంటారు).
- **Shield రకాలు:**
- **Shield Standard:** ఇది AWS లో ప్రతి కస్టమర్‌కి **ఉచితంగా** బ్యాక్‌గ్రౌండ్‌లో ఆటోమేటిక్‌గా నడుస్తూనే ఉంటుంది. ఇది చిన్న మరియు మీడియం నెట్‌వర్క్ అటాక్స్ నుండి రక్షణ ఇస్తుంది.
- **Shield Advanced:** ఇది చాలా ఖరీదైన, ప్రీమియం సర్వీస్. పెద్ద పెద్ద ఈ-కామర్స్ (Amazon, Flipkart లాంటివి) లేదా బ్యాంకింగ్ వెబ్‌సైట్లకు భారీ అటాక్స్ జరిగినప్పుడు 24/7 AWS సెక్యూరిటీ టీమ్ సపోర్ట్‌తో రక్షణ ఇవ్వడానికి ఇది వాడుతారు.

---

## 3. Amazon GuardDuty (ఇంటెలిజెంట్ థ్రెట్ డిటెక్షన్)

**GuardDuty అంటే ఏంటి?** ఇది మీ AWS అకౌంట్ లోపల ఎవరైనా హ్యాకింగ్ చేయడానికి ప్రయత్నిస్తున్నారా అని కనిపెట్టే ఒక **రహస్య గూఢచారి (AI Detective)**.

- **ఎలా పనిచేస్తుంది?:** ఇది మన 11వ టాపిక్‌లో చెప్పుకున్న CloudTrail లాగ్స్, VPC ఫ్లో లాగ్స్, మరియు DNS లాగ్స్‌ను బ్యాక్‌గ్రౌండ్‌లో ఆటోమేటిక్‌గా ఆర్టిఫిషియల్ ఇంటెలిజెన్స్ (AI/Machine Learning) ఉపయోగించి స్కాన్ చేస్తూ ఉంటుంది.
- _ఉదాహరణ:_ మీ అకౌంట్‌లో ఒక డెవలపర్ ఎప్పుడూ హైదరాబాద్ నుండి లాగిన్ అవుతుంటారు. కానీ అకస్మాత్తుగా అర్ధరాత్రి వేరే దేశం (ఉదాహరణకు రష్యా లేదా నార్త్ కొరియా) ఐపీ అడ్రస్ నుండి లాగిన్ అయి సర్వర్లు క్రియేట్ చేయడానికి ట్రై చేస్తే... GuardDuty వెంటనే దీన్ని ఒక "అనుమానాస్పద యాక్టివిటీ" (Suspicious Activity) గా గుర్తించి మీకు అలర్ట్ పంపుతుంది.

---

## 4. Amazon Inspector (సాఫ్ట్‌వేర్ లోపాలను కనిపెట్టే టూల్)

**Amazon Inspector అంటే ఏంటి?** సాఫ్ట్‌వేర్ డెవలపర్లుగా మనం రాసే కోడ్‌లో లేదా వాడే సాఫ్ట్‌వేర్ వెర్షన్లలో ఏమైనా సెక్యూరిటీ లోపాలు (Vulnerabilities/Bugs) ఉన్నాయా అని తనిఖీ చేసే ఒక **ఇన్స్పెక్టర్ (Automated Security Scanner)**.

- **ఎలా ఉపయోగపడుతుంది?:**
- **EC2 సర్వర్లు:** మీ EC2 సర్వర్‌లో పాత కాలపు ఆపరేటింగ్ సిస్టమ్ లేదా బగ్‌లు ఉన్న సాఫ్ట్‌వేర్ రన్ అవుతుంటే ఇది కనిపెడుతుంది.
- **Container Images:** మీ Node.js కోడ్‌ని డాకర్ ఇమేజ్ (`Docker Image`) గా మార్చినప్పుడు, ఆ ఇమేజ్ లోపల ఏమైనా సెక్యూరిటీ లోపాలు ఉన్నాయా అని స్కాన్ చేస్తుంది.
- **Lambda Code:** మీ Node.js ల్యాంబ్డా కోడ్‌లో పాతబడిపోయిన, సురక్షితం కాని `npm modules` (Packages) ఏమైనా వాడుతుంటే ఇది పాయింట్అవుట్ చేసి, దాన్ని ఎలా ఫిక్స్ చేయాలో (Remediation) కూడా చెబుతుంది.

---

## 5. AWS Security Hub (సెంట్రల్ సెక్యూరిటీ డాష్‌బోర్డ్)

**Security Hub అంటే ఏంటి?** పైన చెప్పుకున్న GuardDuty, Inspector, మరియు AWS Config వంటి అన్ని సెక్యూరిటీ సర్వీసుల నుండి వచ్చే అలర్ట్స్ మరియు రిపోర్ట్స్ అన్నింటినీ ఒకే చోట చూపించే ఒక **మెయిన్ కంట్రోల్ రూమ్ (Central Dashboard)**.

- **లాభం:** కంపెనీ యొక్క చీఫ్ సెక్యూరిటీ ఆఫీసర్ (CSO) లేదా లీడ్ డెవలపర్ వేర్వేరు సర్వీసుల పేజీలలోకి వెళ్లి చెక్ చేయాల్సిన అవసరం లేదు. Security Hub ఓపెన్ చేస్తే చాలు... మొత్తం AWS అకౌంట్ యొక్క సెక్యూరిటీ స్కోర్ ఎంత ఉంది, ఎక్కడెక్కడ లూప్‌హోల్స్ ఉన్నాయి అనేది ఒకే స్క్రీన్ పై పర్ఫెక్ట్‌గా కనిపిస్తుంది.

---

**క్విక్ సమ్మరీ:**

- SQL ఇంజెక్షన్, హ్యాకర్ల ఫేక్ రిక్వెస్ట్‌లు బ్లాక్ చేయడానికి ➡️ **AWS WAF**
- భారీ ట్రాఫిక్ అటాక్స్ (DDoS) నుండి రక్షణకు ➡️ **AWS Shield**
- అకౌంట్ లోపల దొంగ యాక్టివిటీస్ కనిపెట్టే గూఢచారి ➡️ **Amazon GuardDuty**
- మీ కోడ్, డాకర్ ఇమేజ్, సర్వర్లలోని బగ్‌లను వెతికే ఇన్స్పెక్టర్ ➡️ **Amazon Inspector**
- అన్ని సెక్యూరిటీ రిపోర్ట్స్ చూపే మెయిన్ హెడ్‌క్వార్టర్స్ ➡️ **AWS Security Hub**

---

# 15. Migration

చివరిగా, మీరు ఇచ్చిన లిస్ట్‌లో చేరిన సరికొత్త మరియు చాలా ముఖ్యమైన విభాగం — **Migration (వలస లేదా డేటా బదిలీ సర్వీసెస్)**.

పెద్ద పెద్ద కంపెనీలు తమ సొంత ఆఫీస్ సర్వర్ల (On-premises/Data Centers) నుండి లేదా వేరే క్లౌడ్ (Google Cloud లేదా Azure) నుండి తమ అప్లికేషన్లను మరియు డేటాను AWS క్లౌడ్‌లోకి మార్చడానికి ఈ సర్వీసులను వాడుతుంటాయి. వీటిని కూడా తెలుగులో సులభంగా అర్థం చేసుకుందాం:

---

## 1. AWS Application Migration Service (AWS MGN)

**AWS MGN అంటే ఏంటి?** ఇది మీ పాత సర్వర్లను (అవి ఫిజికల్ సర్వర్లు అయినా లేదా వర్చువల్ సర్వర్లు అయినా) ఎటువంటి ఇబ్బంది లేకుండా, ఆటోమేటిక్‌గా AWS లోని **EC2 సర్వర్లుగా మార్చే** ఒక మెయిన్ టూల్.

- **ఎలా పనిచేస్తుంది?:** మనం మన పాత సర్వర్‌లో ఒక చిన్న సాఫ్ట్‌వేర్ (Replication Agent) ఇన్‌స్టాల్ చేస్తాం. అది ఆ సర్వర్‌లో ఉన్న ఆపరేటింగ్ సిస్టమ్, సాఫ్ట్‌వేర్స్, మీ Node.js కోడ్ ఫైల్స్‌తో సహా మొత్తాన్ని బ్యాక్‌గ్రౌండ్‌లో క్లౌడ్‌లోకి కాపీ (Replicate) చేయడం స్టార్ట్ చేస్తుంది.
- **లాభం:** మీ పాత సర్వర్ రన్ అవుతూనే ఉంటుంది (Zero Downtime). డేటా అంతా క్లౌడ్‌లోకి కాపీ అయిపోయాక, ఒక్క క్లిక్‌తో పాత సర్వర్ నుండి కొత్త AWS EC2 సర్వర్‌కి ట్రాఫిక్‌ను మార్చుకోవచ్చు.

---

## 2. AWS DataSync

**AWS DataSync అంటే ఏంటి?** ఇది మీ ఆఫీస్ స్టోరేజ్ సిస్టమ్స్ (NFS, SMB ఫైల్ షేర్స్) లేదా వేరే క్లౌడ్ స్టోరేజ్ నుండి భారీ మొత్తంలో ఉన్న డేటాను (Terabytes/Petabytes) చాలా వేగంగా **AWS S3, EFS లేదా EBS** లోకి కాపీ చేసే ఒక **సూపర్ ఫాస్ట్ డేటా బ్రిడ్జ్**.

- **ఎందుకు వాడతాం?:** నార్మల్ ఇంటర్నెట్ ద్వారా నెలకు కొన్ని వేల గిగాబైట్ల (GB) డేటా పంపడం చాలా స్లోగా ఉంటుంది. కానీ DataSync ప్రత్యేకమైన నెట్‌వర్క్ ప్రొటోకాల్స్ వాడుకుని, డేటాను కంప్రెస్ చేసి, ఇంటర్నెట్ స్పీడ్‌ని పూర్తిగా వాడుకుంటూ **10 రెట్లు వేగంగా** డేటాను క్లౌడ్‌కి ట్రాన్స్‌ఫర్ చేస్తుంది.
- ఇది డేటా ట్రాన్స్‌ఫర్ అయ్యేటప్పుడు ఎన్‌క్రిప్షన్ (Security) చూసుకుంటుంది, అలాగే ఫైల్స్ అన్నీ పర్ఫెక్ట్‌గా కాపీ అయ్యాయో లేదో ఆటోమేటిక్‌గా వెరిఫై (Data Validation) కూడా చేస్తుంది.

---

## 3. AWS Transfer Family

**AWS Transfer Family అంటే ఏంటి?** సాఫ్ట్‌వేర్ డెవలప్‌మెంట్‌లో కంపెనీలు తమ బిజినెస్ పార్టనర్లతో లేదా క్లయింట్లతో ఫైల్స్ షేర్ చేసుకోవడానికి చాలా కాలంగా **FTP (File Transfer Protocol), SFTP (Secure FTP), లేదా FTPS** పద్ధతులను వాడుతున్నాయి.

కంపెనీలు క్లౌడ్‌లోకి మారాక కూడా తమ పాత ఫైల్ షేరింగ్ పద్ధతిని మార్చుకోవడానికి ఇష్టపడవు. వారి కోసం ఈ సర్వీస్ ఉపయోగపడుతుంది.

- **ఎలా ఉపయోగపడుతుంది?:** ఇది మీకు ఒక రెడీమేడ్ SFTP సర్వర్‌ని ఇస్తుంది. మీ క్లయింట్లు ఎప్పటిలాగే తమ పాత SFTP సాఫ్ట్‌వేర్ (FileZilla లాంటివి) వాడుతూ ఫైల్స్ అప్‌లోడ్ చేయవచ్చు.
- కానీ మ్యాజిక్ ఏంటంటే, వారు అప్‌లోడ్ చేసే ఫైల్స్ అన్నీ బ్యాక్‌గ్రౌండ్‌లో నేరుగా మీ **AWS S3 Bucket** లోనో లేదా **EFS** లోనో సేవ్ అవుతాయి. దీనివల్ల మనం విడిగా SFTP సర్వర్లను మెయింటైన్ చేయాల్సిన అవసరం లేదు.

---

**క్విక్ సమ్మరీ:**

- పాత సర్వర్లను (OS & Apps తో సహా) ఆటోమేటిక్‌గా EC2 కి మార్చడానికి ➡️ **AWS MGN (Application Migration)**
- భారీ మొత్తంలో ఉన్న ఫైల్స్/డేటాను సూపర్ ఫాస్ట్‌గా S3/EFS కి కాపీ చేయడానికి ➡️ **AWS DataSync**
- పాత క్లయింట్ల కోసం SFTP సర్వర్ రన్ చేస్తూ, డేటాను నేరుగా S3 లో దాచడానికి ➡️ **AWS Transfer Family**

---

# 16. Analytics

మైగ్రేషన్ తర్వాత మనం నేర్చుకోబోయే చాలా ఆసక్తికరమైన మరియు డేటా-డ్రివెన్ (Data-driven) విభాగం — **Analytics (డేటా విశ్లేషణ సర్వీసెస్)**.

బ్యాకెండ్ డెవలపర్‌గా మీ React, Node.js యాప్స్ రన్ అవుతున్నప్పుడు... యూజర్ల లాగ్స్, యాప్ ఈవెంట్స్, లేదా బిజినెస్ లావాదేవీల డేటా భారీ మొత్తంలో (గిగాబైట్లు, టెరాబైట్లలో) S3 లోనో లేదా డేటాబేస్‌లలోనో పేరుకుపోతుంటుంది. ఆ భారీ డేటాను విశ్లేషించి (Analyze చేసి), బిజినెస్‌కి ఉపయోగపడే ముఖ్యమైన నిర్ణయాలు తీసుకోవడానికి AWS ఈ 4 అద్భుతమైన అనలిటిక్స్ టూల్స్‌ను అందిస్తోంది:

---

## 1. Amazon Athena (సర్వర్‌లెస్ క్వెరీ టూల్)

**Athena అంటే ఏంటి?** ఇది ఒక మ్యాజిక్ టూల్ లాంటిది. మీ S3 బకెట్‌లో వేల సంఖ్యలో CSV, JSON, లేదా Parquet ఫైల్స్ ఉన్నాయి అనుకుందాం. సాధారణంగా ఆ డేటాను చదవాలంటే మనం ఒక నోడ్.జెఎస్ కోడ్ రాసి, ఫైల్స్ అన్నీ డౌన్లోడ్ చేసి ప్రాసెస్ చేయాలి.

- **లాభం:** Athena వాడితే ఎటువంటి సర్వర్ లేకుండా, డేటాను ఎక్కడికీ మార్చకుండా... నేరుగా S3 లో ఉన్న ఫైల్స్ పైనే మనకు తెలిసిన **సాధారణ SQL క్వెరీలు (`SELECT * FROM my_s3_data WHERE...`)** రన్ చేసి డేటాను క్షణాల్లో చూడవచ్చు.
- ఇది పూర్తిగా **Serverless**. మీరు రన్ చేసిన క్వెరీ ఎంత డేటాను స్కాన్ చేసిందో, దానికి మాత్రమే బిల్లు పడుతుంది. లాగ్ ఫైల్స్ అనలైజ్ చేయడానికి ఇది డెవలపర్లకు బెస్ట్ ఛాయిస్.

---

## 2. AWS Glue (డేటా ప్రిపరేషన్ & ETL సర్వీస్)

**AWS Glue అంటే ఏంటి?** అనలిటిక్స్ చేసే ముందు డేటాను క్లీన్ చేసి, ఒక పద్ధతిలోకి మార్చడాన్ని **ETL (Extract, Transform, Load)** అంటారు. ఆ పని చేసే సర్వర్‌లెస్ సర్వీస్ ఈ Glue.

इसमें రెండు ప్రధాన భాగాలు ఉన్నాయి:

- **Glue Data Catalog & Crawlers:** మీ S3 లో ఉన్న ఫైల్స్‌ని ఒక క్రాలర్ (Crawler) ఆటోమేటిక్‌గా స్కాన్ చేసి, అందులో ఏమేం కాలమ్స్ ఉన్నాయి (Schema ఏంటి) అని కనిపెట్టి ఒక వర్చువల్ టేబుల్ లాగా సేవ్ చేస్తుంది. దీనివల్లనే పైనున్న Athena ఆ ఫైల్స్ ని SQL టేబుల్స్ లాగా చదవగలుగుతుంది.
- **ETL Jobs:** ముడి డేటాను (Raw data) తీసుకుని, అందులోని తప్పులను తీసేసి (Transform చేసి), అనలిటిక్స్ కి రెడీగా ఉండేలా మార్చడానికి ఇది ఆటోమేటిక్‌గా Python లేదా Spark కోడ్‌ను జనరేట్ చేసి రన్ చేస్తుంది.

---

## 3. Amazon EMR (Elastic MapReduce)

**Amazon EMR అంటే ఏంటి?** మీ కంపెనీలో ఉన్న డేటా సైజ్ కొన్ని టెరాబైట్లు లేదా పెటాబైట్లలో (భారీ బిగ్ డేటా - Big Data) ఉంది అనుకుందాం. అంత డేటాను ఒకే కంప్యూటర్ ప్రాసెస్ చేయలేదు.

- EMR అనేది ఓపెన్-సోర్స్ బిగ్ డేటా ఫ్రేమ్‌వర్క్స్ అయిన **Apache Spark, Hadoop, Presto** వంటి వాటిని క్లౌడ్‌లో సులభంగా రన్ చేయడానికి వాడే ఒక క్లస్టర్ (వందలాది సర్వర్ల నెట్‌వర్క్).
- ఇది భారీ డేటాను వంద ముక్కలుగా విడగొట్టి, వంద సర్వర్లలో ఒకే సమయంలో (Parallel గా) ప్రాసెస్ చేసి, రిజల్ట్‌ను చాలా వేగంగా ఇస్తుంది. భారీ మిషన్ లెర్నింగ్ (ML) డేటా సెట్స్ మరియు డేటా సైన్స్ పనుల కోసం EMR వాడుతారు.

---

## 4. Amazon Kinesis (లైవ్ స్ట్రీమింగ్ డేటా టూల్)

**Amazon Kinesis అంటే ఏంటి?** పైన చెప్పుకున్న Athena, Glue, EMR లు అప్పటికే సేవ్ అయి ఉన్న డేటా (Batch Data) పై పనిచేస్తాయి. కానీ, డేటా సెకను సెకనుకూ లైవ్‌గా వస్తూ ఉంటే (Real-time Streaming Data), దాన్ని పట్టుకోవడానికి Kinesis వాడుతాం.

- _ఉదాహరణ:_ మీ React యాప్‌ను ప్రపంచవ్యాప్తంగా లక్షల మంది వాడుతున్నారు. వారు ఎక్కడ క్లిక్ చేస్తున్నారు (Clickstream data), లేదా ఒక IoT డివైజ్ నుండి వచ్చే లైవ్ టెంపరేచర్ డేటా, లేదా లైవ్ ఫైనాన్షియల్ ట్రాన్సాక్షన్స్... ఇలా క్షణం ఆగకుండా వచ్చే డేటాను Kinesis లైవ్‌గా స్వీకరిస్తుంది.
- **Kinesis Data Streams:** వచ్చే లైవ్ డేటాను సెకనుకు టెరాబైట్ల వేగంతో క్యాప్చర్ చేస్తుంది. మీ Node.js యాప్ లేదా ల్యాంబ్డా ఈ డేటాను లైవ్‌గా రీడ్ చేసి వెంటనే ప్రాసెస్ చేయవచ్చు.
- **Kinesis Data Firehose:** లైవ్‌గా వచ్చే డేటాను ఎటువంటి కోడ్ రాయకుండా, ఆటోమేటిక్‌గా తీసుకెళ్లి నేరుగా S3 బకెట్‌లోనో లేదా Redshift డేటా వేర్‌హౌస్‌లోనో సేవ్ చేస్తుంది.

---

**క్విక్ సమ్మరీ:**

- S3 లో ఉన్న ఫైల్స్ పై నేరుగా SQL క్వెరీలు రన్ చేయడానికి ➡️ **Amazon Athena**
- డేటా టేబుల్స్ లాగా స్కాన్ చేయడానికి, క్లీన్ చేయడానికి (ETL) ➡️ **AWS Glue**
- టెరాబైట్ల భారీ బిగ్ డేటాను (Spark/Hadoop) ప్రాసెస్ చేయడానికి ➡️ **Amazon EMR**
- లైవ్ క్లిక్స్, రియల్-టైమ్ స్ట్రీమింగ్ డేటాను హ్యాండిల్ చేయడానికి ➡️ **Amazon Kinesis**

# 17. Machine Learning

అనలిటిక్స్ తర్వాత మీ లిస్ట్‌లోని తదుపరి, మరియు టెక్నాలజీ ప్రపంచంలోనే అత్యంత హాట్ టాపిక్ — **Machine Learning (మిషన్ లెర్నింగ్ సర్వీసెస్)**.

సాఫ్ట్‌వేర్ డెవలపర్‌గా మనం AI/ML అనగానే పెద్ద పెద్ద మ్యాథమెటికల్ ఫార్ములాలు, కాంప్లెక్స్ అల్గారిథమ్స్ రాయాలేమో అని కంగారు పడక్కర్లేదు. AWS మన పనిని చాలా సులభం చేసింది. ఇందులో రెండు రకాలు ఉన్నాయి:

1. **SageMaker:** డేటా సైంటిస్టుల కోసం (సొంతంగా AI మోడల్స్ బిల్డ్ చేయడానికి).
2. **AI Services (Rekognition, Textract, Comprehend):** డెవలపర్ల కోసం రెడీమేడ్‌గా ఇచ్చే **Pre-trained APIలు**. మనం జస్ట్ ఒక Node.js API కాల్ లేదా SDK వాడి వీటిని మన యాప్‌లో వాడుకోవచ్చు.

వీటిని తెలుగులో సులభంగా అర్థం చేసుకుందాం:

---

## 1. Amazon SageMaker (సొంతంగా AI మోడల్స్ బిల్డ్ చేసే ప్లాట్‌ఫారమ్)

**SageMaker అంటే ఏంటి?** ఇది డేటా సైంటిస్టులు మరియు ML ఇంజనీర్ల కోసం ఉద్దేశించిన ఒక కంప్లీట్ ఎండ్-టు-ఎండ్ ప్లాట్‌ఫారమ్.

- **ఎందుకు వాడతాం?:** మీ కంపెనీకి ఒక ప్రత్యేకమైన అవసరం ఉంది (ఉదాహరణకు: మీ యూజర్ల గత 5 సంవత్సరాల డేటాను బట్టి భవిష్యత్తులో బిజినెస్ ఎలా ఉంటుందో ప్రెడిక్ట్ చేసే ఒక సొంత AI మోడల్ కావాలి).
- **SageMaker ఏం చేస్తుంది?:** డేటాను క్లీన్ చేయడం, AI మోడల్‌కి ట్రైనింగ్ ఇవ్వడం (Training), దాన్ని టెస్ట్ చేయడం, మరియు చివరగా దాన్ని ఒక లైవ్ API (Endpoint) లాగా క్లౌడ్‌లో డిప్లాయ్ చేయడం... ఈ ప్రాసెస్ మొత్తాన్ని ఒకే చోట చేయడానికి SageMaker వాడుతాం.

---

## 2. Amazon Rekognition (ఇమేజ్ మరియు వీడియో అనలిటిక్స్ API)

**Rekognition అంటే ఏంటి?** ఇది కంప్యూటర్ విజన్ (Computer Vision) సర్వీస్. అంటే దీనికి కళ్ళు ఉన్నాయి అనుకోవచ్చు. మీరు ఒక ఇమేజ్ లేదా వీడియో ఇస్తే, అది అందులో ఏముందో స్క్యాన్ చేసి చెబుతుంది.

- **రియల్-టైమ్ డెవలపర్ యూజ్ కేసెస్:**
- **Object & Scene Detection:** ఒక ఫోటో ఇస్తే, అందులో కార్ ఉందా, కుక్క ఉందా, మనుషులు ఉన్నారా అని కనిపెడుతుంది.
- **Facial Analysis:** ఫోటోలోని వ్యక్తి నవ్వుతున్నాడా, బాధగా ఉన్నాడా, వారి వయసు ఎంత ఉండవచ్చు అని అంచనా వేస్తుంది.
- **Face Verification (KYC):** యూజర్ అప్‌లోడ్ చేసిన ఆధార్ కార్డ్ ఫోటో మరియు వారి సెల్ఫీ ఫోటో రెండు ఒక్కటేనా కాదా అని మ్యాచ్ చేస్తుంది (బ్యాంకింగ్ యాప్స్ లో KYC కి ఇది బెస్ట్).
- **Content Moderation:** మీ React యాప్‌లో యూజర్లు ఎవరైనా అసభ్యకరమైన (Adult) ఫోటోలు అప్‌లోడ్ చేస్తుంటే, ఇది ఆటోమేటిక్‌గా వాటిని కనిపెట్టి బ్లాక్ చేస్తుంది.

---

## 3. Amazon Textract (స్మార్ట్ డ్రాఫ్ట్/OCR సర్వీస్)

**Textract అంటే ఏంటి?** సాధారణ OCR (Optical Character Recognition) టూల్స్ కేవలం ఒక ఫోటోలోని అక్షరాలను మాత్రమే టెక్స్ట్‌గా మారుస్తాయి. కానీ **Textract** అనేది AI ఆధారిత OCR.

- **ఎందుకు ప్రత్యేకమైనది?:** ఇది ఒక డాక్యుమెంట్ (PDF లేదా ఇమేజ్) లోని డేటాను కేవలం ముక్కలుగా కాకుండా, **టేబుల్స్ (Tables) మరియు ఫారమ్స్ (Forms - Key-Value pairs)** రూపంలో చాలా స్మార్ట్‌గా గుర్తిస్తుంది.
- _ఉదాహరణ:_ మీరు ఒక హాస్పిటల్ బిల్లు లేదా ఒక కంపెనీ ఇన్వాయిస్ (Invoice) ని ఫోటో తీసి Textract కి ఇస్తే, అది "ప్రొడక్ట్ పేరు ఏంటి, దాని ధర ఎంత, టోటల్ బిల్ ఎంత" అని టేబుల్ ఫార్మాట్‌లో డేటాను క్లియర్ గా విడగొట్టి మీ Node.js యాప్‌కి JSON రూపంలో ఇస్తుంది. డెవలపర్లు ఆ డేటాను నేరుగా డేటాబేస్‌లో సేవ్ చేసుకోవచ్చు.

---

## 4. Amazon Comprehend (టెక్స్ట్ అనలిటిక్స్ / NLP సర్వీస్)

**Comprehend అంటే ఏంటి?** ఇది **NLP (Natural Language Processing)** సర్వీస్. అంటే దీనికి ఒక పెద్ద పారాగ్రాఫ్ టెక్స్ట్ లేదా యూజర్ రివ్యూలు ఇస్తే, అది ఆ భాషను చదివి, అర్థం చేసుకుని, అందులోని అంతరార్థాన్ని చెబుతుంది.

- **రియల్-టైమ్ డెవలపర్ యూజ్ కేసెస్:**
- **Sentiment Analysis (అభిప్రాయ విశ్లేషణ):** మీ ఈ-కామర్స్ వెబ్‌సైట్‌లో ఒక యూజర్ రాసిన రివ్యూని చదివి, అది పాజిటివ్ రివ్యూనా (`Positive`), నెగటివ్ రివ్యూనా (`Negative`), లేదా నార్మల్‌గా ఉందా (`Neutral`) అని కనిపెడుతుంది.
- **Entity Recognition:** ఒక పెద్ద ఆర్టికల్ ఇస్తే అందులో ఉన్న వ్యక్తుల పేర్లు, ఊర్ల పేర్లు, తేదీలు, మరియు బ్రాండ్ పేర్లను విడిగా లిస్ట్ తీసి ఇస్తుంది.
- **Language Detection:** ఇచ్చిన టెక్స్ట్ ఏ భాషలో ఉందో (తెలుగు, ఇంగ్లీష్, హిందీ మొదలైనవి) ఆటోమేటిక్‌గా గుర్తిస్తుంది.

---

**క్విక్ సమ్మరీ:**

- సొంతంగా AI/ML మోడల్స్ బిల్డ్ మరియు డిప్లాయ్ చేయడానికి ➡️ **Amazon SageMaker**
- ఫోటోలు, వీడియోలలోని ముఖాలను, వస్తువులను కనిపెట్టడానికి ➡️ **Amazon Rekognition**
- ఇన్వాయిస్లు, PDF ల నుండి టేబుల్స్ డేటాను స్మార్ట్‌గా తీయడానికి ➡️ **Amazon Textract**
- యూజర్ల రివ్యూలు పాజిటివా, నెగటివా అని టెక్స్ట్ అనలైజ్ చేయడానికి ➡️ **Amazon Comprehend**

# 18. Cost Management

మిషన్ లెర్నింగ్ తర్వాత మీ లిస్ట్‌లోని చివరి, మరియు ఏ కంపెనీకైనా అత్యంత ప్రాముఖ్యమైన టాపిక్ — **Cost Management (ఖర్చుల నియంత్రణ మరియు బడ్జెట్ సర్వీసెస్)**.

క్లౌడ్‌ కంప్యూటింగ్‌లో ఎంత సులభంగా సర్వర్లు క్రియేట్ చేయవచ్చో, అంతే సులభంగా బిల్లు కూడా పెరిగిపోయే ప్రమాదం ఉంది. అందుకే సాఫ్ట్‌వేర్ ఆర్కిటెక్ట్‌గా లేదా డెవలపర్‌గా AWS లో బిల్లును ఎలా ట్రాక్ చేయాలి, ఖర్చులను ఎలా తగ్గించుకోవాలి (**Cost Optimization**) అని తెలియడం చాలా అవసరం.

వీటిని తెలుగులో సులభంగా అర్థం చేసుకుందాం:

---

## 1. AWS Cost Explorer (ఖర్చుల విశ్లేషణ టూల్)

**Cost Explorer అంటే ఏంటి?** ఇది మీ AWS అకౌంట్ యొక్క **ఖర్చుల డైరీ లేదా మనీ మేనేజర్ యాప్** లాంటిది.

- గత 12 నెలల్లో ఏ సర్వీస్‌కి ఎంత ఖర్చయింది, ఈ నెలలో ఎంత బిల్లు రావచ్చు (Forecasting) అని గ్రాఫ్‌లు, చార్ట్‌ల రూపంలో ఇది చూపిస్తుంది.
- _ఉదాహరణ:_ "నాకు కేవలం EC2 సర్వర్ల మీద మాత్రమే ఎంత ఖర్చవుతోంది?" లేదా "ముంబై రీజియన్‌లో ఎంత బిల్లు పడుతోంది?" అని మీరు ఫిల్టర్స్ పెట్టి క్లియర్ గా చూసుకోవచ్చు. బిల్లు అకస్మాత్తుగా పెరిగితే ఎక్కడ పెరిగిందో కనిపెట్టడానికి ఇది బెస్ట్ టూల్.

---

## 2. AWS Budgets (బడ్జెట్ అలారాలు)

**AWS Budgets అంటే ఏంటి?** ఇది మీ అకౌంట్‌కి **ఖర్చుల పరిమితి (Limit)** సెట్ చేసే ఒక అలర్ట్ సిస్టమ్.

- _ఉదాహరణ:_ మీరు కొత్తగా AWS నేర్చుకుంటున్నారు. మీ బడ్జెట్ నెలకు $10 (సుమారు 850 రూపాయలు) అనుకుందాం. మీరు AWS Budgets లో ఒక రూల్ పెట్టవచ్చు: "నా బిల్లు ఎప్పుడైనా నా బడ్జెట్‌లో 80% (అంటే $8) కి చేరినా, లేదా ఈ నెల చివరకు $10 దాటిపోయే అవకాశం ఉన్నా... నాకు వెంటనే ఇమెయిల్ లేదా ఎస్ఎంఎస్ అలర్ట్ పంపు".
- దీనివల్ల నెలాఖరున ఊహించని విధంగా పెద్ద మొత్తంలో బిల్లు రాకుండా ముందే జాగ్రత్త పడవచ్చు.

---

## 3. సర్వర్ల ఖర్చు తగ్గించే వ్యూహాలు (Pricing Models)

మనం 3వ టాపిక్ (Compute) లో EC2 సర్వర్ల గురించి చెప్పుకున్నాం. వాటిని కొనే పద్ధతిని బట్టి బిల్లును భారీగా తగ్గించుకోవచ్చు:

- **Reserved Instances (RI - రిజర్వ్డ్ ఇన్‌స్టాన్సెస్):** మీ కంపెనీ అప్లికేషన్ కనీసం 1 సంవత్సరం లేదా 3 సంవత్సరాల పాటు నిరంతరం నడవాల్సిందే అని మీకు ముందే తెలుసు అనుకుందాం. అప్పుడు మీరు AWS తో ఒక అగ్రిమెంట్ లాగా చేసుకుని "నేను ఈ సర్వర్‌ను 3 ఏళ్ళు వాడుతాను" అని ముందే చెబితే, AWS మీకు నార్మల్ బిల్లుపై **కనీసం 50% నుండి 72% వరకు డిస్కౌంట్** ఇస్తుంది. (ఇది హోటల్‌లో రోజువారీ రెంట్ కట్టకుండా, ఒకేసారి సంవత్సరం లీజుకి తీసుకున్నట్టు).
- **Savings Plans:** ఇది Reserved Instances లాంటిదే కానీ కొంచెం ఫ్లెక్సిబుల్ (Flexible) గా ఉంటుంది. ఇక్కడ మీరు ఒక నిర్దిష్టమైన సర్వర్ సైజ్ అని కాకుండా, "నేను గంటకు $10 ఖర్చు పెడతాను" అని కమిట్ అవుతారు. మీరు EC2 వాడినా, Fargate వాడినా, లేదా AWS Lambda వాడినా... అన్నింటికీ ఈ డిస్కౌంట్ వర్తిస్తుంది.
- **Spot Instances (స్పాట్ ఇన్‌స్టాన్సెస్):** AWS డేటా సెంటర్లలో కొన్ని సర్వర్లు ఎవరూ వాడకుండా ఖాళీగా పడి ఉంటాయి. ఆ ఖాళీ సర్వర్లను AWS వేలం (Bidding) కి పెడుతుంది. దీనివల్ల నార్మల్ ధర కంటే **90% తక్కువ ధరకే** సర్వర్లు లభిస్తాయి.
- _కండిషన్:_ ఒకవేళ ఆ సర్వర్ వేరే ఎవరికైనా పూర్తి ధరకి అవసరమైతే, AWS మీకు కేవలం **2 నిమిషాల ముందే నోటీస్ ఇచ్చి** ఆ సర్వర్‌ని వెనక్కి లాగేసుకుంది. అందుకే దీన్ని మెయిన్ ప్రొడక్షన్ యాప్స్‌కి వాడకూడదు. బ్యాక్‌గ్రౌండ్ డేటా ప్రాసెసింగ్, టెస్టింగ్ లేదా బ్యాచ్ జాబ్స్ లాంటి వాటికి ఇది సూపర్ ఛాయిస్.

## 4. Cost Optimization (ఖర్చుల ఆప్టిమైజేషన్ బెస్ట్ ప్రాక్టీసెస్)

కంపెనీలలో బిల్లు తగ్గించడానికి డెవలపర్లు చేసే కొన్ని ముఖ్యమైన పనులు:

- **Right-sizing (సరైన సైజ్ ఎంచుకోవడం):** మీ Node.js యాప్‌కి కేవలం 2GB RAM సరిపోతుంది, కానీ మీరు పొరపాటున 16GB RAM ఉన్న పెద్ద సర్వర్‌ని క్రియేట్ చేశారు అనుకుందాం. CloudWatch లో చూస్తే CPU వాడకం ఎప్పుడూ 5% కంటే తక్కువ ఉంటుంది. అప్పుడు ఆ సర్వర్ సైజ్ తగ్గించి చిన్న సర్వర్‌కి మార్చడాన్నే Right-sizing అంటారు.
- **S3 Lifecycle Policies:** మనం స్టోరేజ్ టాపిక్‌లో చెప్పుకున్నట్టు, పాత ఫైల్స్‌ని Glacier కి మార్చడం ద్వారా స్టోరేజ్ బిల్లు తగ్గించవచ్చు.
- **Unused Resources:** ఎవరూ వాడకుండా ఖాళీగా ఉన్న Elastic IPలు, పాత EBS హార్డ్ డిస్క్ స్నాప్‌షాట్స్, లేదా వదిలేసిన లోడ్ బ్యాలెన్సర్లను వెతికి డిలీట్ చేయడం.

---

**క్విక్ సమ్మరీ:**

- గత బిల్లులు చూడటానికి, భవిష్యత్తు బిల్లు అంచనా వేయడానికి ➡️ **AWS Cost Explorer**
- బిల్లు పరిమితి దాటితే ఈమెయిల్ అలర్ట్స్ పొందడానికి ➡️ **AWS Budgets**
- 1 లేదా 3 ఏళ్ల పాటు సర్వర్లు వాడతామని డిస్కౌంట్ పొందడానికి ➡️ **Reserved Instances / Savings Plans**

# 19. Backup & Disaster Recovery

ఖర్చుల నియంత్రణ తర్వాత మనం నేర్చుకోబోయే, మరియు మీ లిస్ట్‌లోని అత్యంత కీలకమైన చివరి విభాగం — **Backup & Disaster Recovery (బ్యాకప్ మరియు విపత్తు నిర్వహణ సర్వీసెస్)**.

సాఫ్ట్‌వేర్ ఆర్కిటెక్చర్‌లో "డేటా అత్యంత విలువైంది". భూకంపం రావడం వల్లనో, లేదా హ్యాకర్లు అటాక్ చేయడం వల్లనో, లేదా ఒక డెవలపర్ పొరపాటున డేటాబేస్‌ను డిలీట్ చేయడం వల్లనో మీ ప్రొడక్షన్ డేటా పోతే కంపెనీలు కోట్లలో నష్టపోతాయి. అలాంటి విపత్కర పరిస్థితుల (Disasters) నుండి మీ React, Node.js అప్లికేషన్‌ను మరియు డేటాను ఎలా కాపాడుకోవాలో ఈ విభాగం చెబుతుంది.

దీనిని తెలుగులో సులభంగా అర్థం చేసుకుందాం:

---

## 1. Snapshots (స్నాప్‌షాట్స్)

**Snapshot అంటే ఏంటి?** ఇది మీ హార్డ్ డిస్క్ (EBS Volume) లేదా డేటాబేస్ (RDS) యొక్క ఒక నిర్దిష్ట సమయంలోని **పాయింట్-ఇన్-టైమ్ ఫోటో (Point-in-time Copy)** లాంటిది.

- _ఉదాహరణ:_ ఇవాళ ఉదయం 10 గంటలకి మీ డేటాబేస్ పర్ఫెక్ట్‌గా ఉంది. మీరు దానికి ఒక స్నాప్‌షాట్ తీసి పెట్టారు. మధ్యాహ్నం 12 గంటలకి ఒక బగ్ వల్ల డేటాబేస్ పాడైపోయింది.
- **పరిహారం:** మీరు ఉదయం 10 గంటలకి తీసిన స్నాప్‌షాట్‌ను వాడుకుని, కొత్త డేటాబేస్‌ను క్షణాల్లో రీస్టోర్ (Restore) చేసుకోవచ్చు. ఇవి కేవలం మారిన డేటాను మాత్రమే సేవ్ చేస్తాయి (Incremental), కాబట్టి స్టోరేజ్ బిల్లు కూడా తక్కువ పడుతుంది.

---

## 2. AWS Backup (సెంట్రల్ బ్యాకప్ మేనేజర్)

**AWS Backup అంటే ఏంటి?** క్లౌడ్‌లో EC2 కి విడిగా, S3 కి విడిగా, RDS కి విడిగా మాన్యువల్‌గా బ్యాకప్స్ తీసుకోవడం కష్టం. వీటన్నింటినీ ఒకే చోట నుండి ఆటోమేట్ చేసే **సెంట్రల్ మేనేజర్** ఈ AWS Backup.

- **Automated Policies:** ఇందులో మనం ఒక "Backup Plan" రాయవచ్చు. ఉదాహరణకు: "ప్రతిరోజూ రాత్రి 12 గంటలకి RDS మరియు EBS ల బ్యాకప్ తీయాలి, ఆ బ్యాకప్ ఫైల్స్‌ను 30 రోజుల తర్వాత ఆటోమేటిక్‌గా డిలీట్ చేయాలి".
- **Cross-Account Backup (క్రాస్-అకౌంట్ బ్యాకప్):** ఒకవేళ మీ మెయిన్ AWS అకౌంట్ హ్యాక్ అయినా లేదా డిలీట్ అయినా మీ బ్యాకప్స్ సేఫ్‌గా ఉండాలి కదా! అందుకే AWS Backup ద్వారా, మీ మెయిన్ అకౌంట్‌లోని డేటా బ్యాకప్‌ను పూర్తిగా వేరే సురక్షితమైన **Secondary AWS Account** లోకి ఆటోమేటిక్‌గా కాపీ చేసి దాచవచ్చు.

---

## 3. Multi-Region (మల్టీ-రీజియన్ ఆర్కిటెక్చర్)

మనం బేసిక్స్‌లో చెప్పుకున్నాం — AWS కి ప్రపంచవ్యాప్తంగా డేటా సెంటర్లు (Regions) ఉన్నాయి.

- ఒకవేళ ముంబై (Mumbai) రీజియన్‌లో ఏదైనా ప్రకృతి విపత్తు జరిగి, అక్కడ ఉన్న డేటా సెంటర్లన్నీ డౌన్ అయిపోయినా మీ యాప్ ఆగకూడదు అనుకుంటే... మీ ఇన్‌ఫ్రాస్ట్రక్చర్‌ను ముంబైతో పాటు సింగపూర్ (Singapore) లేదా వర్జీనియా (N. Virginia) రీజియన్‌లలో కూడా సమాంతరంగా రన్ చేయడాన్నే **Multi-Region ఆర్కిటెక్చర్** అంటారు.
- **Cross-Region Replication:** ముంబై S3 లో పడే ప్రతి ఫైల్, RDS లో పడే ప్రతి డేటా రో (Data row) ఆటోమేటిక్‌గా సింగపూర్ రీజియన్‌కి కూడా కాపీ అయిపోతుంది. ముంబై డౌన్ అవ్వగానే, Route 53 ట్రాఫిక్‌ను సింగపూర్‌కి మళ్ళిస్తుంది.

---

## 4. Disaster Recovery (DR) Strategies

క్లౌడ్ ప్రపంచంలో ఒక విపత్తు జరిగినప్పుడు అప్లికేషన్‌ను ఎంత వేగంగా రికవర్ చేయాలి అనే దానికి 4 రకాల వ్యూహాలు (Strategies) ఉన్నాయి. (ఖర్చు మరియు స్పీడ్ ఆధారంగా ఇవి మారుతుంటాయి):

```
[ తక్కువ ఖర్చు / స్లో రికవరీ ] ─────────────────────────────► [ ఎక్కువ ఖర్చు / ఇన్స్టంట్ రికవరీ ]
Backup & Restore  ======>  Pilot Light  ======>  Warm Standby  ======>  Multi-Site (Active-Active)

```

1. **Backup & Restore (బ్యాకప్ & రీస్టోర్):** ఇది అత్యంత చౌకైన పద్ధతి. డేటాను కేవలం బ్యాకప్ (Snapshots/S3) తీసి పెడతాం. విపత్తు జరిగినప్పుడు కొత్తగా సర్వర్లు క్రియేట్ చేసి డేటాను రీస్టోర్ చేస్తాం. దీనికి కొన్ని గంటల సమయం పడుతుంది (High RTO).
2. **Pilot Light (పైలట్ లైట్):** ఇంట్లో గ్యాస్ గీజర్ కి ఎప్పుడూ ఒక చిన్న లైట్ (Pilot light) వెలుగుతూ ఉంటుంది కదా, మనం బటన్ నొక్కగానే పెద్ద మంట వస్తుంది. అలాగే ఇక్కడ మీ డేటాబేస్ మాత్రం వేరే రీజియన్‌లో లైవ్‌గా రన్ అవుతూ డేటాను కాపీ చేసుకుంటుంది, కానీ అప్లికేషన్ సర్వర్లు (EC2) మాత్రం ఆఫ్ చేసి ఉంటాయి. విపత్తు రాగానే ఆ సర్వర్లను ఆన్ చేస్తారు. రికవరీకి కొన్ని నిమిషాలు పడుతుంది.
3. **Warm Standby (వార్మ్ స్టాండ్‌బై):** వేరే రీజియన్‌లో డేటాబేస్‌తో పాటు, మీ అప్లికేషన్ సర్వర్లను కూడా చాలా చిన్న సైజులో (`t2.micro` లాంటివి) ఎప్పుడూ రన్ చేసి ఉంచుతారు. విపత్తు రాగానే ఆటో స్కేలింగ్ ద్వారా వాటి సైజ్ పెంచి ట్రాఫిక్‌ను పంపుతారు. ఇది చాలా వేగంగా (కొన్ని నిమిషాల్లోనే) రికవర్ అవుతుంది.
4. **Multi-Site / Active-Active (మల్టీ-సైట్):** ఇది అత్యంత ఖరీదైన మరియు విలాసవంతమైన పద్ధతి. రెండు వేర్వేరు రీజియన్లలో (ఉదాహరణకు ముంబై మరియు సింగపూర్) మీ పూర్తి స్థాయి React, Node.js యాప్ మరియు డేటాబేస్ 100% లైవ్‌గా రన్ అవుతుంటాయి. ట్రాఫిక్ ఇద్దరికీ సమానంగా వెళ్తుంది. ఒక రీజియన్ పూర్తిగా పోయినా, యూజర్‌కి కనీసం ఒక మిల్లీసెకన్ కూడా తేడా తెలియకుండా యాప్ రన్ అవుతూనే ఉంటుంది (Zero Downtime).

---

**క్విక్ సమ్మరీ:**

- హార్డ్ డిస్క్ లేదా డేటాబేస్ యొక్క తాత్కాలిక కాపీ ➡️ **Snapshots**
- ఒకే చోట నుండి అన్ని బ్యాకప్స్ ఆటోమేట్ చేయడానికి ➡️ **AWS Backup**
- హ్యాకర్ల నుండి రక్షణకు బ్యాకప్‌ను వేరే అకౌంట్‌లో దాచడం ➡️ **Cross-Account Backup**
- ఒక దేశంలోని డేటా సెంటర్ పూర్తిగా డౌన్ అయినా యాప్ రన్ చేయడం ➡️ **Multi-Region**
- విపత్తు సమయాల్లో యాప్‌ను రీస్టోర్ చేసే పద్ధతులు ➡️ **Disaster Recovery Strategies**

# 20. Advanced Services

ఈ 19 టాపిక్స్‌తో సిలబస్ పూర్తయింది అనుకున్నాం, కానీ మీరు **Topic 20** లో క్లౌడ్ ఆర్కిటెక్చర్‌కి కిరీటం లాంటి అత్యంత అడ్వాన్స్‌డ్ సర్వీసులను తీసుకొచ్చారు!

ఒక సీనియర్ ఫుల్-స్టాక్ లేదా ఆర్కిటెక్ట్ లెవెల్లో... కాంప్లెక్స్ బిజినెస్ వర్క్‌ఫ్లోస్ మేనేజ్ చేయడానికి, యూజర్ అథెంటికేషన్ (Sign-in/Sign-up) సెటప్ చేయడానికి, మరియు మల్టిపుల్ AWS అకౌంట్లను కంట్రోల్ చేయడానికి ఈ **Advanced Services** చాలా కీలకం.

వీటిని కూడా మన స్టైల్‌లో క్లియర్ గా, తెలుగులో అర్థం చేసుకుందాం:

---

## 1. AWS Step Functions (వర్క్‌ఫ్లో ఆర్కెస్ట్రేషన్)

**Step Functions అంటే ఏంటి?** మీ Node.js బ్యాకెండ్‌లో ఒక పెద్ద బిజినెస్ ప్రాసెస్ ఉంది అనుకుందాం.

- _ఉదాహరణ (ఈ-కామర్స్ ఆర్డర్):_ యూజర్ ప్రొడక్ట్ కొనగానే:

1. ముందుగా `Lambda-1` (పేమెంట్ చెక్ చేయాలి).
2. పేమెంట్ సక్సెస్ అయితే ➡️ `Lambda-2` (ఇన్వెంటరీ అప్‌డేట్ చేయాలి) ➡️ ఆపై `SNS` (ఇమెయిల్ పంపాలి).
3. ఒకవేళ పేమెంట్ ఫెయిల్ అయితే ➡️ `Lambda-3` (ఆర్డర్ క్యాన్సిల్ చేసి యూజర్‌కి అలర్ట్ పంపాలి).

ఇలాంటి మల్టిపుల్ ల్యాంబ్డా ఫంక్షన్స్ మరియు సర్వీసులను ఒకదాని తర్వాత ఒకటి కండిషన్స్ (`if-else`, loops, error handling) బట్టి నడిపించే **విజువల్ వర్క్‌ఫ్లో మేనేజర్** ఈ Step Functions. దీనివల్ల కోడ్‌లో కాంప్లెక్స్ స్టేట్ మేనేజ్‌మెంట్ రాయాల్సిన అవసరం తప్పుతుంది.

---

## 2. Amazon Cognito (యూజర్ సైన్-ఇన్ & సెక్యూరిటీ)

**Cognito అంటే ఏంటి?** మీ React.js మరియు Node.js యాప్‌ల కోసం **Sign-Up, Sign-In, Password Reset, మరియు MFA (Two-Factor Authentication)** ఫీచర్లను కేవలం కొన్ని నిమిషాల్లో రెడీ చేసే ఒక సర్వర్‌లెస్ యూజర్ మేనేజ్‌మెంట్ సర్వీస్.

- **User Pools:** ఇది మీ యూజర్ల డేటాబేస్ లాంటిది. ఇందులో యూజర్లు తమ ఇమెయిల్, పాస్‌వర్డ్‌తో అకౌంట్ క్రియేట్ చేసుకోవచ్చు. అలాగే Google, Facebook, Apple లేదా మీ ఆఫీస్ **SSO (Single Sign-On)** ద్వారా డైరెక్ట్‌గా లాగిన్ అయ్యేలా సెటప్ చేయవచ్చు.
- **Identity Pools:** లాగిన్ అయిన యూజర్లకి మీ AWS రిసోర్సెస్ (ఉదాహరణకు: డైరెక్ట్‌గా S3 బకెట్‌లో ఫోటో అప్‌లోడ్ చేయడానికి) తాత్కాలిక అనుమతులు (Temporary AWS Credentials) ఇవ్వడానికి ఇది వాడుతారు.

---

## 3. AWS Organizations & AWS Control Tower (మల్టిపుల్ అకౌంట్స్ మేనేజ్‌మెంట్)

ఒక పెద్ద కంపెనీలో ఒకే ఒక AWS అకౌంట్ ఉండదు. `Dev` కి ఒక అకౌంట్, `Test` కి ఒకటి, `Prod` కి ఒకటి, `Finance` బిల్లింగ్ కి ఒకటి... ఇలా వందల అకౌంట్స్ ఉంటాయి.

- **AWS Organizations:** ఈ వందలాది AWS అకౌంట్లను ఒకే ఒక "Master Account" కింద లింక్ చేసి, అన్నింటికీ ఒకే సెంట్రల్ బిల్లింగ్ (Consolidated Billing) సెట్ చేయడానికి మరియు అన్ని అకౌంట్లపై సెక్యూరిటీ రూల్స్ (SCP - Service Control Policies) పెట్టడానికి ఇది వాడుతారు.
- **AWS Control Tower:** ఒక కొత్త అకౌంట్ క్రియేట్ చేసినప్పుడు, అది కంపెనీ రూల్స్ ప్రకారం సురక్షితంగా ఉందో లేదో చెక్ చేస్తూ, బెస్ట్ ప్రాక్టీసెస్‌తో కూడిన ఒక సురక్షితమైన ఎన్విరాన్‌మెంట్‌ను (Landing Zone) ఆటోమేటిక్‌గా సెటప్ చేసే గవర్నెన్స్ టూల్ ఇది.

---

## 4. AWS Systems Manager (SSM) & AWS AppConfig

- **AWS Systems Manager (SSM):** ఇది మీ క్లౌడ్ ఇన్‌ఫ్రాస్ట్రక్చర్‌ని మేనేజ్ చేసే ఒక రిమోట్ కంట్రోల్. దీనిలోని **Parameter Store** ని డెవలపర్లు ఎక్కువగా వాడుతుంటారు. మీ Node.js యాప్ యొక్క కాన్ఫిగరేషన్ కీస్ (DB URLs, API Keys) ని ఇందులో భద్రంగా దాచుకోవచ్చు. పైగా దీని ద్వారా మీ EC2 సర్వర్లలోకి SSH కీస్ లేకుండానే సెక్యూర్ గా లాగిన్ అవ్వచ్చు (Session Manager).
- **AWS AppConfig:** మీ React లేదా Node.js అప్లికేషన్‌లో ఏదైనా కొత్త ఫీచర్‌ని రిలీజ్ చేశారు. దాన్ని ఒకేసారి అందరికీ కాకుండా కేవలం 10% యూజర్లకే చూపించాలి (Feature Flags / Feature Toggles) అనుకుంటే, మళ్లీ మళ్లీ కోడ్ డిప్లాయ్ చేయకుండా, కేవలం AppConfig లో ఒక్క క్లిక్‌తో ఆ ఫీచర్‌ని ఆన్/ఆఫ్ లేదా స్కేల్ చేయవచ్చు.

---

## 5. AWS AppSync (మేనేజ్డ్ GraphQL API)

మనం 10వ టాపిక్‌లో **API Gateway (REST/HTTP API)** గురించి చెప్పుకున్నాం. కానీ మీ React యాప్ లో మీకు REST API వద్దు, కేవలం **GraphQL** మాత్రమే కావాలి అనుకుంటే **AWS AppSync** బెస్ట్ ఛాయిస్.

- ఇది ఒక ఫుల్లీ మేనేజ్డ్ GraphQL సర్వీస్. దీని ద్వారా మీ React యాప్ ఒకే ఒక క్వెరీతో DynamoDB, Lambda, లేదా RDS నుండి కావాల్సిన డేటాను ఖచ్చితమైన సైజులో లాక్కోవచ్చు.
- ఇందులో **Real-time Subscriptions** మరియు **Offline Data Sync** (ఇంటర్నెట్ పోయినా యాప్ లోకల్‌గా పనిచేసి, నెట్ రాగానే క్లౌడ్‌తో సింక్ అవ్వడం) లాంటి పవర్ఫుల్ ఫీచర్స్ ఇన్-బిల్ట్‌గా ఉంటాయి.

---

**క్విక్ సమ్మరీ:**

- మల్టిపుల్ ల్యాంబ్డా ఫంక్షన్స్ తో కాంప్లెక్స్ వర్క్‌ఫ్లోస్ బిల్డ్ చేయడానికి ➡️ **Step Functions**
- React/Node యాప్స్ కి లాగిన్, సైన్-అప్ మరియు సోషల్ లాగిన్స్ ఇవ్వడానికి ➡️ **Amazon Cognito**
- వందలాది AWS అకౌంట్లను ఒకే చోట గ్రూప్ చేయడానికి ➡️ **AWS Organizations / Control Tower**
- సర్వర్ కాన్ఫిగరేషన్స్, పారామీటర్స్ మరియు సెక్యూర్ లాగిన్ కోసం ➡️ **Systems Manager (SSM)**
- అప్లికేషన్ కోడ్ మార్చకుండా ఫీచర్ ఫ్లాగ్స్ (On/Off) మేనేజ్ చేయడానికి ➡️ **AWS AppConfig**
- సూపర్ ఫాస్ట్, రియల్-టైమ్ GraphQL API ల కోసం ➡️ **AWS AppSync**

# 21. Architecture Patterns

మీరు 20 టాపిక్స్‌తో సిలబస్ ముగిసింది అనుకుంటే... ఇప్పుడు క్లౌడ్ అండ్ సాఫ్ట్‌వేర్ ఇంజనీరింగ్‌లోనే అత్యంత పవర్‌ఫుల్ అండ్ అల్టిమేట్ టాపిక్‌ను టేబుల్ పైకి తెచ్చారు — **Architecture Patterns & Concepts (ఆర్కిటెక్చర్ డిజైన్ పద్ధతులు)**.

ఇప్పటివరకు మనం నేర్చుకున్న 20 విభాగాలు (EC2, S3, RDS, Lambda, SQS, WAF మొదలైనవి) అన్నీ విడివిడి ఇటుకలు లాంటివి. మరి ఆ ఇటుకలన్నింటినీ కలిపి ఒక పెద్ద, బలమైన భవనాన్ని (ఒక సూపర్ అప్లికేషన్‌ను) ఎలా నిర్మించాలో చెప్పే **ఇంజనీరింగ్ బ్లూప్రింట్లే** ఈ ఆర్కిటెక్చర్ ప్యాటర్న్స్.

సాఫ్ట్‌వేర్ ఆర్కిటెక్ట్‌గా లేదా సీనియర్ డెవలపర్‌గా మీకు ఇవి ప్రాణంతో సమానం. వీటిని తెలుగులో చాలా విజువల్ గా, సులభంగా అర్థం చేసుకుందాం:

---

## 🏗️ Core Architecture Patterns (ఆర్కిటెక్చర్ రకాలు)

### 1. Three-tier Architecture (సాంప్రదాయ మూడు అంచెల పద్ధతి)

ఇది దశాబ్దాలుగా వాడుతున్న మోస్ట్ స్టాండర్డ్ ప్యాటర్న్. మీ అప్లికేషన్‌ను 3 విభాగాలుగా విడగొడతాం:

- **Presentation Tier (ఫ్రంటెండ్):** యూజర్‌కి కనిపించే స్క్రీన్ (React.js యాప్ - S3/CloudFront లో ఉంటుంది).
- **Application/Logic Tier (బ్యాకెండ్):** మీ మెయిన్ బిజినెస్ లాజిక్ (Node.js/Express API - EC2 లేదా ALB వెనుక ఉంటుంది).
- **Data Tier (డేటాబేస్):** మీ డేటా స్టోరేజ్ (Amazon RDS/MySQL).

> **లాభం:** ఒక దానికి ఒకటి సంబంధం లేకుండా విడివిడిగా అప్‌డేట్ చేసుకోవచ్చు. సెక్యూరిటీ పరంగా డేటాబేస్‌ను ప్రైవేట్ సబ్‌నెట్‌లో దాచవచ్చు.

### 2. Microservices (మైక్రోసర్వీసెస్)

ఒక పెద్ద వెబ్‌సైట్ యొక్క పూర్తి కోడ్‌ను ఒకే ప్రాజెక్ట్‌గా (Monolith) రాయకుండా, చిన్న చిన్న స్వతంత్ర అప్లికేషన్లుగా విడగొట్టడం.

- _ఉదాహరణ (Amazon యాప్):_ `User Service`, `Order Service`, `Payment Service`, `Cart Service`... ఇలా అన్నింటికీ విడివిడి Node.js కోడ్, విడివిడి డేటాబేస్‌లు ఉంటాయి. ఇవి ఒకదానితో ఒకటి API Gateway లేదా SQS ద్వారా మాట్లాడుకుంటాయి. ఒక సర్వీస్ డౌన్ అయినా మిగతా వెబ్‌సైట్ నడుస్తూనే ఉంటుంది.

### 3. Serverless Architecture (సర్వర్‌లెస్ ప్యాటర్న్)

"మనం ఎటువంటి సర్వర్లు (EC2) రన్ చేయం, వాటిని మేనేజ్ చేయం" అనే ఫిలాసఫీ ఇది. మనం కేవలం మన Node.js కోడ్ ముక్కను రాసి **AWS Lambda** లో పెడతాం. ట్రాఫిక్ వచ్చినప్పుడు మాత్రమే ల్యాంబ్డా ఆన్ అవుతుంది, పని పూర్తి కాగానే ఆఫ్ అవుతుంది. స్టోరేజ్ కోసం **S3**, డేటాబేస్ కోసం **DynamoDB** వాడుతాం. ఖర్చు చాలా తక్కువ, స్కేలింగ్ అపరిమితం.

### 4. Event-driven Architecture (ఈవెంట్ ఆధారిత పద్ధతి)

సర్వీసులు ఒకదానితో ఒకటి నేరుగా (Synchronous గా) మాట్లాడకుండా, కేవలం "ఈవెంట్స్" (కదలికలు) ఆధారంగా పనిచేయడం.

- _ఉదాహరణ:_ యూజర్ ఆర్డర్ పెట్టగానే ఒక ఈవెంట్ జనరేట్ అయి **EventBridge** లేదా **SNS/SQS** క్యూలోకి వెళ్తుంది. దాన్ని చూసి పేమెంట్ సర్వీస్ అలర్ట్ అవుతుంది. ఒకవేళ పేమెంట్ సర్వీస్ కాసేపు డౌన్ అయినా, క్యూలో మెసేజ్ సేఫ్‌గా ఉంటుంది. అప్లికేషన్లు ఒకదానిపై ఒకటి ఆధారపడవు (Decoupled).

### 5. Multi-Region Deployment (మల్టీ-రీజియన్ డిప్లాయ్‌మెంట్)

మీ అప్లికేషన్‌ను ప్రపంచంలోని రెండు వేర్వేరు AWS డేటా సెంటర్లలో (ఉదాహరణకు ముంబై మరియు వర్జీనియా) ఒకేసారి లైవ్‌గా రన్ చేయడం. ఒక దేశంలో భూకంపం వచ్చి డేటా సెంటర్ పూర్తిగా పోయినా, రెండో రీజియన్ నుండి వెబ్‌సైట్ రన్ అవుతుంది (Zero Downtime).

## 📈 System Availability & Scaling Concepts (సిస్టమ్ గుణాలు)

- **High Availability (HA - నిరంతర లభ్యత):** మీ వెబ్‌సైట్ 24/7 ఎప్పుడూ ఆన్‌లైన్‌లో ఉండేలా డిజైన్ చేయడం. దీనికోసం మనం లోడ్ బ్యాలెన్సర్ (ALB) వెనుక కనీసం 2 లేదా 3 Availability Zones (వేర్వేరు బిల్డింగ్స్) లో సర్వర్లను రన్ చేస్తాం. ఒక బిల్డింగ్‌లో పవర్ పోయినా, ఇంకో బిల్డింగ్ నుండి యాప్ నడుస్తుంది.
- **Fault Tolerance (తప్పులను తట్టుకునే శక్తి):** అప్లికేషన్‌లో ఏదైనా ఒక హార్డ్‌వేర్ పార్ట్ లేదా సర్వర్ అకస్మాత్తుగా ఫెయిల్ అయినా... యూజర్‌కి కనీసం ఒక మిల్లీసెకన్ కూడా తేడా తెలియకుండా, పర్ఫార్మెన్స్ తగ్గకుండా సిస్టమ్ దాన్ని తట్టుకుని నిలబడటం. (HA అంటే యాప్ రన్ అవ్వడం, FT అంటే అసలు లోపం వచ్చినట్టు కూడా యూజర్‌కి తెలియకపోవడం).
- **Scalability (స్కేలబిలిటీ):** ట్రాఫిక్ పెరిగే కొద్దీ అప్లికేషన్ తన సామర్థ్యాన్ని పెంచుకోవడం. ఇందులో రెండు రకాలు:
- **Vertical Scaling (Scaling Up):** ఉన్న ఒక్క సర్వర్ సైజ్ పెంచడం (చిన్న రామ్ నుండి పెద్ద రామ్ కి మార్చడం - దీనికి ఒక లిమిట్ ఉంటుంది).
- **Horizontal Scaling (Scaling Out):** ఒకే సైజ్ ఉన్న సర్వర్ల సంఖ్యను పెంచడం (1 సర్వర్ కాస్తా 10 సర్వర్లుగా మారడం - **AWS Auto Scaling** చేసే పని ఇదే).

## 🚀 Modern Deployment Strategies (కోడ్ డిప్లాయ్‌మెంట్ పద్ధతులు)

మనం 13వ టాపిక్ (DevOps) లో కోడ్ డిప్లాయ్ చేయడం గురించి చెప్పుకున్నాం. ప్రొడక్షన్‌లో లైవ్ యూజర్లకి ఎటువంటి ఇబ్బంది లేకుండా కొత్త కోడ్‌ను ఎలా రిలీజ్ చేయాలో ఈ పద్ధతులు చెబుతాయి:

### Blue-Green Deployment

ఇందులో మన దగ్గర రెండు ఒకే రకమైన ప్రొడక్షన్ ఎన్విరాన్‌మెంట్లు ఉంటాయి:

- **Blue:** ప్రస్తుతం లైవ్ యూజర్లు వాడుతున్న పాత కోడ్ ఉన్న సర్వర్లు.
- **Green:** మనం కొత్తగా అప్‌డేట్ చేసిన కోడ్ ఉన్న సర్వర్లు.
- **ప్రాసెస్:** కొత్త కోడ్‌ను ముందుగా 'Green' లో డిప్లాయ్ చేసి అంతా బాగుందో లేదో టెస్ట్ చేస్తాం. అంతా పర్ఫెక్ట్ అనుకున్నాక, లోడ్ బ్యాలెన్సర్ లేదా Route 53 ద్వారా 100% ట్రాఫిక్‌ను ఒక్క క్షణంలో Blue నుండి Green కి మళ్ళిస్తాం. ఒకవేళ కొత్త కోడ్‌లో ఏదైనా బగ్ కనిపిస్తే, వెంటనే మళ్లీ Blue కి రోల్‌బ్యాక్ (Rollback) చేస్తాం. యూజర్‌కి అసలు వెబ్‌సైట్ డౌన్ అయినట్టే తెలియదు.

### Canary Deployment

ఇది కోడ్‌ను ఒకేసారి అందరికీ కాకుండా, ఒక చిన్న పక్షి (Canary) ని ప్రయోగించినట్టు చాలా జాగ్రత్తగా రిలీజ్ చేసే పద్ధతి.

- **ప్రాసెస్:** మీ కొత్త కోడ్‌ను ఒకే ఒక సర్వర్‌లో పెట్టి, వచ్చే ట్రాఫిక్‌లో కేవలం **5% లేదా 10% ట్రాఫిక్‌ను** మాత్రమే ఆ కొత్త సర్వర్‌కి పంపుతారు. మిగతా 90% యూజర్లు పాత కోడ్‌నే చూస్తారు.
- ఆ 10% మంది యూజర్లకి ఎర్రర్స్ రాకుండా యాప్ పర్ఫెక్ట్‌గా నడిస్తే... మెల్లగా 20%, 50% పెంచుకుంటూ చివరకు 100% ట్రాఫిక్‌ను కొత్త కోడ్‌కి మారుస్తారు. ఏమైనా బగ్ వస్తే కేవలం ఆ 5% మందికే ఎఫెక్ట్ అవుతుంది, వెంటనే కోడ్‌ను వెనక్కి తీసేయవచ్చు.

---

**క్విక్ సమ్మరీ:**

- ఫ్రంటెండ్, బ్యాకెండ్, డేటాబేస్ విడగొట్టడం ➡️ **Three-tier Architecture**
- పెద్ద యాప్‌ను చిన్న చిన్న స్వతంత్ర సర్వీసులుగా మార్చడం ➡️ **Microservices**
- సర్వర్లు లేకుండా కేవలం కోడ్ రన్ చేయడం ➡️ **Serverless**
- క్యూలు మరియు ఈవెంట్ల ద్వారా అప్లికేషన్లు లింక్ చేయడం ➡️ **Event-driven**
- సర్వర్ల సంఖ్యను పెంచడం ➡️ **Horizontal Scaling**
- పాత (Blue), కొత్త (Green) ఎన్విరాన్‌మెంట్లతో సేఫ్ డిప్లాయ్‌మెంట్ ➡️ **Blue-Green**
- కేవలం 5% మంది యూజర్లకే ముందుగా కొత్త కోడ్ చూపించడం ➡️ **Canary**

# 22. Hands-on Projects

ఇప్పటివరకు మీరు నేర్చుకున్న 21 విభాగాల థియరీ మరియు ఆర్కిటెక్చర్ కాన్సెప్ట్స్‌ని... ప్రాక్టికల్‌గా మీ చేతులతో బిల్డ్ చేసి, మీ రెజ్యూమేలో పెట్టుకోవడానికి వీలైన అత్యుత్తమ **22వ విభాగం — Hands-on Projects (లైవ్ ప్రాజెక్ట్స్)**.

సాఫ్ట్‌వేర్ డెవలపర్‌గా మీరు ఇంటర్వ్యూకి వెళ్ళినప్పుడు "నాకు AWS వచ్చు" అని చెప్పడం కంటే, "నేను AWS లో ఈ 11 ప్రాజెక్ట్స్ బిల్డ్ చేశాను" అని చెబితే మీ ప్రొఫైల్ రేంజ్ మారిపోతుంది. ఈ ప్రాజెక్ట్‌లన్నింటినీ మీరు మీ **React.js (ఫ్రంటెండ్)** మరియు **Node.js (బ్యాకెండ్)** నాలెడ్జ్‌తో ఎలా లింక్ చేసి బిల్డ్ చేయాలో ఒక్కొక్కటిగా క్లియర్ గా చూద్దాం:

### 1. Static Website on S3 + CloudFront

- **యాప్ రకం:** ఫ్రంటెండ్ డిప్లాయ్‌మెంట్ (React యాప్).
- **ఎలా చేయాలి?:** మీ React ప్రాజెక్ట్‌ను `npm run build` చేసి, ఆ ఫోల్డర్‌ను ఒక ప్రైవేట్ **S3 Bucket** లో పెట్టాలి. దాని ముందు **CloudFront (CDN)** సెట్ చేసి, **OAC (Origin Access Control)** ద్వారా S3 ని లాక్ చేయాలి. Route 53 లో మీ సొంత డొమైన్ లింక్ చేసి, ACM ద్వారా `https://` సర్టిఫికేట్ యాడ్ చేయాలి.

### 2. CRUD REST API with Lambda + API Gateway + DynamoDB

- **యాప్ రకం:** ప్యూర్ సర్వర్‌లెస్ బ్యాకెండ్ (Serverless CRUD).
- **ఎలా చేయాలి?:** ఒక యూజర్ మేనేజ్‌మెంట్ లేదా ప్రొడక్ట్ లిస్ట్ కోసం Node.js లో CRUD ఆపరేషన్స్ కోడ్ రాయాలి. **API Gateway (HTTP API)** ద్వారా `/users` (GET, POST, PUT, DELETE) రూట్స్ క్రియేట్ చేసి, వాటిని **AWS Lambda** కి లింక్ చేయాలి. లాంబ్డా ద్వారా డేటాను **DynamoDB** టేబుల్‌లో స్టోర్ మరియు రిట్రీవ్ చేయాలి.

### 3. EC2 + RDS Web Application

- **యాప్ రకం:** ట్రెడిషనల్ సర్వర్-బేస్డ్ వెబ్ యాప్.
- **ఎలా చేయాలి?:** ఒక **EC2 (Ubuntu)** సర్వర్ క్రియేట్ చేసి, అందులో మీ Node.js/Express.js API ని రన్ చేయాలి (PM2 వాడవచ్చు). డేటాబేస్ కోసం ఒక **Amazon RDS (MySQL or PostgreSQL)** ఇన్‌స్టాన్స్ క్రియేట్ చేసి, మీ Node.js కోడ్‌ని దానికి కనెక్ట్ చేయాలి. సర్వర్ ముందు ట్రాఫిక్ కోసం ఒక **Application Load Balancer (ALB)** సెట్ చేయాలి.

### 4. VPC with Public & Private Subnets

- **యాప్ రకం:** కోర్ నెట్‌వర్కింగ్ & సెక్యూరిటీ సెటప్.
- **ఎలా చేయాలి?:** మీ సొంత **VPC (Virtual Private Cloud)** క్రియేట్ చేయాలి. అందులో ఇంటర్నెట్ యాక్సెస్ ఉన్న రెండు **Public Subnets** (ఇందులో ALB ని పెడతాం), మరియు ఇంటర్నెట్ లేని రెండు **Private Subnets** (ఇందులో మీ EC2 బ్యాకెండ్ సర్వర్లు మరియు RDS డేటాబేస్ పెడతాం) క్రియేట్ చేయాలి. ప్రైవేట్ సబ్‌నెట్ లోని సర్వర్లు సాఫ్ట్‌వేర్ అప్‌డేట్ చేసుకోవడానికి **NAT Gateway** సెట్ చేయాలి.

### 5. CI/CD Pipeline

- **యాప్ రకం:** డెవఆప్స్ ఆటోమేషన్ (DevOps).
- **ఎలా చేయాలి?:** మీ React/Node.js కోడ్‌ను **AWS CodeCommit** (లేదా GitHub) లో పెట్టాలి. కోడ్ పుష్ అవ్వగానే ట్రిగ్గర్ అయ్యేలా **AWS CodePipeline** సెట్ చేయాలి. అది **AWS CodeBuild** ని పిలిచి `buildspec.yml` ద్వారా టెస్ట్ కేసులు మరియు బిల్డ్ రన్ చేయిస్తుంది. బిల్డ్ సక్సెస్ అయ్యాక **AWS CodeDeploy** ద్వారా కోడ్‌ను ఆటోమేటిక్‌గా EC2 లేదా S3 లోకి పంపుతుంది.

### 6. File Upload System

- **యాప్ రకం:** ప్రైవేట్ క్లౌడ్ స్టోరేజ్ సిస్టమ్.
- **ఎలా చేయాలి?:** మీ React యాప్ నుండి యూజర్లు ప్రొఫైల్ పిక్చర్స్ లేదా పిడిఎఫ్ ఫైల్స్ అప్‌లోడ్ చేసే సిస్టమ్. భద్రత కోసం ఫ్రంటెండ్ నేరుగా S3 కి అప్‌లోడ్ చేయకుండా, ముందుగా మీ Node.js (Lambda) బ్యాకెండ్‌ని అడగాలి. బ్యాకెండ్ **S3 Presigned URL** ని జనరేట్ చేసి ఇస్తుంది (ఇది కేవలం 5 నిమిషాలు మాత్రమే పనిచేస్తుంది). ఆ యుఆర్ఎల్ వాడి React యాప్ ఫైల్‌ను సురక్షితంగా S3 లో అప్‌లోడ్ చేస్తుంది.

### 7. Image Processing Pipeline

- **యాప్ రకం:** ఈవెంట్-డ్రివెన్ మీడియా ప్రాసెస్.
- **ఎలా చేయాలి?:** యూజర్ ఒక పెద్ద సైజ్ ఇమేజ్ (ఫోటో) అప్‌లోడ్ చేయగానే, ఆటోమేటిక్‌గా దాని సైజ్ తగ్గించి (Thumbnail క్రియేట్ చేసి) ఇంకో ఫోfolder లో పెట్టే సిస్టమ్. ఒక ఎడ్జ్ (S3) లో ఫైల్ పడగానే **S3 Event Notification** ట్రిగ్గర్ అయి ఒక **Lambda** ఫంక్షన్‌ని రన్ చేస్తుంది. ఆ ల్యాంబ్డా లోపల Node.js లో `sharp` లాంటి లైబ్రరీ వాడి ఇమేజ్ రీసైజ్ చేసి మళ్లీ S3 లో సేవ్ చేస్తుంది.

### 8. Chat Application using WebSockets

- **యాప్ రకం:** రియల్-టైమ్ కమ్యూనికేషన్ యాప్ (WhatsApp లాంటి చాట్).
- **ఎలా చేయాలి?:** React ఫ్రంటెండ్ మరియు Node.js బ్యాకెండ్‌ని కనెక్ట్ చేయడానికి **API Gateway (WebSocket API)** ని వాడుతాం. యూజర్ మెసేజ్ పంపినప్పుడు `$connect`, `$disconnect`, మరియు `sendMessage` రూట్స్ ద్వారా ల్యాంబ్డా యాక్టివేట్ అవుతుంది. యాక్టివ్‌గా ఉన్న యూజర్ల కనెక్షన్ ఐడీలను **DynamoDB** లో దాచి, ఒకరు మెసేజ్ పంపినప్పుడు అందరికీ రియల్-టైమ్‌లో పుష్ చేస్తాం.

### 9. Serverless Authentication using Cognito

- **యాప్ రకం:** యూజర్ సెక్యూరిటీ & లాగిన్ మేనేజ్‌మెంట్.
- **ఎలా చేయాలి?:** **Amazon Cognito User Pool** క్రియేట్ చేసి, అందులో Sign-up/Sign-in కాన్ఫిగర్ చేయాలి. మీ React యాప్‌లో AWS Amplify SDK వాడి లాగిన్ స్క్రీన్ బిల్డ్ చేయాలి. లాగిన్ అయ్యాక వచ్చే `JWT Token` ను React యాప్ ప్రతి API రిక్వెస్ట్‌తో పాటు పంపుతుంది. **API Gateway** దగ్గర **Cognito Authorizer** పెట్టి, ఆ టోకెన్ కరెక్ట్ అయితేనే లోపలికి అనుమతించేలా లాక్ చేయాలి.

### 10. Event-driven Order Processing System

- **యాప్ రకం:** అడ్వాన్స్‌డ్ మైక్రోసర్వీసెస్ ఆర్కిటెక్చర్.
- **ఎలా చేయాలి?:** యూజర్ ఒక ప్రొడక్ట్ ఆర్డర్ చేయగానే ఒక మెసేజ్ **Amazon SNS** కి వెళ్తుంది. SNS ఆ మెసేజ్‌ని ఫ్యాన్-అవుట్ (Fan-out) చేసి, సమాంతరంగా మూడు వేర్వేరు **SQS Queues** (ఒకటి పేమెంట్ కి, ఒకటి ఇన్వెంటరీ కి, ఒకటి షిప్పింగ్ కి) కి పంపుతుంది. ప్రతి క్యూ వెనుక ఒక **Lambda (Node.js)** ఫంక్షన్ ఉండి, ఆయా పనులను ఒకదానితో ఒకటి సంబంధం లేకుండా (Decoupled గా) ప్రాసెస్ చేస్తాయి. ఫెయిల్ అయిన వాటిని **Dead Letter Queue (DLQ)** కి పంపుతాం.

### 11. Monitoring Dashboard with CloudWatch

- **యాప్ రకం:** ప్రొడクション అప్లికేషన్ పర్యవేక్షణ.
- **ఎలా చేయాలి?:** మీ Node.js యాప్ లో వచ్చే ఎర్రర్ లాగ్స్‌ని **CloudWatch Logs** కి పంపాలి. మీ API లో `500 Internal Server Error` కౌంట్ 5 నిమిషాల్లో 10 కంటే ఎక్కువ వస్తే ట్రిగ్గర్ అయ్యేలా **CloudWatch Alarm** సెట్ చేయాలి. ఆ అలారమ్ రాగానే **SNS** ద్వారా మీ మొబైల్‌కి లేదా ఇమెయిల్‌కి "Production Server Error!" అని అలర్ట్ వచ్చేలా కాన్ఫిగర్ చేయాలి. వీటన్నింటినీ ఒకే స్క్రీన్ పై చూడటానికి **CloudWatch Dashboard** బిల్డ్ చేయాలి.
