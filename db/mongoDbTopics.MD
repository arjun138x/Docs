## **1. Introduction to NoSQL and MongoDB**

### **What is NoSQL?**

NoSQL (Not Only SQL) databases are designed for flexibility, scalability, and performance, unlike traditional relational databases (SQL).

#### **Key Features of NoSQL:**

- **Schema-less:** No predefined schema, allowing flexible document structures.
- **Scalability:** Horizontally scalable using sharding.
- **High Performance:** Optimized for read and write operations.
- **Support for Unstructured Data:** Handles JSON, XML, key-value pairs, graphs, etc.

### **Types of NoSQL Databases:**

- **Document-Based** (MongoDB, CouchDB)
- **Key-Value Stores** (Redis, DynamoDB)
- **Column-Family Stores** (Cassandra, HBase)
- **Graph Databases** (Neo4j, ArangoDB)

### **What is MongoDB?**

MongoDB is a **document-oriented** NoSQL database that stores data in **JSON-like** format called BSON.

#### **Why Use MongoDB?**

- **Flexible Schema:** No need to define tables or relationships.
- **Scalability:** Supports sharding (horizontal scaling).
- **High Availability:** Uses replication for failover handling.
- **Rich Query Language:** Supports filtering, aggregation, and full-text search.

## **1. Creating Documents** (`insertOne()`, `insertMany()`)

### **`insertOne()` - Insert a Single Document**

```javascript
db.users.insertOne({

  age: 28,
  email: "alice@example.com",
  skills: ["JavaScript", "MongoDB"],

});
```

‚úÖ **Result:**  
A single document is inserted into the `users` collection.

---

### **`insertMany()` - Insert Multiple Documents**

```javascript
db.users.insertMany([
  {
    name: "Bob",
    age: 32,
    email: "bob@example.com",
    skills: ["React", "Node.js"],
    isActive: false,
  },
  {
    name: "Charlie",
    age: 24,
    email: "charlie@example.com",
    skills: ["Python", "Django"],
    isActive: true,
  },
]);
```

‚úÖ **Result:**  
Multiple documents are added at once.

## **2. Reading Documents** (`find()`, `findOne()`, Query Operators)

### **`findOne()` - Retrieve a Single Document**

```javascript
db.users.findOne({ name: "Alice" });
```

‚úÖ **Result:**  
Returns the first document that matches the query.

---

### **`find()` - Retrieve Multiple Documents**

```javascript
db.users.find();
```

‚úÖ **Result:**  
Returns all documents in the collection.

```javascript
db.users.find({ isActive: true });
```

‚úÖ **Result:**  
Returns all users where `isActive` is `true`.

---

### **Query Operators**

#### **Comparison Operators**

```javascript
db.users.find({ age: { $gt: 25 } }); // Users older than 25
db.users.find({ age: { $gte: 30 } }); // Users 30 or older
db.users.find({ age: { $lt: 30 } }); // Users younger than 30
db.users.find({ age: { $ne: 28 } }); // Users NOT 28 years old
```

#### **Logical Operators**

```javascript
db.users.find({ $or: [{ age: { $lt: 25 } }, { isActive: true }] }); // Users younger than 25 OR active
db.users.find({ $and: [{ age: { $gte: 30 } }, { isActive: false }] }); // Users 30+ AND inactive
```

#### **Projection (Select Specific Fields)**

```javascript
db.users.find({}, { name: 1, email: 1, _id: 0 }); // Only show name and email, hide `_id`
```

#### **`$unset` - Remove a Field**

```javascript

```

‚úÖ **Result:**  
Removes the `skills` field from Alice‚Äôs document.

#### **`$inc` - Increment a Number**

```javascript
db.users.updateOne({ name: "Alice" }, { $inc: { age: 1 } });
```

‚úÖ **Result:**  
Increments Alice's age by `1`.

---

## **4. Deleting Documents** (`deleteOne()`, `deleteMany()`)

### **`deleteOne()` - Remove a Single Document**

```javascript

```

‚úÖ **Result:**

---

### **`deleteMany()` - Remove Multiple Documents**

### **Final Thoughts üí°**

# **3. Querying and Filtering Data**

- Comparison Operators (`$eq`, `$ne`, `$gt`, `$gte`, `$lt`, `$lte`)
- Logical Operators (`$and`, `$or`, `$not`, `$nor`)
- Array Operators (`$in`, `$nin`, `$all`, `$elemMatch`)
- Projection (`find({}, { field: 1 })`)
- Sorting and Limiting (`sort()`, `limit()`, `skip()`)

---

## **1. Comparison Operators (`$eq`, `$ne`, `$gt`, `$gte`, `$lt`, `$lte`)**

Used to filter documents based on specific conditions.

### **Example Dataset (users collection)**

```json
[
  { "name": "Alice", "age": 25, "city": "New York", "isActive": true },
  { "name": "Bob", "age": 30, "city": "Los Angeles", "isActive": false },
  { "name": "Charlie", "age": 28, "city": "Chicago", "isActive": true }
]
```

### **Using Comparison Operators**

```javascript
db.users.find({ age: { $eq: 25 } }); // Find users with age exactly 25
db.users.find({ age: { $ne: 25 } }); // Find users with age NOT 25
db.users.find({ age: { $gt: 25 } }); // Find users older than 25
db.users.find({ age: { $gte: 28 } }); // Find users 28 or older
db.users.find({ age: { $lt: 30 } }); // Find users younger than 30
db.users.find({ age: { $lte: 28 } }); // Find users 28 or younger
```

‚úÖ **Result Example:**

- `$gt: 25` ‚Üí Returns Bob (30) and Charlie (28).
- `$eq: 25` ‚Üí Returns Alice (25).

---

## **2. Logical Operators (`$and`, `$or`, `$not`, `$nor`)**

Used to combine multiple conditions.

### **Using Logical Operators**

```javascript
db.users.find({
  $and: [{ age: { $gt: 25 } }, { isActive: true }],
});
// Find users older than 25 AND active (Charlie)

db.users.find({
  $or: [{ city: "New York" }, { city: "Los Angeles" }],
});
// Find users in New York OR Los Angeles (Alice & Bob)

db.users.find({
  age: { $not: { $gt: 25 } },
});
// Find users NOT older than 25 (Alice)

db.users.find({
  $nor: [{ age: { $gte: 30 } }, { isActive: false }],
});
// Find users who are NOT 30+ AND NOT inactive (Only Alice & Charlie)
```

‚úÖ **Result Example:**

- `$and: [{ age: { $gt: 25 } }, { isActive: true }]` ‚Üí Returns Charlie.
- `$or: [{ city: "New York" }, { city: "Los Angeles" }]` ‚Üí Returns Alice & Bob.

---

## **3. Array Operators (`$in`, `$nin`, `$all`, `$elemMatch`)**

### **Example Dataset (products collection)**

```json
[
  { "name": "Laptop", "category": ["Electronics", "Computers"], "price": 1200 },
  { "name": "Headphones", "category": ["Electronics", "Audio"], "price": 100 },
  { "name": "Shoes", "category": ["Fashion", "Footwear"], "price": 50 }
]
```

### **Using Array Operators**

```javascript
db.products.find({ category: { $in: ["Electronics", "Fashion"] } });
// Find products in "Electronics" OR "Fashion" category (Laptop, Headphones, Shoes)

db.products.find({ category: { $nin: ["Electronics"] } });
// Find products NOT in "Electronics" category (Shoes)

db.products.find({ category: { $all: ["Electronics", "Computers"] } });
// Find products that contain BOTH "Electronics" AND "Computers" (Laptop)

db.products.find({
  category: { $elemMatch: { $eq: "Electronics" } },
});
// Find products where at least one element in the category array is "Electronics"
```

‚úÖ **Result Example:**

- `$in: ["Electronics", "Fashion"]` ‚Üí Returns Laptop, Headphones, Shoes.
- `$all: ["Electronics", "Computers"]` ‚Üí Returns Laptop.

---

## **4. Projection (`find({}, { field: 1 })`)**

Used to **select specific fields** to return in query results.

### **Example Query**

```javascript
db.users.find({}, { name: 1, age: 1, _id: 0 });
```

‚úÖ **Result:**

```json
[
  { "name": "Alice", "age": 25 },
  { "name": "Bob", "age": 30 },
  { "name": "Charlie", "age": 28 }
]
```

- **`_id: 0`** ‚Üí Excludes the `_id` field.
- **Default behavior:** `_id` is always included unless explicitly excluded.

---

## **5. Sorting and Limiting (`sort()`, `limit()`, `skip()`)**

Used to **sort**, **limit**, and **paginate** results.

### **Sorting (`sort()`)**

```javascript
db.users.find().sort({ age: 1 }); // Sort users by age in ascending order
db.users.find().sort({ age: -1 }); // Sort users by age in descending order
```

‚úÖ **Result Example:**

- `{ age: 1 }` ‚Üí Alice (25), Charlie (28), Bob (30).
- `{ age: -1 }` ‚Üí Bob (30), Charlie (28), Alice (25).

---

### **Limiting Results (`limit()`)**

```javascript
db.users.find().limit(2);
```

‚úÖ **Result:**  
Returns only the first **2** documents.

---

### **Skipping Documents (`skip()`)**

```javascript
db.users.find().skip(1).limit(2);
```

‚úÖ **Result:**

- Skips **first document**, returns **next 2**.
- Useful for **pagination** (`page = (skip + limit)`).

---

### **Pagination Example** (Skip & Limit Combined)

```javascript
const page = 2;
const pageSize = 2;
db.users
  .find()
  .skip((page - 1) * pageSize)
  .limit(pageSize);
```

‚úÖ **Effect:**

- `page = 1` ‚Üí Skips 0, returns first **2** documents.
- `page = 2` ‚Üí Skips **2**, returns next **2** documents.

# **4. Indexing in MongoDB**

- Importance of Indexing
- Creating Indexes (`createIndex()`)
- Types of Indexes (Single Field, Compound, Multikey, Text, Geospatial)
- Index Performance (`explain()`, `hint()`)

---

Indexes in MongoDB **improve query performance** by reducing the number of documents that need to be scanned. Without indexes, MongoDB performs a **collection scan**, which can be slow for large datasets.

---

## **1. Importance of Indexing**

üîπ **Faster Queries:** Indexes allow MongoDB to retrieve data efficiently.  
üîπ **Reduces Workload:** Instead of scanning the entire collection, MongoDB uses indexes to locate data quickly.  
üîπ **Improves Sorting Performance:** Sorting without an index requires MongoDB to load all documents into memory.  
üîπ **Supports Unique Constraints:** Indexes can enforce uniqueness in fields (e.g., unique usernames or emails).  
üîπ **Optimizes Geospatial and Full-Text Search:** Special indexes allow for text search and location-based queries.

üìå **Example Without Indexing**

```javascript
db.users.find({ age: 30 });
```

- Without an index, MongoDB will **scan every document** in the collection.
- This is **slow** when the collection contains millions of documents.

üìå **Example With Indexing**

```javascript
db.users.createIndex({ age: 1 });
```

- Now, MongoDB uses the index to **quickly find documents with age = 30**.

---

## **2. Creating Indexes (`createIndex()`)**

### **Basic Index on a Single Field**

```javascript
db.users.createIndex({ age: 1 }); // Ascending order
db.users.createIndex({ age: -1 }); // Descending order
```

‚úÖ **Effect:**

- Speeds up queries like `db.users.find({ age: 30 })`.
- The order (`1` or `-1`) affects sorting but not filtering.

---

### **3. Types of Indexes**

### **1Ô∏è‚É£ Single Field Index**

- Indexes a single field.

```javascript
db.users.createIndex({ email: 1 });
```

‚úÖ Optimizes queries like `db.users.find({ email: "user@example.com" })`.

---

### **2Ô∏è‚É£ Compound Index (Multiple Fields)**

- Indexes **multiple fields** in a specific order.

```javascript
db.users.createIndex({ age: 1, city: 1 });
```

‚úÖ Optimizes queries like:

```javascript
db.users.find({ age: 30, city: "New York" });
```

‚ö†Ô∏è **Order matters** in compound indexes.

- `{ age: 1, city: 1 }` works for `(age, city)`, but not for `(city, age)`.

---

### **3Ô∏è‚É£ Multikey Index (For Arrays)**

- Indexes **elements inside an array**.

```javascript
db.products.createIndex({ tags: 1 });
```

‚úÖ Optimizes queries like:

```javascript
db.products.find({ tags: "electronics" });
```

‚ö†Ô∏è **Only one multikey index per query** is allowed.

---

### **4Ô∏è‚É£ Text Index (For Full-Text Search)**

- Enables **search within text fields**.

```javascript
db.articles.createIndex({ content: "text" });
```

‚úÖ Optimizes queries like:

```javascript
db.articles.find({ $text: { $search: "mongodb" } });
```

‚ö†Ô∏è **Only one text index per collection**.

---

### **5Ô∏è‚É£ Geospatial Index (For Location-Based Queries)**

- Required for **location-based searches**.

```javascript
db.places.createIndex({ location: "2dsphere" });
```

‚úÖ Optimizes queries like:

```javascript
db.places.find({
  location: {
    $near: {
      $geometry: { type: "Point", coordinates: [-73.856077, 40.848447] },
      $maxDistance: 5000,
    },
  },
});
```

‚ö†Ô∏è **Use `2dsphere` for real-world locations, `2d` for flat coordinates**.

---

## **4. Index Performance (`explain()`, `hint()`)**

### **Checking Index Usage with `explain()`**

```javascript
db.users.find({ age: 30 }).explain("executionStats");
```

‚úÖ Shows:

- **`COLLSCAN`** (Collection Scan) if no index is used. ‚ùå
- **`IXSCAN`** (Index Scan) if an index is used. ‚úÖ

---

### **Forcing Index Usage with `hint()`**

```javascript
db.users.find({ age: 30 }).hint({ age: 1 });
```

‚úÖ Forces MongoDB to use the `age` index.

- Useful when multiple indexes exist, and you want to test their impact.

# **5. Aggregation Framework**

- Aggregation Pipeline (`$match`, `$group`, `$project`, `$sort`, `$limit`)
- `$lookup` for Joins
- `$unwind` for Arrays
- `$bucket` and `$facet` for Data Grouping
- Performance Optimization in Aggregation

---

The **Aggregation Pipeline** is a **powerful framework** in MongoDB that allows for **data transformation, filtering, grouping, and analysis** in a sequence of stages.

---

## **1Ô∏è‚É£ Basics of the Aggregation Pipeline**

üîπ **Works in Stages** ‚Üí Data flows through multiple **stages**, each performing a specific operation.  
üîπ **Faster than Map-Reduce** ‚Üí Aggregation is **optimized for performance** compared to traditional queries.  
üîπ **Common Use Cases** ‚Üí Reporting, data analytics, data transformation, and complex queries.

### **Basic Syntax**

```javascript
db.collection.aggregate([{ stage1 }, { stage2 }, { stage3 }]);
```

Each stage modifies the data and passes it to the next stage.

---

## **2Ô∏è‚É£ Aggregation Pipeline Stages**

### **1Ô∏è‚É£ `$match` ‚Üí Filter Documents (Like `find()`)**

```javascript
db.orders.aggregate([{ $match: { status: "shipped" } }]);
```

‚úÖ **Filters only shipped orders.**

- `$match` **should be used at the start** for best performance.

---

### **2Ô∏è‚É£ `$group` ‚Üí Group Data**

```javascript
db.orders.aggregate([
  {
    $group: {
      _id: "$customerId",
      totalSpent: { $sum: "$amount" },
    },
  },
]);
```

‚úÖ **Groups orders by `customerId` and calculates total spent.**

- `_id` is the field to group by.
- `$sum` computes the total amount spent by each customer.

---

### **3Ô∏è‚É£ `$project` ‚Üí Restructure Data (Select Fields)**

```javascript
db.users.aggregate([
  {
    $project: {
      fullName: { $concat: ["$firstName", " ", "$lastName"] },
      email: 1,
      _id: 0,
    },
  },
]);
```

‚úÖ **Concatenates `firstName` and `lastName` into `fullName`**

- `_id: 0` ‚Üí Excludes `_id` field.
- `$concat` joins strings.

---

### **4Ô∏è‚É£ `$sort` ‚Üí Sort Data**

```javascript
db.orders.aggregate([
  { $sort: { amount: -1 } }, // Sort by amount (highest first)
]);
```

‚úÖ **Sorts orders by amount in descending order.**

- `1` for **ascending**, `-1` for **descending**.

---

### **5Ô∏è‚É£ `$limit` ‚Üí Limit Results**

```javascript
db.orders.aggregate([{ $sort: { amount: -1 } }, { $limit: 5 }]);
```

‚úÖ **Returns the top 5 highest orders.**

- Works well with `$sort` for **leaderboards, top lists, etc.**

---

## **3Ô∏è‚É£ `$lookup` ‚Üí Join Two Collections (SQL `JOIN`)**

MongoDB **does not have direct joins**, but `$lookup` can join two collections.

### **Example: Joining Users and Orders**

```javascript
db.users.aggregate([
  {
    $lookup: {
      from: "orders",
      localField: "userId",
      foreignField: "customerId",
      as: "orders",
    },
  },
]);
```

‚úÖ **Joins `users` with `orders` where `userId` matches `customerId`.**

- **`from`** ‚Üí Target collection (`orders`).
- **`localField`** ‚Üí Field in the first collection (`userId`).
- **`foreignField`** ‚Üí Matching field in the second collection (`customerId`).
- **`as`** ‚Üí Name of the resulting array (`orders`).

üìå **Example Output:**

```json
{
  "name": "Alice",
  "userId": 1,
  "orders": [
    { "orderId": 101, "amount": 50 },
    { "orderId": 102, "amount": 75 }
  ]
}
```

---

## **4Ô∏è‚É£ `$unwind` ‚Üí Flatten Arrays**

Used to **break an array into multiple documents**.

### **Example: Flattening an Orders Array**

```javascript
db.users.aggregate([{ $unwind: "$orders" }]);
```

‚úÖ **Converts each order in the `orders` array into a separate document.**

üìå **Before:**

```json
{
  "name": "Alice",
  "orders": [
    { "orderId": 101, "amount": 50 },
    { "orderId": 102, "amount": 75 }
  ]
}
```

üìå **After `$unwind`:**

```json
{ "name": "Alice", "orderId": 101, "amount": 50 }
{ "name": "Alice", "orderId": 102, "amount": 75 }
```

---

## **5Ô∏è‚É£ `$bucket` ‚Üí Group Data into Ranges**

Divides values into **custom-defined buckets**.

### **Example: Grouping Orders by Price Range**

```javascript
db.orders.aggregate([
  {
    $bucket: {
      groupBy: "$amount",
      boundaries: [0, 100, 500, 1000, 5000],
      default: "Other",
      output: { count: { $sum: 1 } },
    },
  },
]);
```

‚úÖ **Groups orders into price ranges:**

- **`0 - 100`**
- **`100 - 500`**
- **`500 - 1000`**
- **`1000 - 5000`**
- `"Other"` for amounts beyond 5000.

---

## **6Ô∏è‚É£ `$facet` ‚Üí Multiple Aggregations in One Query**

Allows running **multiple aggregations** at once.

### **Example: Two Different Analyses in One Query**

```javascript
db.orders.aggregate([
  {
    $facet: {
      topOrders: [{ $sort: { amount: -1 } }, { $limit: 5 }],
      totalSpentByCustomer: [
        { $group: { _id: "$customerId", total: { $sum: "$amount" } } },
      ],
    },
  },
]);
```

‚úÖ **Results:**

- `topOrders` ‚Üí Top 5 highest orders.
- `totalSpentByCustomer` ‚Üí Total spent per customer.

---

## **7Ô∏è‚É£ Aggregation Performance Optimization üöÄ**

1Ô∏è‚É£ **Use `$match` First**

- `$match` **filters data before processing** to reduce workload.

```javascript
db.orders.aggregate([
  { $match: { status: "shipped" } }, // ‚úÖ Place early for efficiency
  { $group: { _id: "$customerId", totalSpent: { $sum: "$amount" } } },
]);
```

2Ô∏è‚É£ **Use Indexes**

- If filtering by `status`, create an index:

```javascript
db.orders.createIndex({ status: 1 });
```

- This **makes `$match` faster**.

3Ô∏è‚É£ **Avoid `$unwind` if Possible**

- `$unwind` can generate **many** documents.
- Instead, **use `$lookup` and `$group` when possible**.

4Ô∏è‚É£ **Use `$limit` and `$sort` Wisely**

- `$sort` on **large collections** can be slow.
- Ensure the **sorting field is indexed**.

5Ô∏è‚É£ **Check Performance with `explain()`**

```javascript
db.orders.aggregate([...]).explain("executionStats")
```

- Look for **COLLSCAN (bad)** vs. **IXSCAN (good)**.

# **6. Data Modeling in MongoDB**

- Embedded vs. Referenced Documents
- One-to-One, One-to-Many, and Many-to-Many Relationships
- Schema Design Best Practices

---

MongoDB is **schema-flexible**, but designing a good schema is **crucial** for performance and scalability.

This guide covers:  
‚úÖ **Embedded vs. Referenced Documents**  
‚úÖ **One-to-One, One-to-Many, and Many-to-Many Relationships**  
‚úÖ **Schema Design Best Practices**

---

## **1Ô∏è‚É£ Embedded vs. Referenced Documents**

MongoDB supports two ways to **relate data**:

1. **Embedded (Denormalized)** ‚Üí Store related data **inside** the document.
2. **Referenced (Normalized)** ‚Üí Store related data in **separate collections** and reference them using IDs.

---

### **üìç Embedded Documents (Denormalized)**

- **Faster reads** since all data is stored in a single document.
- **Best for closely related data** (e.g., user profile & address).

üìå **Example: Embedding Address inside User Document**

```json
{
  "_id": 1,
  "name": "Alice",
  "email": "alice@example.com",
  "address": {
    "street": "123 Main St",
    "city": "New York",
    "zip": "10001"
  }
}
```

‚úÖ **Pros:**

- Faster reads (no joins needed).
- Simpler queries.

‚ùå **Cons:**

- Increases document size.
- Harder to update specific fields (e.g., address changes often).

---

### **üìç Referenced Documents (Normalized)**

- **Best for frequently updated or shared data** (e.g., orders & customers).
- **Uses `$lookup`** to join collections when needed.

üìå **Example: Storing Addresses in a Separate Collection**  
**Users Collection:**

```json
{
  "_id": 1,
  "name": "Alice",
  "email": "alice@example.com",
  "addressId": 101
}
```

**Addresses Collection:**

```json
{
  "_id": 101,
  "street": "123 Main St",
  "city": "New York",
  "zip": "10001"
}
```

‚úÖ **Pros:**

- Reduces duplication (addresses can be reused).
- Smaller document size.
- Easier updates (change an address without affecting user data).

‚ùå **Cons:**

- Slower reads (requires `$lookup`).
- More complex queries.

---

## **2Ô∏è‚É£ One-to-One, One-to-Many, and Many-to-Many Relationships**

### **üìç One-to-One Relationship**

‚úÖ **Example: A user has one profile**  
**üìå Best Approach:** **Embed profile data inside the user document.**

```json
{
  "_id": 1,
  "name": "Alice",
  "profile": {
    "bio": "Software Engineer",
    "avatar": "alice.jpg"
  }
}
```

‚úÖ **Best for:** Rarely changing data, fast access.  
‚ùå **Avoid** embedding if the sub-document grows too large.

---

### **üìç One-to-Many Relationship**

‚úÖ **Example: A blog has many comments**  
**üìå Option 1: Embed (if few comments per post)**

```json
{
  "_id": 1,
  "title": "MongoDB Guide",
  "comments": [
    { "user": "Alice", "text": "Great article!" },
    { "user": "Bob", "text": "Very helpful!" }
  ]
}
```

‚úÖ **Fast reads**  
‚ùå Not scalable if comments grow indefinitely.

**üìå Option 2: Reference (if many comments per post)**  
**Posts Collection:**

```json
{
  "_id": 1,
  "title": "MongoDB Guide",
  "comments": [101, 102]
}
```

**Comments Collection:**

```json
{ "_id": 101, "postId": 1, "user": "Alice", "text": "Great article!" }
{ "_id": 102, "postId": 1, "user": "Bob", "text": "Very helpful!" }
```

‚úÖ **Better scalability for large comments**  
‚ùå Requires `$lookup` to fetch comments.

---

### **üìç Many-to-Many Relationship**

‚úÖ **Example: Students & Courses (Many students enroll in many courses)**  
**üìå Best Approach: Use Reference IDs in an Array**

**Students Collection:**

```json
{
  "_id": 1,
  "name": "Alice",
  "courses": [201, 202]
}
```

**Courses Collection:**

```json
{
  "_id": 201,
  "title": "MongoDB 101",
  "students": [1, 2, 3]
}
```

‚úÖ Flexible, scalable  
‚ùå Requires `$lookup` for joining collections.

---

## **3Ô∏è‚É£ Schema Design Best Practices üöÄ**

### **‚úÖ Choose Embedded vs. Referenced Based on Data Usage**

- **Use Embedded Documents** when **data is always accessed together** (e.g., user profile).
- **Use Referenced Documents** when **data is large, shared, or updated frequently** (e.g., orders, customers).

---

### **‚úÖ Optimize for Read or Write Performance**

üìå **If Read-heavy ‚Üí Embed Data**  
üìå **If Write-heavy ‚Üí Use References**  
üìå **For Balanced Workloads ‚Üí Hybrid Approach**

---

### **‚úÖ Keep Documents Small (<16MB Limit)**

- MongoDB has a **16MB document size limit**.
- Keep embedded arrays **small** (<100 elements).
- If an array grows too large, use **pagination** or reference another collection.

üìå **Example: Splitting Large Orders into Multiple Documents**  
Instead of storing **1000+ order items** inside an order, break them into smaller documents.

---

### **‚úÖ Use Indexes for Fast Queries**

Indexes speed up searches, sorts, and lookups.

üìå **Example: Creating an Index on Email**

```javascript
db.users.createIndex({ email: 1 });
```

‚úÖ **Queries like `db.users.find({ email: "alice@example.com" })` run faster.**

---

### **‚úÖ Avoid Deep Nesting (Keep Documents Flat)**

MongoDB supports **nested documents**, but deep nesting **hurts performance**.

‚ùå **Bad Example (Deep Nesting)**

```json
{
  "user": {
    "profile": {
      "social": {
        "twitter": "@alice"
      }
    }
  }
}
```

‚úÖ **Better Approach (Flat Structure)**

```json
{
  "userId": 1,
  "profile": { "twitter": "@alice" }
}
```

üìå **Reason:** Queries become faster when **documents remain shallow**.

---

### **‚úÖ Use `$lookup` Only When Necessary**

- `$lookup` (joins) can **slow down queries** if used too much.
- If required, **index the foreign keys** to speed up joins.

üìå **Example: Indexing Foreign Key in Orders Collection**

```javascript
db.orders.createIndex({ customerId: 1 });
```

‚úÖ Now, `db.orders.aggregate([ { $lookup: { ... } } ])` runs faster.

---

## **üöÄ Final Takeaways**

| **Decision**        | **Best Choice**                           |
| ------------------- | ----------------------------------------- |
| User Profile        | **Embedded**                              |
| Orders & Customers  | **Referenced**                            |
| Blog & Comments     | **Hybrid (Embed small, Reference large)** |
| Tags & Categories   | **Referenced (Many-to-Many)**             |
| Data Changes Often  | **Referenced**                            |
| Read-Heavy Queries  | **Embedded**                              |
| Write-Heavy Updates | **Referenced**                            |

---

# **7. Transactions and ACID Compliance**

- Session-based Transactions (`startSession()`, `commitTransaction()`, `abortTransaction()`)
- Multi-Document Transactions

---

MongoDB supports **transactions** to ensure atomicity and consistency across multiple documents. This is useful when updating **multiple collections** or **multiple documents** within a collection.

This guide covers:  
‚úÖ **Session-based Transactions** (`startSession()`, `commitTransaction()`, `abortTransaction()`)  
‚úÖ **Multi-Document Transactions**

---

## **1Ô∏è‚É£ What Are Transactions?**

- Transactions **group multiple operations** into a single unit.
- If any operation fails, **all changes are rolled back** (ACID compliance).
- Transactions **work only in replica sets and sharded clusters** (not standalone servers).

---

## **2Ô∏è‚É£ Session-Based Transactions**

MongoDB transactions require a **session** to execute multiple operations together.

### **üìå Steps to Use Transactions**

1Ô∏è‚É£ **Start a session**  
2Ô∏è‚É£ **Start a transaction**  
3Ô∏è‚É£ **Execute multiple operations**  
4Ô∏è‚É£ **Commit the transaction** (if successful) or **abort** (if an error occurs)

---

### **‚úÖ Example: Transferring Money Between Accounts**

üí° Scenario: **Move $100 from Alice to Bob**

```javascript
const session = db.getMongo().startSession(); // Start a session

session.startTransaction(); // Begin transaction

try {
  // Deduct $100 from Alice
  db.accounts.updateOne(
    { name: "Alice" },
    { $inc: { balance: -100 } },
    { session }
  );

  // Add $100 to Bob
  db.accounts.updateOne(
    { name: "Bob" },
    { $inc: { balance: 100 } },
    { session }
  );

  // Commit the transaction
  session.commitTransaction();
  print("‚úÖ Transaction committed successfully!");
} catch (error) {
  // If any error occurs, rollback changes
  session.abortTransaction();
  print("‚ùå Transaction aborted due to error:", error);
} finally {
  session.endSession(); // End the session
}
```

---

### **üìå Key Functions**

| Function              | Description                           |
| --------------------- | ------------------------------------- |
| `startSession()`      | Starts a new session                  |
| `startTransaction()`  | Begins a new transaction              |
| `commitTransaction()` | Commits all operations if successful  |
| `abortTransaction()`  | Rolls back all operations if any fail |
| `endSession()`        | Closes the session                    |

---

## **3Ô∏è‚É£ Multi-Document Transactions**

MongoDB normally applies **operations per document**, but transactions let us **update multiple documents across collections atomically**.

### **‚úÖ Example: Booking a Flight**

üí° **Scenario:** Book a flight by updating **two collections**:

1. **Reduce available seats in flights collection**
2. **Create a booking record in the reservations collection**

```javascript
const session = db.getMongo().startSession();
session.startTransaction();

try {
  // Step 1: Reduce available seats for the flight
  db.flights.updateOne(
    { flightNumber: "AI123", availableSeats: { $gte: 1 } },
    { $inc: { availableSeats: -1 } },
    { session }
  );

  // Step 2: Add a booking entry
  db.reservations.insertOne(
    {
      userId: 101,
      flightNumber: "AI123",
      status: "Confirmed",
    },
    { session }
  );

  // Commit transaction
  session.commitTransaction();
  print("‚úÖ Flight booked successfully!");
} catch (error) {
  // Rollback changes if something goes wrong
  session.abortTransaction();
  print("‚ùå Booking failed. Rolling back transaction.", error);
} finally {
  session.endSession();
}
```

---

## **4Ô∏è‚É£ Best Practices for Transactions üöÄ**

üîπ **Use transactions only when necessary** ‚Äì Simple operations don‚Äôt need transactions.  
üîπ **Ensure indexes are used** to speed up queries inside transactions.  
üîπ **Keep transactions short** to avoid performance issues.  
üîπ **Use transactions in replica sets/sharded clusters** (they won‚Äôt work on standalone MongoDB).

---

# **8. Replication and High Availability**

- What is Replication?
- Replica Sets (Primary, Secondary, Arbiter)
- Failover and Automatic Elections

---

**Replication** in MongoDB ensures **high availability, fault tolerance, and data redundancy** by copying data across multiple servers.

This guide covers:  
‚úÖ **What is Replication?**  
‚úÖ **Replica Sets (Primary, Secondary, Arbiter)**  
‚úÖ **Failover and Automatic Elections**

---

## **1Ô∏è‚É£ What is Replication?**

Replication **keeps multiple copies** of your data on different servers.

- Ensures **data availability** even if one server goes down.
- Helps with **disaster recovery** and **load balancing** for reads.

üí° **MongoDB achieves replication using Replica Sets.**

---

## **2Ô∏è‚É£ Replica Sets (Primary, Secondary, Arbiter)**

A **Replica Set** is a group of **MongoDB servers** (nodes) that store the same data.

### **üìç Components of a Replica Set**

| Role          | Description                                              |
| ------------- | -------------------------------------------------------- |
| **Primary**   | The main server that handles all write operations.       |
| **Secondary** | Copies data from the Primary and can serve read queries. |
| **Arbiter**   | Helps with elections but **doesn‚Äôt store data**.         |

üìå **A typical replica set consists of at least 3 nodes:**

- **1 Primary**
- **1 or more Secondary nodes**
- **(Optional) 1 Arbiter**

üìå **Example Setup:**

```
+-----------+      +------------+      +------------+
|  Primary  | ---> | Secondary 1 | --->| Secondary 2 |
+-----------+      +------------+      +------------+
       |
       v
  (Arbiter)  üó≥Ô∏è  [Optional]
```

---

### **‚úÖ How to Create a Replica Set**

1Ô∏è‚É£ Start MongoDB instances with replication enabled:

```bash
mongod --replSet "rs0" --port 27017 --dbpath /data/db1
mongod --replSet "rs0" --port 27018 --dbpath /data/db2
mongod --replSet "rs0" --port 27019 --dbpath /data/db3
```

2Ô∏è‚É£ Connect to one instance and initiate the replica set:

```javascript
rs.initiate({
  _id: "rs0",
  members: [
    { _id: 0, host: "localhost:27017" },
    { _id: 1, host: "localhost:27018" },
    { _id: 2, host: "localhost:27019" },
  ],
});
```

3Ô∏è‚É£ Check the replica set status:

```javascript
rs.status();
```

---

## **3Ô∏è‚É£ Failover and Automatic Elections üó≥Ô∏è**

When the **Primary node fails**, MongoDB **automatically elects a new Primary** from the Secondary nodes.

### **üìå How Automatic Elections Work:**

1Ô∏è‚É£ The Primary **goes down**.  
2Ô∏è‚É£ Secondary nodes **vote** for a new Primary.  
3Ô∏è‚É£ The node with the **highest priority and latest data** becomes the new Primary.  
4Ô∏è‚É£ A new Primary **takes over write operations**.

‚è≥ **Election time:** Usually within **10-12 seconds**.

---

### **‚úÖ How to Force a Re-Election**

üìå If the **Primary is slow or unresponsive**, you can force an election:

```javascript
rs.stepDown();
```

This makes the **current Primary step down**, and a new Primary is elected.

---

## **üöÄ Best Practices for Replication**

üîπ **Always use an odd number of nodes** (to prevent voting ties).  
üîπ **Place replicas in different data centers** for disaster recovery.  
üîπ **Monitor replication lag** to ensure secondaries are up-to-date.  
üîπ **Use read preferences** to optimize performance:

- `primary` ‚Üí Read from Primary (default).
- `secondary` ‚Üí Read from Secondary.
- `nearest` ‚Üí Read from the closest node.

# **9. Sharding (Horizontal Scaling)**

- What is Sharding?
- Shard Key Selection
- Configuring a Sharded Cluster

---

Sharding in MongoDB helps distribute large datasets **across multiple servers**, improving **performance, scalability, and fault tolerance**.

This guide covers:  
‚úÖ **What is Sharding?**  
‚úÖ **Shard Key Selection**  
‚úÖ **Configuring a Sharded Cluster**

---

## **1Ô∏è‚É£ What is Sharding?**

**Sharding** is a **horizontal scaling** technique where data is **split across multiple servers (shards)**.

üí° **Why Use Sharding?**  
‚úîÔ∏è **Handles large datasets** (scales beyond a single server‚Äôs storage).  
‚úîÔ∏è **Improves read/write performance** (queries run on multiple shards).  
‚úîÔ∏è **Ensures high availability** (data is replicated across shards).

üìå **Example: Without vs. With Sharding**

üö´ **Without Sharding (Single Server, Bottleneck)**

```
+--------------------+
| Large Database    |
| (Performance Issues) |
+--------------------+
```

‚úÖ **With Sharding (Multiple Shards, Faster Queries)**

```
+------------+   +------------+   +------------+
| Shard 1   |   | Shard 2   |   | Shard 3   |
| (Data A)  |   | (Data B)  |   | (Data C)  |
+------------+   +------------+   +------------+
```

---

## **2Ô∏è‚É£ Shard Key Selection**

A **shard key** is the field used to distribute data across shards.

### **‚úÖ Choosing the Right Shard Key**

| Shard Key Type                   | Pros                   | Cons                           |
| -------------------------------- | ---------------------- | ------------------------------ |
| **Hashed** (`hashed` index)      | Even data distribution | Slower range queries           |
| **Ranged** (`ascending` index)   | Fast range queries     | Uneven distribution (hotspots) |
| **Zone-Based** (`tagged` shards) | Custom distribution    | Complex setup                  |

üí° **Best Practices:**  
‚úîÔ∏è **Choose a field with high cardinality** (many unique values).  
‚úîÔ∏è **Avoid monotonically increasing keys** (e.g., timestamps).  
‚úîÔ∏è **Use compound keys** (`{ region: 1, userId: 1 }`) for better distribution.

---

## **3Ô∏è‚É£ Configuring a Sharded Cluster**

### **üìå Components of a Sharded Cluster**

| Component          | Role                                      |
| ------------------ | ----------------------------------------- |
| **Shards**         | Stores data (each shard is a replica set) |
| **Config Servers** | Stores metadata about shards              |
| **Mongos Router**  | Routes queries to the correct shard       |

üìå **Sharded Cluster Architecture**

```
+-------------------+
| Mongos (Router)  |  <--- Clients connect here
+-------------------+
      |
      v
+-------------------+      +-------------------+
| Config Server 1  | --->  | Config Server 2  |
+-------------------+      +-------------------+
      |
      v
+------------+  +------------+  +------------+
|  Shard 1   |  |  Shard 2   |  |  Shard 3   |
+------------+  +------------+  +------------+
```

---

### **‚úÖ Steps to Set Up a Sharded Cluster**

1Ô∏è‚É£ **Start Config Servers**

```bash
mongod --configsvr --replSet configReplSet --port 27019 --dbpath /data/configdb
```

2Ô∏è‚É£ **Initiate Config Replica Set**

```javascript
rs.initiate({
  _id: "configReplSet",
  configsvr: true,
  members: [{ _id: 0, host: "localhost:27019" }],
});
```

3Ô∏è‚É£ **Start Shard Servers**

```bash
mongod --shardsvr --replSet shardReplSet1 --port 27018 --dbpath /data/shard1
```

4Ô∏è‚É£ **Start Mongos Router**

```bash
mongos --configdb configReplSet/localhost:27019 --port 27017
```

5Ô∏è‚É£ **Connect and Enable Sharding**

```javascript
sh.enableSharding("myDatabase");
sh.shardCollection("myDatabase.users", { userId: "hashed" });
```

---

## **üöÄ Best Practices for Sharding**

‚úîÔ∏è **Choose the right shard key** to avoid hotspots.  
‚úîÔ∏è **Use hashed sharding for even distribution**.  
‚úîÔ∏è **Monitor shard balancing** (`balancerStatus()`).  
‚úîÔ∏è **Distribute shards across multiple data centers** for fault tolerance.

# **10. Security in MongoDB**

- Authentication & Authorization (Role-Based Access Control)
- Encryption (TLS/SSL, Field-Level Encryption)
- Auditing and Logging

---

Security is **critical** in MongoDB to protect data from **unauthorized access, breaches, and tampering**.

This guide covers:  
‚úÖ **Authentication & Authorization (Role-Based Access Control)**  
‚úÖ **Encryption (TLS/SSL, Field-Level Encryption)**  
‚úÖ **Auditing and Logging**

---

## **1Ô∏è‚É£ Authentication & Authorization (Role-Based Access Control)**

MongoDB **authentication** verifies **who** is accessing the database, while **authorization** controls **what actions** they can perform.

### **üìå Authentication Methods**

| Authentication Method                                          | Description                                                           |
| -------------------------------------------------------------- | --------------------------------------------------------------------- |
| **SCRAM (Salted Challenge Response Authentication Mechanism)** | Default authentication method (secure password-based authentication). |
| **X.509 Certificates**                                         | Uses SSL/TLS certificates for authentication.                         |
| **LDAP (Lightweight Directory Access Protocol)**               | Integrates with corporate directories (e.g., Active Directory).       |
| **Kerberos Authentication**                                    | Uses a central authentication server for single sign-on (SSO).        |

#### **‚úÖ Enabling Authentication**

1Ô∏è‚É£ **Start MongoDB with authentication enabled**

```bash
mongod --auth --dbpath /data/db
```

2Ô∏è‚É£ **Create an admin user**

```javascript
use admin;
db.createUser({
  user: "adminUser",
  pwd: "securePassword",
  roles: [{ role: "root", db: "admin" }]
});
```

3Ô∏è‚É£ **Connect with authentication**

```bash
mongo -u "adminUser" -p "securePassword" --authenticationDatabase "admin"
```

---

### **üìç Role-Based Access Control (RBAC)**

RBAC **restricts users** to specific **permissions** based on assigned roles.

| Built-in Role | Description                                     |
| ------------- | ----------------------------------------------- |
| `read`        | Can only read data.                             |
| `readWrite`   | Can read and write data.                        |
| `dbAdmin`     | Can manage indexes, collections, and profiling. |
| `userAdmin`   | Can create and modify user roles.               |
| `root`        | Full access (admin only).                       |

#### **‚úÖ Example: Creating a Read-Only User**

```javascript
use myDatabase;
db.createUser({
  user: "readonlyUser",
  pwd: "password123",
  roles: [{ role: "read", db: "myDatabase" }]
});
```

## **2Ô∏è‚É£ Encryption (TLS/SSL, Field-Level Encryption)**

Encryption ensures that **data is protected** both **in transit** (between clients and servers) and **at rest** (stored data).

### **üìå Transport Encryption (TLS/SSL)**

üîí **Encrypts communication** between the client and the MongoDB server using SSL/TLS.

#### **‚úÖ Enabling TLS/SSL Encryption**


3Ô∏è‚É£ **Connect using TLS**

```bash

```

---

### **üìå Field-Level Encryption (FLE)**

üîí **Encrypts specific fields** instead of encrypting the whole database.

**Why Use FLE?**  
‚úîÔ∏è Protects sensitive fields (e.g., credit cards, passwords).  
‚úîÔ∏è Encryption/decryption happens **on the client-side**, so **MongoDB never sees the plaintext data**.

#### **‚úÖ Example: Encrypting a Field in a Document**

```javascript

  creditCard: "ENCRYPTED_STRING",
```

# **11. MongoDB Performance Tuning**

Optimizing MongoDB performance requires **understanding query execution, managing connections efficiently, and fine-tuning read/write strategies**.

This guide covers:  
‚úÖ **Profiling Queries (`db.setProfilingLevel()`)**  
‚úÖ **Connection Pooling**  
‚úÖ **Write Concerns and Read Preferences**

---

## **1Ô∏è‚É£ Profiling Queries (`db.setProfilingLevel()`)**

**Query Profiling** helps **identify slow queries** and optimize performance. MongoDB‚Äôs profiler logs slow queries in the `system.profile` collection.

### **üìå Profiling Levels**

| Level | Description                                            |
| ----- | ------------------------------------------------------ |
| `0`   | **Off** ‚Äì No profiling (default).                      |
| `1`   | **Only slow queries** (queries exceeding a threshold). |
| `2`   | **All queries** are logged (use with caution).         |

### **‚úÖ Enable Profiling for Slow Queries**

```javascript
db.setProfilingLevel(1, 100); // Logs queries slower than 100ms
```

### **‚úÖ View Slow Queries**

```javascript
db.system.profile.find().sort({ ts: -1 }).limit(5);
```

### **‚úÖ Enable Full Query Logging (Debugging)**

```javascript
db.setProfilingLevel(2);
```

üìå **Tip:** Keep profiling level `1` in production to avoid performance overhead.

---

## **2Ô∏è‚É£ Connection Pooling**

MongoDB handles thousands of client connections using **connection pooling**.

üìå **Connection pooling** reduces the overhead of creating new connections by **reusing existing ones**.

### **üìå How Connection Pooling Works?**

‚úîÔ∏è **Clients request a connection.**  
‚úîÔ∏è **If a connection exists, it is reused.**  
‚úîÔ∏è **If no available connection exists, a new one is created (up to a limit).**  
‚úîÔ∏è **Idle connections are closed after a timeout.**

### **‚úÖ Configuring Connection Pooling in Node.js (Mongoose)**

```javascript
const mongoose = require("mongoose");

mongoose.connect("mongodb://localhost:27017/mydb", {
  useNewUrlParser: true,
  useUnifiedTopology: true,
  maxPoolSize: 50, // Maximum connections in the pool
  minPoolSize: 10, // Minimum idle connections
});
```

üìå **Best Practices:**  
‚úîÔ∏è Set `maxPoolSize` based on your application‚Äôs needs.  
‚úîÔ∏è Use **connection pooling in production** for scalability.  
‚úîÔ∏è **Monitor connection usage** with `db.serverStatus().connections`.

---

## **3Ô∏è‚É£ Write Concerns and Read Preferences**

**Write Concerns** and **Read Preferences** ensure **data consistency, durability, and performance** based on application needs.

---

### **üìå Write Concerns (Ensuring Data Durability)**

Write concerns **define how MongoDB acknowledges write operations**.

| Write Concern       | Description                                                                        |
| ------------------- | ---------------------------------------------------------------------------------- |
| `{ w: 0 }`          | **Unacknowledged** ‚Äì No response from the server (fastest, but risky).             |
| `{ w: 1 }`          | **Acknowledged** ‚Äì Confirms write to the Primary (default).                        |
| `{ w: "majority" }` | **Majority Acknowledged** ‚Äì Confirms write to most nodes in a replica set (safer). |
| `{ w: 2 }`          | **Custom Acknowledged** ‚Äì Ensures write is stored on at least 2 nodes.             |

### **‚úÖ Example: Setting Write Concern**

```javascript
db.orders.insertOne(
  { orderId: 123, status: "shipped" },
  { writeConcern: { w: "majority", j: true, wtimeout: 2000 } }
);
```

üìå **Best Practice:** Use `{ w: "majority" }` in production to prevent data loss.

---

### **üìå Read Preferences (Optimizing Read Performance)**

Read preferences **control where MongoDB reads data from in a replica set**.

| Read Preference      | Description                                                       |
| -------------------- | ----------------------------------------------------------------- |
| `primary`            | **Reads from the Primary node** (default, strong consistency).    |
| `primaryPreferred`   | **Prefers Primary** but falls back to Secondary if unavailable.   |
| `secondary`          | **Reads from Secondary nodes** (faster but eventual consistency). |
| `secondaryPreferred` | **Prefers Secondary** but reads from Primary if needed.           |
| `nearest`            | Reads from the **nearest** node (lowest latency).                 |

### **‚úÖ Example: Setting Read Preference**

```javascript
db.orders.find().readPref("secondaryPreferred");
```

üìå **Best Practice:**  
‚úîÔ∏è Use `primary` for **critical data consistency**.  
‚úîÔ∏è Use `secondaryPreferred` for **read-heavy applications** (analytics, reporting).  
‚úîÔ∏è Use `nearest` for **geo-distributed applications**.

---

## **üöÄ Best Practices for Performance Optimization**

‚úîÔ∏è Use **profiling** to detect slow queries (`db.setProfilingLevel(1, 100)`).  
‚úîÔ∏è **Optimize connection pooling** (`maxPoolSize` & `minPoolSize`).  
‚úîÔ∏è Adjust **write concerns** (`w: "majority"`) for durability.  
‚úîÔ∏è Use **read preferences** based on **consistency vs. performance trade-off**.  
‚úîÔ∏è **Monitor and tune performance** (`db.serverStatus()`, `db.currentOp()`).

# **13. Backup and Restore**

- `mongoexport` and `mongoimport`

Ensuring **data backup and recovery** is **critical** for maintaining MongoDB databases. MongoDB offers several tools for **backing up, restoring, exporting, and importing data**.

This guide covers:  
‚úÖ **`mongodump` and `mongorestore`** (Binary Backup & Restore)  
‚úÖ **`mongoexport` and `mongoimport`** (JSON/CSV Export & Import)  
‚úÖ **Cloud Backup Solutions (MongoDB Atlas)**

---

## **1Ô∏è‚É£ `mongodump` and `mongorestore` (Binary Backup & Restore)**

The **`mongodump`** and **`mongorestore`** commands create and restore **binary backups** of your MongoDB databases.

### **üìå `mongodump` (Backup MongoDB Database)**

This creates a **binary snapshot** of your database.

```bash
mongodump --db myDatabase --out /backup/
```

üìå **Options:**  
‚úîÔ∏è `--db myDatabase` ‚Üí Backup a specific database.  
‚úîÔ∏è `--out /backup/` ‚Üí Save the backup in `/backup/` directory.  
‚úîÔ∏è `--archive=mybackup.gz --gzip` ‚Üí Create a **compressed** backup.  
‚úîÔ∏è `--host <host> --port <port>` ‚Üí Specify a remote database.

---

### **üìå `mongorestore` (Restore MongoDB Backup)**

This restores the backup created by `mongodump`.

```bash
mongorestore --db myDatabase /backup/myDatabase
```

üìå **Options:**  
‚úîÔ∏è `--drop` ‚Üí Drops existing collections before restoring.  
‚úîÔ∏è `--archive=mybackup.gz --gzip` ‚Üí Restore from a compressed backup.

‚úÖ **Example: Restore from a Compressed Archive**

```bash
mongorestore --gzip --archive=mybackup.gz
```

---

## **2Ô∏è‚É£ `mongoexport` and `mongoimport` (JSON/CSV Export & Import)**

The **`mongoexport`** and **`mongoimport`** commands are used for exporting and importing **JSON or CSV data**.

### **üìå `mongoexport` (Export Data as JSON/CSV)**

```bash
mongoexport --db myDatabase --collection users --out users.json
```

üìå **Options:**  
‚úîÔ∏è `--collection users` ‚Üí Export data from the `users` collection.  
‚úîÔ∏è `--jsonArray` ‚Üí Export data as a **JSON array** (useful for imports).  
‚úîÔ∏è `--type=csv --fields name,email,age --out users.csv` ‚Üí Export as CSV.

‚úÖ **Example: Export Users as CSV**

```bash
mongoexport --db myDatabase --collection users --type=csv --fields name,email,age --out users.csv
```

---

### **üìå `mongoimport` (Import JSON/CSV Data)**

```bash
mongoimport --db myDatabase --collection users --file users.json --jsonArray
```

üìå **Options:**  
‚úîÔ∏è `--jsonArray` ‚Üí Required if JSON is an array (`[ {...}, {...} ]`).  
‚úîÔ∏è `--drop` ‚Üí Deletes existing data before importing.

‚úÖ **Example: Import CSV Data**

```bash
mongoimport --db myDatabase --collection users --type=csv --headerline --file users.csv
```

‚úîÔ∏è `--headerline` ‚Üí Uses the first row as field names.

---

## **3Ô∏è‚É£ Cloud Backup Solutions (MongoDB Atlas)**

MongoDB Atlas provides **automated backups** with **point-in-time recovery** and **cloud storage**.

### **üìå Enabling Automated Backups in MongoDB Atlas**

1. **Go to** MongoDB Atlas Dashboard.
2. **Select your cluster** and navigate to **Backup**.
3. **Enable Backup** and configure snapshot settings.
4. Choose **Daily, Weekly, or Hourly backups**.

‚úÖ **Restoring from a Backup**

1. Go to **Backup > Snapshots**.
2. Select a **backup snapshot**.
3. Click **Restore** to a new cluster or **download** for local use.

üìå **Best Practice:** Always enable **Cloud Backups** for production databases.

---

## **üöÄ Best Practices for MongoDB Backups**

‚úîÔ∏è **Use `mongodump` for full backups**, **`mongoexport` for data migrations**.  
‚úîÔ∏è **Automate backups** using **cron jobs** or **MongoDB Atlas**.  
‚úîÔ∏è **Compress backups (`--gzip`)** to save space.
